# Internal filtering — `AQL::CONDITIONS` + `AQL::BINDS`

This page covers **server-only filters**: AQL conditions that must apply to a model query but that you do **not** want to expose to the `?filter=` URL because they touch a sensitive field, a computed field, or because they belong to the controller's internal business logic.

Two complementary mechanisms:

1. **`FilterType::VIRTUAL`** — for a key to be accepted from the `?filter=` URL but for **the framework to emit no AQL clause** for it (the controller is responsible for injecting the real condition).
2. **`AQL::CONDITIONS` + `AQL::BINDS`** — to inject raw AQL conditions directly in the call to `list()`, `count()`, `get()`, etc., without exposing the slightest key to the URL.

For **URL-exposed** filtering (whitelisted via `AQL::FILTERS`), see [HTTP filters `?filter=`](filter.md).

## `FilterType::VIRTUAL` — URL key without AQL clause

### When to use

`FilterType::VIRTUAL` is useful when you want a client to be able to write `?filter={"key":"current","val":true}` in the URL — but the decision of what "current" means happens server-side, without exposing the underlying technical field.

Typical cases:

- **Field computed at output**: `current` on `/me/sessions`, derived from the current request's `tokenHash` (not a persisted field).
- **Sensitive field**: you want to filter on a hash or token without the hash ever appearing in the user's request.

### Pattern

Declare the key as `FilterType::VIRTUAL` in the model's `AQL::FILTERS`. The framework accepts the key in `?filter=`, keeps it in the response (`url` field), but **emits no AQL clause**. The controller overrides `beforeModelCall()` to inject the real condition via `AQL::CONDITIONS` + `AQL::BINDS`.

```php
// MeSessions model DI
AQL::FILTERS =>
[
    // ... other exposed keys
    Prop::CURRENT => FilterType::VIRTUAL ,
]

// MeSessionsController::beforeModelCall
protected function beforeModelCall( ?Request $request , array &$init ) : void
{
    parent::beforeModelCall( $request , $init ) ;

    if ( $this->isCurrentFilterRequested( $init ) )
    {
        $hash  = $this->extractTokenHashFromRequest( $request ) ;
        $binds = $init[ AQL::BINDS ] ?? [] ;

        $init[ AQL::CONDITIONS ][] = equal
        (
            key     ( Session::TOKEN_HASH , AQL::DOC )      ,
            aqlBind ( $hash , $binds , 'meSessionsCurrentHash' )
        ) ;

        $init[ AQL::BINDS ] = $binds ;
    }
}
```

Client-side, the JSON response always contains `"url": "https://.../me/sessions?filter=current:true"` — traceability is preserved. But no `tokenHash` ever transits in the HTTP request.

### Advantages

- **Confidentiality**: the technical field (`tokenHash`) stays invisible client-side.
- **Traceability**: the URL stays readable and reproducible for the client.
- **Clean separation**: the model declares the intent ("this key is filterable"), the controller provides the concrete semantics.

## `AQL::CONDITIONS` + `AQL::BINDS` — server-only conditions

### When to use

For any condition that **must not be filterable from the URL**: internal code, sensitive field, complex condition, audience restriction based on request context.

Three typical cases:

1. **Per-user restriction**: a `/me/orders` endpoint always filters on `doc.userId == @currentUserId`. The client shouldn't even be able to try writing `?filter={"key":"userId","val":"..."}`.
2. **Never-exposed field**: filter on `doc.deletedAt == null` to exclude *soft-deletes*, without declaring `deletedAt` in `AQL::FILTERS`.
3. **Complex condition**: sub-query, multi-field comparison, custom predicate.

### Pattern

Inject AQL conditions and *bind variables* directly in the model call:

```php
use oihana\arango\enums\AQL ;
use oihana\arango\enums\Boolean ;
use function oihana\arango\db\binds\aqlBind ;
use function oihana\arango\db\helpers\functions\key ;
use function oihana\arango\db\operators\equal ;

$binds = [] ;

$sessions = $this->model->list
([
    AQL::CONDITIONS =>
    [
        equal( key( Session::USER_ID , AQL::DOC ) , aqlBind( $userKey , $binds , 'sessionUserId' ) ) ,
        equal( key( Schema::ACTIVE   , AQL::DOC ) , Boolean::TRUE                                  ) ,
    ] ,
    AQL::BINDS => $binds ,
]) ;
```

Injected conditions are added to the generated `FILTER` AQL **after** the conditions coming from `?filter=`. *Bind variables* are merged with those auto-generated by the `FilterTrait` pipeline.

### Condition-builder sugar — `isProperty()` / `isAdditionalType()`

The `equal( key( ... ) , aqlValue( ... ) )` idiom above is common enough that two helpers in `oihana\arango\helpers\conditions` ([`src/oihana/arango/helpers/conditions/`](../../../src/oihana/arango/helpers/conditions/)) fold it into a single call — an array ready to spread into `AQL::CONDITIONS`.

```php
function isProperty( string $property , null|string|array $value , string $docRef = AQL::DOC ) : array
```

Builds a single-element condition array: an `IN` when `$value` is an array, an `==` otherwise. The right-hand side goes through `aqlValue()`, so a literal scalar is quoted/escaped while a bind parameter (`'@var'`) or another document reference (`'doc2.name'`) is kept raw.

```php
use function oihana\arango\helpers\conditions\isProperty ;

isProperty( 'status' , 'active' )                 ; // [ "doc.status == 'active'" ]
isProperty( 'status' , [ 'active' , 'pending' ] )  ; // [ "doc.status IN ['active','pending']" ]
isProperty( 'tags'   , '@tags' )                   ; // [ 'doc.tags == @tags' ]
isProperty( 'name'   , 'doc2.name' )               ; // [ 'doc.name == doc2.name' ]
isProperty( 'status' , 'active' , 'other' )        ; // [ "other.status == 'active'" ]
```

Throws `InvalidArgumentException` on a `null`, `''` or `[]` value, and — transitively, via `aqlValue()` — `UnsupportedOperationException` on an unsupported PHP value type.

```php
function isAdditionalType( array|string $schemaType , string $docRef = AQL::DOC ) : array
```

A thin `isProperty()` wrapper scoped to `Schema::ADDITIONAL_TYPE` (`additionalType`) — handy for filtering a polymorphic collection by schema.org type.

```php
use function oihana\arango\helpers\conditions\isAdditionalType ;

isAdditionalType( 'Person' )                     ; // [ "doc.additionalType == 'Person'" ]
isAdditionalType( [ 'Person' , 'Organization' ] ) ; // [ "doc.additionalType IN ['Person','Organization']" ]
```

> **Not a bind.** Unlike `aqlBind()`, these two helpers embed the value through `aqlValue()` directly — appropriate for a **known/trusted** value (an enum literal, a config-driven type list, another document's field), never for unsanitized client input. A value coming from the request still goes through `aqlBind()` as shown above.

Both return an array, so they spread straight into `AQL::CONDITIONS`:

```php
$init[ AQL::CONDITIONS ] =
[
    ...isAdditionalType( [ 'Person' , 'Organization' ] ) ,
    ...isProperty( Schema::ACTIVE , 'true' ) ,
] ;
```

### Composition with `?filter=`

The two mechanisms coexist without conflict:

```php
// The controller exposes ?filter=, but forces a userId server-side
public function list( ?Request $request = null , ?Response $response = null , array $args = [] , array $init = [] ) : mixed
{
    $userKey = $this->getCurrentUserKey( $request ) ;
    $binds   = [] ;

    $init[ AQL::CONDITIONS ][] = equal
    (
        key( 'userId' , AQL::DOC ) ,
        aqlBind( $userKey , $binds , 'currentUserId' )
    ) ;
    $init[ AQL::BINDS ] = $binds ;

    return parent::list( $request , $response , $args , $init ) ;
}
```

The client can still use `?filter={"key":"created","val":"2026-01-01","op":"ge"}`; the controller silently adds `&& doc.userId == @currentUserId`.

The `InjectFilterTrait` (see [Slim controllers](../controllers/README.md)) provides syntactic sugar for this exact case.

## URL vs internal decision rule

| Need | Mechanism |
|---|---|
| The client must filter freely on this field | `AQL::FILTERS` + `?filter=` (see [filter.md](filter.md)) |
| The client passes a `?filter=` key, the server translates to a hidden real condition | `FilterType::VIRTUAL` + `AQL::CONDITIONS` |
| The server imposes a condition never visible client-side | `AQL::CONDITIONS` + `AQL::BINDS` |
| The server wants to preserve client-side condition readability but without exposing the technical field | `FilterType::VIRTUAL` |

## Anti-patterns

### Concatenating a PHP value into an AQL condition

```php
// NEVER — AQL injection risk
$init[ AQL::CONDITIONS ][] = "doc.userId == '$userKey'" ;
```

Always go through `aqlBind()`:

```php
$init[ AQL::CONDITIONS ][] = equal
(
    key( 'userId' , AQL::DOC ) ,
    aqlBind( $userKey , $binds , 'currentUserId' )
) ;
$init[ AQL::BINDS ] = $binds ;
```

### Exposing a sensitive field via `AQL::FILTERS`

```php
// Wrong — client can now filter on the hash
AQL::FILTERS =>
[
    Prop::TOKEN_HASH => FilterType::STRING ,
]
```

For sensitive fields, use `FilterType::VIRTUAL` on a semantic key (`current`) and hide the technical field (`tokenHash`).

### Forgetting to merge `$binds`

```php
// Wrong — $binds modified by aqlBind() is not re-assigned to $init
$init[ AQL::CONDITIONS ][] = equal( ... , aqlBind( ... , $binds , ... ) ) ;
// $init[ AQL::BINDS ] missing here
```

Always reassign `$init[ AQL::BINDS ] = $binds` after a series of `aqlBind()` calls.
