# Glossary

This page defines the key terms used in the framework documentation. It does not try to replace the [official ArangoDB documentation](https://docs.arangodb.com/stable/) — each entry links to the reference concept where applicable — but to settle on a shared vocabulary.

## ArangoDB vocabulary

### ACID

Acronym for Atomicity, Consistency, Isolation, Durability. ArangoDB guarantees these four properties on its transactions, including when they touch multiple documents and multiple collections.

### AQL

*ArangoDB Query Language*. Declarative SQL-inspired language that expresses both document queries (`FOR doc IN users FILTER ... RETURN doc`) and graph traversals (`FOR v, e IN 1..3 OUTBOUND start GRAPH 'g' ...`). The entire `db/` layer of the framework produces AQL.

### Bind variable

Variable injected into an AQL query as `@var` (value) or `@@coll` (collection name), with its value supplied separately as `bindVars`. Guarantees no AQL injection and enables *query cache* reuse. The `aqlBind()` helper (under `db/binds/`) automates the safe declaration.

### Collection

Container for documents sharing the same nature. Two kinds: **document collection** (default) and **edge collection**. A collection lives in a `database`.

### Cursor

Client-side iterator over the result of an AQL query. ArangoDB streams results in batches — the *cursor* exposes the methods to advance (`hasNext`, `next`, `fetch`).

### Database

Top-level namespace on an ArangoDB server. A *database* contains collections, graphs, views, indexes, user-defined AQL functions. AQL queries are scoped to a *database*.

### Document

Storage unit: a JSON object identified by `_key`, `_id` and `_rev`. ArangoDB is *schemaless* by default; a JSON Schema validator can be attached to a collection if you want to enforce a shape.

### Edge

Special document stored in an **edge collection**, which must contain the `_from` and `_to` fields. Each references the `_id` of a document (the edge's endpoint). Allows building native graphs.

### Foxx

JavaScript microservices embedded in the database, executed by ArangoDB's V8 engine. Lets you expose an HTTP API directly from the server without an external application layer. `oihana/php-arango` does not rely on Foxx.

### Graph

Logical definition that groups one or more edge collections and their vertex collections, and enables AQL traversals via `GRAPH 'name'`. At small scale, *ad hoc* traversals on anonymous edge collections are often enough.

### Index

Acceleration structure attached to a collection. Main types: `persistent` (lookups and uniqueness constraints), `ttl` (automatic document expiration), `geo` (geospatial queries), `fulltext` (deprecated in favor of *ArangoSearch* views), `vector` (similarity search, ArangoDB 3.12+), `mdi` (multi-dimensional). See [`indexes.md`](indexes.md).

### MVCC

*Multi-Version Concurrency Control*. ArangoDB's concurrency mechanism: each write to a document creates a new revision (`_rev`); concurrent reads see a consistent *snapshot*. The `_rev` field enables optimistic detection of write conflicts.

### RocksDB

ArangoDB's default storage engine since version 3.4. Disk-persistent, ACID transactions, compression. Maintained by Meta.

### Traversal

Graph walk in AQL via `FOR v, e, p IN min..max <DIRECTION> <startVertex> GRAPH 'g'`. `DIRECTION` is `OUTBOUND`, `INBOUND` or `ANY`. `v` is the visited vertex, `e` the edge taken, `p` the full path (useful to filter on a path).

### Vertex

Graph node. In ArangoDB, a *vertex* is simply a document of a collection (as opposed to an *edge* that lives in an edge collection). The term is used in the traversal context.

### View

Logical view on top of one or several collections, mainly used for advanced text search via *ArangoSearch* (linguistic analyzers, BM25, *facets*).

### `_from` / `_to`

Required fields of an *edge*. `_from` holds the `_id` of the source document, `_to` the `_id` of the target document. The presence of both is validated by ArangoDB upon insertion in an edge collection.

### `_id`

Full identifier of a document, of the form `<collection>/<_key>` (e.g. `users/42`). Used for references between documents (notably `_from` and `_to` on *edges*).

### `_key`

Internal primary key of a document within its collection. Either provided at insertion time or auto-generated by ArangoDB. Unique per collection but not per *database*.

### `_rev`

MVCC revision identifier of a document. Changes at every write. Enables optimistic conflict detection (`If-Match` HTTP header, or `OPTIONS { ignoreRevs: false }` AQL clause).

## `oihana/php-arango` vocabulary

### Alteration (`alt`)

Transformation function applied to a field's value **before** comparison in a filter. Exposed over HTTP via `?filter={"key":"name","val":"john","alt":"lower"}`, which produces `FILTER LOWER(doc.name) == "john"`. See [`filter.md`](filter.md).

### Authorizer

*Callable* of the form `Closure(string $subject): bool` injected into `$init[Arango::AUTHORIZER]`. Consulted by the framework to decide whether a projection controlled by `AQL::REQUIRES` should be included. Stays agnostic of the authorization system in use (Casbin, OPA, custom control).

### Capability

Fine-grained permission carried by a URL parameter value (e.g. `?skin=full`) or by a field key, rather than by an HTTP verb. The framework exposes the `Capability::PARAMS` pattern to tie a Casbin permission to a value. See `CapabilityGuardTrait`.

### Cascade

Automatic propagation of a deletion to the *edges* connected to a document. Emitted by `Documents::delete()` via an `afterDelete` signal that the `EdgesFromTrait` and `EdgesToTrait` traits intercept. See also [Signal](#signal).

### Container (DI)

Dependency injection container compliant with PSR-11 (`Psr\Container\ContainerInterface`). The framework accepts a container in the constructor of models, controllers and commands, and resolves its dependencies (ArangoDB connection, schemas, logger, signals) by service identifier. PHP-DI is used in examples but the code is not tied to it.

### Definition

PHP file returning an array of DI definitions consumed by the container. In a host application, the `oihana/php-arango` *definitions* live under `api/definitions/@arango/`. Convention: one file per model, one file per controller.

### Facet

Aggregation by value exposed to the HTTP API: for a given field, returns the list of distinct values and their count. Declared via `AQL::FACETS` on a `Documents` model. Useful for checkbox filtering UIs.

### Field

Descriptor of a model's field. Configurable through the `Field::FILTER` markers (filter type), `Field::SKINS` (list of *skins* that enable the field), `Field::FIELDS` (sub-projection if the field is a nested document).

### Filter

Descriptor of a field filterable from the URL. Declared in `AQL::FILTERS` with a `FilterType::*`. The client sends `?filter={"key":"...","val":"...","op":"...","alt":"..."}`; the framework produces the corresponding `FILTER` AQL.

### `FilterType::VIRTUAL`

Special filter type: the key is accepted from the URL, but the framework **emits no AQL clause**. Used when a controller wants to accept a filter without exposing the underlying field (sensitive field, computed field). The controller then injects the real condition via `AQL::CONDITIONS` + `AQL::BINDS`.

### Model (`Documents`, `Edges`)

High-level class that represents an ArangoDB collection and exposes the CRUD + listing + search + projection operations. `Documents` for a document collection, `Edges` for an edge collection. Configured through an array of `AQL::*` keys at construction time.

### Projection

Selection of the fields returned by the framework for a given document, dependent on the current *skin* and on the `Field::SKINS` markers placed on the fields. See [`edges-joins-projection.md`](edges-joins-projection.md).

### Signal

Application event emitted by a model (e.g. `afterDelete`) and propagated through the `oihana/php-signals` bus. Allows decoupling between the main action (deleting a document) and its side effects (purging *edges*).

### Skin

Named projection of a document, passed via the `?skin=` URL parameter. Canonical values: `default` (light listing), `full` (full record), `main` (minimal skin to break INBOUND cycles), `internal` (server-only projection, never exposed over HTTP — see [`tips.md`](tips.md)). Controllers add their business-specific *skins* (e.g. `image`, `offers`).

### Trait composition

Central architecture pattern of the framework: the `Documents` and `Edges` classes contain almost no code of their own — they aggregate about fifty single-responsibility traits (`DocumentsGetTrait`, `FilterTrait`, `SortTrait`, ...). Allows consuming a subset of the framework without inheriting the rest.
