## [Unreleased]

### Added

- **Polymorphic edges ‚Äî per-branch gating, the `FALLBACK` branch, and a shared branch assembler (lot 2 of 3: the security layer).** Each branch of a polymorphic edge is now permission-gated on its own (`Field::REQUIRES` / `AQL::REQUIRES` via `isAuthorized()`): a denied branch is dropped from the `APPEND` ‚Äî its edge collection is never traversed, so neither a value nor an existence bit of the hidden type can leak (fail-closed). A new optional `Arango::FALLBACK` branch catches discriminator values matching none of the declared types, guarded by `NOT IN [ ‚Ä¶ ]` over all map keys (gated or not, so a document of a denied type routes to nothing, never to the fallback ‚Äî no oracle). When every branch is dropped the `LET` holds `[]`, so the projection resolves to `null` (`Filter::EDGE`) / `[]` (`Filter::EDGES`) rather than a broken clause. **The branch machinery is no longer duplicated:** the validation + per-branch gating + fallback + `APPEND` combine ‚Äî until now living in the join builder ‚Äî is extracted into a shared `buildPolymorphicRelationVariable()` that both `buildPolymorphicJoinVariable()` and `buildPolymorphicEdgeVariable()` delegate to; a `fn($branch, $guard): string` callback is the only relation-specific part (a join wraps `buildJoinSubquery()`, an edge delegates to the already-parenthesized `buildEdgeSubquery()`). The anti-oracle gate is therefore written and tested once. The join output is byte-identical (its 17 characterization tests pass unchanged).
  - **Tests:** new `BuildPolymorphicRelationVariableTest` exhaustively covers the shared assembler with a stub branch builder (validation, guards, ordering, per-branch gating, fallback `NOT IN`, denied fallback, denied type still excluded from the fallback, every-branch-denied ‚Üí `[]`); `BuildPolymorphicEdgeVariableTest` gains the edge-level gating / fallback cases against real `MockEdges` (denied branch dropped, all denied ‚Üí `[]`, fallback guarded by `NOT IN`, denied fallback dropped, non-array fallback throws). 100% line coverage on the shared assembler and both thin adapters.

- **Polymorphic edges ‚Äî an edge can now pick its traversed collection from a discriminator field of the start vertex (lot 1 of 3: the engine).** The edge counterpart of the polymorphic join: a regular edge traverses one fixed collection (`AQL::MODEL`); a *polymorphic* edge instead declares `Arango::DISCRIMINATOR` (the start-vertex field that decides ‚Äî e.g. `kind`) and `Arango::MAP` (a `type => edge-definition` table, one branch per value ‚Äî e.g. `warehouse => warehouse_edges`, `company => company_edges`), so the traversed collection is chosen at query time from a value of the parent document. AQL forbids a computed collection in `FOR ‚Ä¶ IN <dir> ‚Ä¶ <collection>`, so the edge is compiled as an **`APPEND` of guarded static traversals**: one regular traversal per branch, each guarded by an equality on the discriminator (`FILTER doc.kind == "warehouse"`) so only the matching branch yields rows. The resulting `LET` holds an **array** exactly like a regular edge, so the projection layer unwraps it with `FIRST()` for a `Filter::EDGE` or keeps the whole array for a `Filter::EDGES` ‚Äî no change to the projection. Each branch is a full edge definition, so it may declare its own `AQL::DIRECTION`, `AQL::MAX_DEPTH`, etc. No new `Filter` marker: an edge definition routes to the polymorphic builder as soon as it carries `Arango::MAP` + `Arango::DISCRIMINATOR` (via the shared `isPolymorphic()`), and the two existing composable gates (field-level `Field::REQUIRES`, definition-level `AQL::REQUIRES`) still gate the whole edge. Builds on lot 0 (the `buildEdgeSubquery` extraction): `buildPolymorphicEdgeVariable()` calls it per branch with the injected guard. Per-branch permission gating and the `FALLBACK` branch land in lot 2; the integration test and FR/EN docs in lot 3. **`Filter::EDGES_COUNT` on a polymorphic definition is out of scope** (the count builder uses `LENGTH(traversal)`, not an `APPEND`) ‚Äî a count entry stays on the count builder.
  - **Tests:** `IsPolymorphicEdgeTest` (the detector: map + discriminator present / missing / empty / wrong type); `BuildPolymorphicEdgeVariableTest` (two-branch `APPEND` with quoted guards, single branch without `APPEND`, `Arango::UNIQUE` override, direction varying per branch, a custom start vertex honored in the guard and traversal, the malformed-definition throws); `BuildVariablesTest` gains the dispatch cases (a `MAP` + `DISCRIMINATOR` edge routes to the polymorphic builder, is still dropped by the field-level gate, and an `EDGES_COUNT` entry keeps routing to the count builder). 100% line coverage on the touched files.

- **Polymorphic joins ‚Äî live validation and bilingual docs (lot 3 of 3, closing the feature).** A `PolymorphicJoinIntegrationTest` drives the real `buildVariables()` dispatch (`isPolymorphic` ‚Üí the polymorphic builder ‚Üí `APPEND` of guarded branches) against a seeded, disposable ArangoDB: a `#Warehouse` scope resolves from `warehouses`, a `#Company` scope from `subsidiaries`, an undeclared scope to `null` without a fallback and from `regions` with one ‚Äî proving the generated `APPEND( ( FOR ‚Ä¶ ) , ( FOR ‚Ä¶ ) )` parses AND routes to the right collection on a real server (skipped when no DB is reachable). FR/EN wiki `edges-joins-projection.md` gains a *Polymorphic join* section (the discriminator / `MAP` / `FALLBACK` declaration, the `APPEND`-of-guarded-branches AQL, the array-`LET`-unwrapped-by-the-projection note, and the per-branch gating + fallback `NOT IN` no-oracle guarantees), added to both tables of contents.

- **Polymorphic joins ‚Äî per-branch permission gating and the `FALLBACK` branch (lot 2 of 3: the security layer).** Each branch of a polymorphic join is now permission-gated on its own (`Field::REQUIRES` / `AQL::REQUIRES` on the branch definition, via the shared `isAuthorized()`): a denied branch is **dropped from the `APPEND`** ‚Äî its collection is never queried, so neither a value nor an existence bit of the hidden type can leak (fail-closed). It composes (logical AND) with the field- and definition-level gates that already guard the whole join. A new optional `Arango::FALLBACK` branch (a join definition) catches discriminator values matching **none of the declared types**, guarded by `NOT IN [ ‚Ä¶ ]` over **all** map keys ‚Äî gated or not, so a document of a permission-denied type routes to *nothing*, never to the fallback (no oracle). When every branch is gated out the `LET` holds `[]`, so the projection resolves to `null` (`Filter::JOIN`) / `[]` (`Filter::JOINS`) rather than a broken statement. The integration test and FR/EN docs land in lot 3.
  - **Tests:** `BuildPolymorphicJoinVariableTest` gains the gating cases (a denied branch dropped from the union, every branch denied ‚Üí `LET x = []`, the `FALLBACK` branch guarded by `NOT IN` over the known types, a denied fallback dropped, a denied branch type still excluded from the fallback guard, a non-array `FALLBACK` throws). 100% line coverage on the touched file.

- **Polymorphic joins ‚Äî a join can now pick its target collection from a discriminator field of the parent document (lot 1 of 3: the engine).** A regular join binds one fixed collection (`AQL::MODEL`); a *polymorphic* join instead declares `Arango::DISCRIMINATOR` (the parent field that decides ‚Äî e.g. `selector.areaScope`) and `Arango::MAP` (a `type => join-definition` table, one branch per value ‚Äî e.g. `‚Ä¶#Warehouse => warehouses`, `‚Ä¶#Company => subsidiaries`), so the collection is chosen at query time from a value of the document itself. AQL forbids a computed collection in `FOR ‚Ä¶ IN ‚Ä¶`, so the join is compiled as an **`APPEND` of guarded static branches**: one regular join sub-query per branch, each guarded by an equality on the discriminator (`&& doc.selector.areaScope == "‚Ä¶#Warehouse"`) so only the matching branch yields rows. The resulting `LET` holds an **array** exactly like a regular join, so the projection layer unwraps it with `FIRST()` for a `Filter::JOIN` or keeps the whole array for a `Filter::JOINS` ‚Äî no change to the projection. The parent key path (`Arango::PROPERTY`, default the field name) and an optional foreign key attribute (`Arango::KEY`) declared at the top level are shared as defaults across branches; a branch may override its own key. No new `Filter` marker: a join definition simply routes to the polymorphic builder as soon as it carries `Arango::MAP` + `Arango::DISCRIMINATOR` (via the shared `isPolymorphic()`), and the two existing composable gates (field-level `Field::REQUIRES`, definition-level `AQL::REQUIRES`) still gate the whole join. Internally, `buildJoinVariable()`'s sub-query body is extracted into a reusable `buildJoinSubquery()` (byte-identical output for regular joins) that the new `buildPolymorphicJoinVariable()` calls per branch with the injected guard. New `Arango` keys: `DISCRIMINATOR`, `MAP`, `FALLBACK` (the last reserved for the unmatched-value branch, wired in lot 2). Per-branch permission gating and the `FALLBACK` branch land in lot 2; the integration test and FR/EN docs in lot 3.
  - **Tests:** `IsPolymorphicJoinTest` (the detector: map + discriminator present / missing / empty / wrong type); `BuildJoinSubqueryTest` (the extracted body: no-`LET` output shape, `extraConditions` guard injection); `BuildPolymorphicJoinVariableTest` (two-branch `APPEND` with quoted guards, single branch without `APPEND`, key-path default, `Arango::UNIQUE` override, shared `Arango::KEY` inherited and branch override, the `Filter::JOINS` array form with `IN` + `SORT`, real `schema.oihana.xyz` URI type values quoted); `BuildVariablesTest` gains the dispatch cases (a `MAP` + `DISCRIMINATOR` definition routes to the polymorphic builder, and is still dropped by the field-level gate). 100% line coverage on the five touched files.

- **Bounds gain per-field exclusion options and a `count` ‚Äî a `0`-as-"not filled" no longer drags the minimum to zero.** A bound definition (new `Bound` enum) can now drop values from the `MIN` / `MAX` aggregate: `Bound::POSITIVE => true` (keep `> 0`), `Bound::MIN` / `Bound::MAX` (the accepted input domain, e.g. a temperature range), and `Bound::IGNORE` (sentinel value(s), a scalar or a list ‚Üí `NOT IN [ ‚Ä¶ ]`). Options combine with a logical AND. An excluded value maps to `null` ‚Äî which `MIN` / `MAX` already ignore ‚Äî so the guard is **per field**: a document dropped from one field's extent still frames the others (a flat `FILTER` could not do that). Each bound also returns a **`count`** (non-null values after exclusions), so a client knows whether a range control is worth showing. The whitelist gains `Bound::PROPERTY` (the path, was `Facet::PROPERTY`) and `Bound::REQUIRES` (the permission subject, a coherent alias of `Field::REQUIRES`) ‚Äî the response shape becomes `{ min, max, count }`.
  - **Tests:** `BoundsQueryTraitTest` gains the option cases (`POSITIVE` guard on `MIN` / `MAX` / `SUM`, the accepted-domain `>=` / `<=`, `IGNORE` as scalar and list, options combining with AND, the bare-field non-null count); `BoundsRequiresGateTest` exercises `Bound::REQUIRES` on the definition.
  - **Docs:** FR/EN `db/bounds.md` gains an *exclusion options* section (with the declaration-domain vs observed-output naming note) and the `{ min, max, count }` shape.

- **`?metaOnly=` ‚Äî one generic flag to return the metadata without the documents (lot 3 of 4).** `?metaOnly=true` skips the document-fetch query and returns an empty result set while the response *metadata* ‚Äî an exact `total`, the requested facet counts (`?facetCounts=`) **and** the numeric bounds (`?bounds=`) ‚Äî is still computed. It generalises the old counts-only `?facetsOnly=`: where that flag only kept the facet counts, `?metaOnly=` spans facets and bounds alike, so a request can build the whole filter sidebar (categorical facets + numeric ranges + total) in a single documentless round-trip. New `Arango::META_ONLY` + `PrepareMetaOnlyTrait`; the controller ORs the two flags so **`?facetsOnly=` stays a truthy alias** (deprecated, `@deprecated` on the constant and the trait ‚Äî no behaviour change for existing callers).
  - **Tests:** `PrepareMetaOnlyTraitTest` (boolean forms, request-over-args precedence, empty-value fallback); `DocumentsControllerReadTest` gains `testListMetaOnlyReturnsBoundsWithoutDocuments` (documents skipped, exact total, bounds still computed) and `testListFacetsOnlyStaysATruthyAliasOfMetaOnly` (the deprecated flag still skips the documents).

- **`?bounds=` ‚Äî the bounds engine is now reachable from the list endpoint (lot 2 of 4: the controller surface).** `GET ‚Ä¶?bounds=width,height` runs `Documents::bounds()` over the **same filtered set** as the list (a request narrowed by `?filter=` / `?search` / a facet reports the bounds of that subset) and attaches a `field => { min, max }` map to the response under `bounds`, beside the document list ‚Äî the numeric counterpart of the `?facetCounts=` sidebar. New `PrepareBoundsTrait::prepareBounds()` parses the CSV (mirroring `PrepareFacetCountsTrait`); `DocumentsControllerListTrait` resolves it and, on a real `Documents` model, attaches `$options[Arango::BOUNDS]`. Combinable with `?facetCounts=` in a single call (categorical facets + numeric bounds, one round-trip). The `metaOnly` flag (bounds without the document payload) lands in lot 3.
  - **Tests:** `PrepareBoundsTraitTest` (CSV parse / trim, request-over-args precedence, non-array args ignored, null when absent); `DocumentsControllerReadTest::testListComputesBoundsWhenRequested` (the `?bounds=width` wiring runs `bounds()` over the same filter and attaches the extent to the payload).
  - **Docs:** new FR/EN `db/bounds.md` (`?bounds=` syntax, `AQL::BOUNDS` declaration with flat / nested `[*]` measures, shared filter, the `REQUIRES` bound-oracle gate, `?metaOnly=`); `db/facets.md` `?facetsOnly=` section marked deprecated with a cross-link, and both `db/README.md` indexes gain the page.

- **Numeric bounds ‚Äî a model can now expose the `{ min, max }` extent of its fields, computed over the filtered set (lot 1 of 4: the engine).** The counterpart of facet counts for continuous measures: where a facet count returns a per-value distribution (meaningless on a `float`), a bound returns the two scalars that frame a field ‚Äî typically to size a min/max range control from the current results. A model declares a **fail-closed** whitelist (`AQL::BOUNDS`, mirroring `$sortable` / `$groupable` ‚Äî a `null` `$bounds` makes nothing boundable), accepting both a **bare field name** (list entry, `[ 'width', 'height' ]`) and a **keyed definition** (`'price' => [ Facet::PROPERTY => 'offers[*].price' ]`), mixed; the new `Documents::bounds()` returns a `field => { min, max }` map. The extent is aggregated over the **same conjunctive filter** as `list()`, so a request narrowed by `?filter=` / `?search` / a facet reports the bounds of that subset. Flat scalar fields share **one** `COLLECT AGGREGATE` ‚Äî a single pass frames all of them at once (`AGGREGATE width_min = MIN(doc.width), width_max = MAX(doc.width), ‚Ä¶`); a nested measure declared with the `[*]` array-expansion marker (`offers[*].price`) unwinds the array into its own `FIRST(( ‚Ä¶ ))` `LET`, merged into the flat block. `MIN` / `MAX` ignore `null`, so a field absent from a document does not skew its extent.
  - **Security (bound oracle).** Each field is permission-gated exactly like a facet count, but through the depth-aware `isPathAuthorized()` walker (`Field::REQUIRES` inherited from `$fields` at the **exact sub-field**, plus an optional subject declared on the bound definition). A refused field is dropped from the query ‚Äî a `{ min, max }` leaks a *real value* of the hidden field, a sharper oracle than a count, so the gate is not optional. Fail-open without an injected authorizer, matching the facet gate.
  - **API.** New `AQL::BOUNDS` (config whitelist key) + `Arango::BOUNDS` (runtime `?bounds=` key). New traits: `aql\BoundTrait` (the `$bounds` whitelist + `initializeBounds()`, composed into `AQLQueryTrait` / `ListQueryTrait` beside `FacetTrait`), `queries\BoundsQueryTrait` (`buildBoundsQuery()`), `documents\DocumentsBoundsTrait` (`bounds()`, composed into `DocumentsMethodsTrait`). The controller surface (`?bounds=`) and the `metaOnly` flag land in the next lots.
  - **Tests:** `BoundsQueryTraitTest` (flat fields in one COLLECT, renamed property, single- and multi-hop `[*]` unwind, flat/nested mix via `MERGE`, unknown key ignored, fail-closed on `null`, empty when nothing requested, array-of-fields input, shared list conditions); `BoundsRequiresGateTest` (refused / granted flat field, ungated field unaffected, explicit `REQUIRES` on the bound definition, a locked deep sub-field, fail-open without an authorizer); `DocumentsBoundsTraitTest` (object / array / null result shaping, empty when nothing boundable).

### Changed

- **The edge traversal sub-query body is now a single reusable source (internal refactor, no behaviour change).** `buildEdgeVariable()`'s traversal body ‚Äî the parenthesized `( FOR v, e IN <dir> startVertex edge_collection ‚Ä¶ RETURN ‚Ä¶ )` ‚Äî is extracted into a new `buildEdgeSubquery()`; `buildEdgeVariable()` becomes a thin wrapper that resolves the `LET` variable name and prefixes the body. Output is byte-for-byte identical (the 30+ characterization tests pass unchanged). `buildEdgeSubquery()` gains an `$extraConditions` param emitted as a `FILTER` right after the traversal (empty ‚Üí no `FILTER`, so the legacy output is unchanged). This mirrors the join split (`buildJoinVariable` / `buildJoinSubquery`) and prepares a forthcoming *polymorphic edge* ‚Äî one whose traversed edge collection is chosen from a discriminator field of the start vertex ‚Äî that wraps several guarded traversals in an `APPEND`. Guarded additionally by a new `BuildEdgeSubqueryTest`.

- **The shared `FOR` + `FILTER` scope of the list family is now a single source (internal refactor, no behaviour change).** `buildListQuery()`, `buildCountQuery()`, `buildFacetCountsQuery()` and `buildBoundsQuery()` each re-built the same `FOR` (View + `SEARCH`, or collection) and the same conjunctive `FILTER` (conditions + active + facets + filter + search + runtime conditions) ‚Äî four byte-for-byte copies that had to be kept in lockstep, or `count()` / a facet bucket / a bound would frame a set the list never shows. Extracted into a new `FilteredScopeTrait::buildFilteredScope()` returning the `[ $for, $filter ]` pair, composed by the four builders; each call site drops to a one-liner. `CountQueryTrait` read the runtime conditions under `Arango::CONDITIONS` where the others used `AQL::CONDITIONS` ‚Äî both resolve to `'conditions'`, so the unification is a no-op. The full suite is unchanged (the four builders emit identical AQL), guarded additionally by a new `FilteredScopeTraitTest`.

- **`?filter=` / `match` ‚Äî three malformed spellings now fail loud instead of silently returning `0`.** Each of them built a *valid but always-false* AQL (`total: 0`), indistinguishable from a genuine empty result ‚Äî a debugging trap, and the opposite of the recent fail-closed turn (sort / group / facets). To be distinguished from an **unknown key**, which stays silently ignored (loosens, never restricts). The three forms, all detected at build time:
  - **Nested object leaf after `[*]`** (`offers[*].seller.id`, where `seller` is an object) ‚Äî *auto-corrected*. The dotted sub-path was unrecognised and fell through to `doc.offers[*].seller.id == @v` (an array projection compared to a scalar, never true); `prepareFilterArray()` now widens the sub-field capture to a **dotted** path and builds the correct inline `LENGTH(doc.offers[* FILTER CURRENT.seller.id == @v]) > 0`. A **direct** sub-field (`offers[*].priceCurrency`) already worked; the greedy base keeps the multi-level existential form (`employee[*].contactPoint[*].verified`) unchanged. `assertAttributeName()` still guards the dotted name against injection.
  - **Operator object in the simple `match` form** (`{"price":{"op":"gt","val":0}}`) ‚Äî *rejected*. The `{op,val}` object was bound as the compared value (`CURRENT.price == @{op,val}`, never true); `buildCombinedInlineFilter()` now throws a `ValidationException` when a simple-form value is **non-scalar**, pointing to the explicit `all` form (where an object equality is legitimate).
  - **Unsupported operator inside a `match`** (`between`, and generically any operator outside the inline alias set ‚Äî `contains`, `sw`, `regex`, or a typo) ‚Äî *rejected*. It degraded to `==` against the raw value (`CURRENT.price == @[10,100]`, never true); `buildInlineFilterCondition()` now throws a `ValidationException` for any operator not in `FilterComparator::__ALIAS__` (`eq, ne, gt, ge, lt, le, in, nin, like, nlike, match, nmatch`) instead of the silent `==` fallback. Message points to two conditions (`ge` + `le`) for a range. `in` / `nin` stay supported (array value legitimate).
  - **Backward compatibility:** additive ‚Äî legitimate forms are unchanged (direct `x[*].field`, simple `match` of scalar equalities, the `all` / `any` / `none` form, the multi-level existential form, and the silently-ignored unknown key). Only genuinely malformed spellings ‚Äî that nobody writes on purpose and that returned `0` ‚Äî now raise.
  - **Tests:** `BuildInlineFilterConditionTest` (`between` / unknown operator throw, `in` still builds `IN @v`), `BuildCombinedInlineFilterTest` (operator-object and list values throw, object equality via `all` and `null` value still accepted), `HasHierarchicalFilterTest` (a nested object leaf builds the correct inline `CURRENT.seller.id`). 100% line coverage on the three touched files.
  - **Docs:** FR/EN `db/filter.md` gains a *two `match` forms + three rejected traps* section (simple vs `all`, each trap with ‚úó written / ‚úì correct).

- **Aggregate facets now gate the aggregated field ‚Äî closes the `agg:max` bound oracle, and a URL can no longer aggregate an arbitrary field (BREAKING).** An aggregate facet (`Facet::EDGE_AGGREGATE` / `Facet::JOIN_AGGREGATE`) computes a statistic (`avg` / `max` / `sum` / ‚Ä¶) over a related field and compares it to a threshold; the field was URL-overridable (`{ "field": "‚Ä¶" }`) and only injection-checked, so a client could aggregate **any** related field ‚Äî hidden or not ‚Äî and reconstruct a bound of its value by dichotomy on the threshold. Two layers now guard it: **(1, fail-closed whitelist)** the URL may only pick a field the facet declares as aggregatable in `AQL::FIELDS` (a string is the single allowed field, a list is the allowed set with its first entry the default); a field outside the declared set ‚Äî or any field when none is declared ‚Äî neutralises the facet to `Boolean::FALSE`. **(2, opt-in per-user gate)** when the facet declares its target model (`AQL::MODEL`), the aggregated field inherits that model's `Field::REQUIRES` per request through the shared `isPathAuthorized()` walker; a refused field neutralises the facet. Absent `AQL::MODEL`, layer 2 is skipped ‚Äî the whitelist still bounds the surface. **Backward compatibility:** an aggregate facet that let the URL choose the field must now declare the allowed set in `AQL::FIELDS` ‚Äî exactly the `?sort=` / `?groupBy=` fail-closed migration; a facet already declaring `AQL::FIELDS` and querying that same field is unchanged. Relation-facet locking stays explicit (`Field::REQUIRES` on the facet definition ‚Äî no inheritance from a homonymous `$edges` / `$joins` relation).
  - **Tests:** `FacetAggregateRequiresGateTest` (Levier 2 ‚Äî a refused / granted target field, an ungated field, the no-`AQL::MODEL` skip, the unknown-model skip, and fail-open without an authorizer); `FacetAggregateTest` gains the Levier 1 cases (a URL field outside the whitelist, and with no whitelist declared, both neutralised) and its facets now declare `AQL::FIELDS`.

- **`?filter=` array-of-objects leaves are now gated too ‚Äî closes the `contactPoint[*].email` / `match` sub-field oracle.** Completes the deep `?filter=` gate for the object-array expansion path (a distinct builder from the scalar / document / relation leaves): a single-field expansion (`contactPoint[*].email`) is gated in `buildArrayTraversal()` via `isPathAuthorized()` on the array element's sub-field, and every field referenced by a `match` (the simple `{ propertyID, value }` form or the explicit `{ all | any | none: [ { key, ‚Ä¶ } ] }` form) is gated in `prepareFilter()` through the new `collectMatchFields()` extractor. A refused sub-field neutralises the whole predicate to `Boolean::FALSE` ‚Äî never dropped, which under a `none` match would loosen into an existence oracle. **Backward compatibility:** additive ‚Äî an object array whose element sub-fields carry no `Field::REQUIRES` filters byte-for-byte as before.
  - **Tests:** `FilterDeepRequiresGateTest` gains object-array cases (a refused / granted `contactPoint[*].email` and its fail-open without an authorizer; a refused / granted simple `match`, and a refused explicit-`all` `match`).

- **`?filter=` deep leaves are now gated at the exact sub-field, across nested documents and relations ‚Äî closes the `employee[*].salary` / `address.city` filter oracle.** The permission gate on `?filter=` resolved `Field::REQUIRES` from the **root segment only**, so a leaf hidden deeper leaked through: a sub-field of a nested document (`address.city` ‚Äî `address` readable, `city` locked) or a leaf inside an *authorized* relation (`employee[*].salary` ‚Äî the `employee` edge allowed, but `salary` locked in the employee model) turned an include / exclude (or a bindable comparison) into an oracle on the hidden value. The hierarchical filter builder now threads the **current model's projection** ‚Äî switched to the target model's `$fields` when an edge / join is crossed ‚Äî and the **relative field path** (reset at each crossing) down to every leaf, gated through the shared `isPathAuthorized()` walker (`Field::FIELDS` + fail-closed `SKIN_FIELDS` union, every level of the sub-path). A refused leaf is neutralised to `Boolean::FALSE` and ‚Äî the security crux ‚Äî sinks the **whole traversal** *before* the `all` / `none` quantifier negation, so a refused leaf under `quant:all` can never become `NOT(false) = true`, an existence oracle (¬´ Option B ¬ª). When the target model of a relation is unresolvable, the leaf fails open (nothing to inherit). **Backward compatibility:** additive ‚Äî a model with no nested / target-model `Field::REQUIRES` filters byte-for-byte as before. **Array-of-objects leaves** (`contactPoint[*].email`, `match`) flow through a distinct builder and are gated too ‚Äî see the *array-of-objects* entry above.
  - **Tests:** unit (`FilterDeepRequiresGateTest` ‚Äî a refused / granted nested-document leaf (`address.city`), a refused edge leaf neutralising the whole traversal and a granted one traversing, the `all`-quantifier leaf staying `false` (no existence oracle), a fail-open on an unresolved target model, a refused / granted join leaf, and deep leaves failing open without an authorizer).

- **`?groupBy=` deep paths are now gated at the exact sub-field, not just the root ‚Äî closes the `address.city` group-by oracle.** The permission gate on a grouping dimension / aggregate resolved `Field::REQUIRES` from the **root segment only** (`explode('.', $field)[0]`), so a sub-field locked in the `Field::FIELDS` of a structural field (`address.city` ‚Äî `address` unlocked, `city` carrying a `REQUIRES`) leaked its distinct values (dimension) or a bound (aggregate) while the root stayed open. A new shared helper **`isPathAuthorized()`** walks the whole dotted path, reading `Field::REQUIRES` at **every** level and descending through the declared sub-fields ‚Äî `Field::FIELDS` **and**, fail-closed, every `AQL::SKIN_FIELDS` bucket (a sub-field locked in *any* projection stays gated, since a permission is skin-independent). `GroupTrait` now gates the dimension and the aggregated field through it instead of the root-only `isAttributeAuthorized()`. A single-segment path behaves exactly as before ‚Äî `isPathAuthorized()` is a strict superset of `isAttributeAuthorized()`, and each segment is stripped of the `[*]` marker so the same helper serves the hierarchical filter leaves next. **Backward compatibility:** additive ‚Äî a model with no nested `Field::REQUIRES` groups byte-for-byte as before; the gate only adds refusals on sub-fields that were previously leaking.
  - **Tests:** unit (`IsPathAuthorizedTest` ‚Äî the shared walker across null / absent / scalar roots, single-segment parity with `isAttributeAuthorized()`, a locked leaf / locked intermediate / granted leaf / undeclared sub-field, descent past a scalar and a sub-field-less intermediate, the fail-closed `SKIN_FIELDS` bucket union with a malformed bucket ignored, and `[*]` stripping; `GroupRequiresGateTest` ‚Äî a refused deep dimension and a refused deep aggregate are dropped, a granted deep dimension is grouped).

### Fixed

- **`returnFields()` anchors relation `LET` sub-queries on the projected document when projecting through a vertex traversal.** When a model's declared fields include a relation (`Filter::EDGES` / `EDGE` / `EDGES_COUNT` / `JOIN` / `JOINS`) and that model is projected as the **target of a vertex traversal** (`Edges::getVertices()` and its inbound/outbound variants, which call `returnFields()` with `DOC_REF` = the traversal loop variable, e.g. `vertex`), the generated relation `LET` sub-queries started from an **unbound** `doc_`-prefixed alias (`doc_vertex`) while the `RETURN` projection used the bare loop variable (`vertex`) ‚Äî the two diverged, and ArangoDB raised `collection or view not found: doc_vertex`. The cause: the prepared-fields branch of `returnFields()` prefixed the document reference with `AQL::DOC_PREFIX` before handing it to `buildVariables()`, but nothing binds a `doc_<ref>` loop in that path (unlike the facet helpers, which open their own correlated `FOR doc_<key>`), and the sibling `RETURN` projection (`aqlFields()`) used the un-prefixed reference. The branch now passes `$docRef` verbatim to both, so the relation `LET`s and the projection anchor on the **same** variable ‚Äî exactly as the `*` (whole-document) branch already did. In a main query (`DOC_REF` = `doc`) the reference is unchanged, so `list()` / `get()` output is identical.

- **`Filter::EDGES_COUNT` now composes inside a vertex traversal.** A model declaring a `Filter::EDGES_COUNT` field worked in `list()` / `get()` but raised an `AQL: variable 'vertex' is assigned multiple times` error the moment that model was the **target** of a traversal (`Edges::getVertices()` / `getOutboundVertices()` / `getInboundVertices()`). The reason: the count sub-query, built by `buildEdgeCountVariable()`, hard-coded its inner loop as `FOR vertex IN <dir> <start> <edge> RETURN vertex`. In `list()` / `get()` the outer model loop is `doc`, so no clash ‚Äî but a vertex traversal's outer loop is already named `vertex` (`AQL::VERTEX`), and when the target model re-emitted its count `LET` over that traversed vertex, the inner `FOR vertex` collided with the outer one. The inner count loop now uses a **unique** variable derived from the `LET` name (`<name>_v`, itself a `randomKey` in the live flow) instead of the shared `vertex`; the `LET` variable name ‚Äî i.e. the projected field's public API ‚Äî is unchanged, and the count semantics are identical. - **Analyzer diff ‚Äî a declared `pipeline` no longer drifts on its sub-analyzer defaults.** `analyzerDiff()` compares a declaration to the server with **subset** semantics: it iterates only the declared keys, so the server defaults a declaration omits (a `norm` declared `{ locale }` reads back `{ locale, case, accent }`) are ignored ‚Äî but only because the top-level loop walks declared keys one by one.   - **Tests:** unit (`AnalyzerManagementTraitTest` ‚Äî a declared `norm` ‚Üí `ngram` pipeline IN_SYNC against the server-defaulted chain; real drift still caught on a changed sub-analyzer type, a reordered chain, a different chain length, and a changed sub-property) + live (`PipelineAnalyzerSyncIntegrationTest` ‚Äî on a real server the created pipeline re-diffs IN_SYNC despite the server filling every sub-analyzer default, and a `RawAnalyzer` dumped from that server round-trips too).

- **Wrap a reference under a named key ‚Äî `Filter::WRAP`.** The symmetric companion of `Field::SCOPE`: where `Field::SCOPE` hoists a **scalar** edge metadata flat next to the vertex fields, `Filter::WRAP` nests an **object** ‚Äî it wraps the whole current reference under a named key, instead of flattening its fields at the root.   - `ArangoMaskingTrait` now consumes `oihana\masking\‚Ä¶` and owns the ArangoDB identity list itself (`SYSTEM_ATTRIBUTES` = `_key`, `_id`, `_rev`, `_from`, `_to`, sourced from `org\schema\constants\Schema` ‚Äî no magic strings), handing it to `maskDocument()` as the protected attributes so those fields stay untouched on every edition.

### Fixed

- **A field declared with the `identity` analyzer no longer reports a permanent false drift.** `buildViewLink()` emitted an explicit `{"analyzers":["identity"]}` for such a field, but `identity` is the link's default analyzer, so the server stores the field as `{}` (the redundant mention is dropped) ‚Äî `viewDiff()` then compared declared `["identity"]` to server `{}` and reported `DRIFTED` forever (noise on every deploy, `doctor --apply` "repairing" in a loop). The link builder now omits the `analyzers` key when a field's resolved analyzer equals the link default (computed explicitly, not hard-coded), so the declared shape matches the stored one and `viewSync()` stops sending redundant data. Empty field nodes are serialized as JSON objects (`{}`) on the wire ‚Äî `json_encode([])` would emit `[]`, which the server rejects as an invalid link definition.
  - **Tests:** unit (`ViewSearchTest` ‚Äî identity field emits no `analyzers`, default-analyzer fields too, `text_fr` keeps it; `ViewTest` ‚Äî empty field node serialized as `{}`) + live (`ViewSyncIntegrationTest::testIdentityFieldDoesNotDrift` ‚Äî round-trip idempotent, no drift on re-sync, the identity field stays token-exact searchable).
  - **Docs:** FR/EN `db/search-views.md` (token-exact "code" field note in the per-field analyzer section).

- **`doctor` no longer reports the migrations tracking collection as an orphan.** No model declares the tracking collection (`migrationsCollection`, default `migrations`), yet both `migrate` and `doctor --apply` write their journal there ‚Äî so after a single `migrate` or `doctor --apply`, every following `doctor` flagged it as an orphan (and offered it to `--prune`). `doctorOrphans()` now excludes it by its **configured name**, which also keeps it out of the `--prune` selection. Views orphan detection is unchanged.

### Documentation

- **`?filter=` relation traversal is now documented (edges / joins / nested documents).** Hierarchical filter keys (`address.city`, `company.name`, `employee[*].salary`) ‚Äî already supported by `HasHierarchicalFilter` ‚Äî were only mentioned in passing. A new "Filtering through relations" section in FR/EN `db/filter.md` covers the mental model, the DI declaration of the three flavours (`Filter::DOCUMENT`, `Filter::JOIN`/`JOINS`, `Filter::EDGE`/`EDGES` with their `AQL::JOINS`/`AQL::EDGES` maps), the generated AQL (existence check `LENGTH(‚Ä¶ LIMIT 1 RETURN 1) > 0`), the cardinality ‚Üî `[*]` notation rule, multi-level traversal, AND/OR combination, `alt` on leaves, and performance/leniency ‚Äî plus the key scope caveat (the filter targets the document **at the far end** of the relation, never the edge's own metadata).
- **Corrected the misleading "relations ‚Üí `?facets` only" guidance.** FR/EN `db/search-and-filtering.md` (comparison table, "when to use which" bullets, and the "same intent" examples) stated relation filtering was facets-only; it now reflects that `?filter=` can traverse relations via a path, reserving `?facets=` for UI facets and aggregates, with cross-links to the new filter section.

### Fixed

- `aqlCollect()` could emit invalid AQL combining `AGGREGATE` and `WITH COUNT INTO`, which are mutually exclusive: when both keys were supplied it appended both clauses. `AGGREGATE` now takes precedence and `WITH COUNT INTO` is dropped (to count alongside aggregates, express the count as a `LENGTH(1)` aggregate). The only caller (`EdgesCountTrait`, which passes `WITH COUNT` alone) is unaffected. The companion `aqlCollectReturn()` mirrors the same rule.
- `aqlUpsertExpression()` never rejected an `init` that defined **both** `FILTER` and `SEARCH`: the second guard repeated the first one's condition (`!isset($filter) && !isset($search)`) instead of detecting the both-defined case, so it was dead code and a conflicting init silently fell through to `$search ?? $filter` (SEARCH wins). The guard is now `isset($filter) && isset($search)` and throws `FILTER and SEARCH cannot be defined at the same time.`, matching the documented exclusive `UPSERT [ searchExpression | FILTER filterExpression ]` syntax. The two guards are split into independent `if` clauses. Calls passing exactly one of the two (every existing usage) are unchanged. Characterized in the new `AqlUpsertExpressionTest`.

The `$limit` parameter also accepts `int|string|null` and ignores non-numeric/non-positive values instead of raising a `TypeError` at call time (e.g. a `remove` filter built from a query string). `FilterFunction::REMOVES` (`removes`) keeps using `REMOVE_VALUES`.
- `ArangoCommand` was uninstantiable whenever its `init` array omitted the `directory` key: the constructor's fallback `$this->directory = $init[‚Ä¶] ?? $this->directory` read the `$directory` property *before initialization* (declared `public ?string $directory ;` with no default in `ArangoDumpAction` / `ArangoRestoreAction` / `ArangoListDumpsAction`), throwing `Error: Typed property ‚Ä¶ must not be accessed before initialization`. The triplicated declaration is extracted into a single `oihana\arango\commands\traits\DirectoryTrait` (property defaulted to `null` + an `initializeDirectory()` initializer matching the `initializeEncrypt()` / `initializePassphrase()` pattern), the three actions now `use DirectoryTrait`, and the constructor calls `$this->initializeDirectory( $init )`. Behavior for configs that already supplied a directory is unchanged.
- `ArangoDB` fa√ßade hardening: `getCursor()` was typed `: Cursor` (non-nullable) but `$cursor` is null before the first `execute()` and after `streamDocuments()` resets it, so calling it in those states threw a `TypeError` ‚Äî it now returns `?Cursor`. `getExtra()` / `getFoundRows()` likewise dereferenced a possibly-null cursor and now return `[]` / `0` when no query has run. The constructor caught only `ArangoException`, but the client builder throws `InvalidArgumentException` for an invalid config (e.g. empty endpoint), so build failures were never logged; the `catch` is widened to `Throwable` (logged, then rethrown unchanged).
- `CollectionManagementTrait` (`ArangoDB` fa√ßade) called an **undefined** `toLegacyException()` in the `catch` blocks of `collectionRename()`, `getIndex()` and `getIndexes()` ‚Äî a leftover shim from the migration off the official ArangoDB-PHP fork. On any client error those three methods crashed with `Error: Call to undefined method` instead of surfacing the failure. The dead `try/catch` wrappers are removed so the `oihana\arango\clients\exceptions\ArangoException` propagates directly, and the redundant `ArangoException as ArangoClientException` import alias (both names resolved to the same class) is dropped. The boolean / log-and-swallow methods are unchanged.
- `PropertyController` PATCH in `raw` mode (`init[Arango::RAW] = true`) returned `null` instead of the submitted property value: the handler read the prepared payload with object-property syntax (`$payload->{$this->property}`), but `PayloadsTrait::propertyPayload()` yields an **array** (`[ property => value ]`), so the access always resolved to `null`. It now reads the value with `oihana\core\accessors\getKeyValue( (array) $payload , $this->property )`, which handles arrays and objects uniformly. Non-`raw` mode (which reloads via `model->get()`) is unchanged. Characterized then fixed in `PropertyControllerTest`.
- `aqlFields` `Field::QUOTED`: it double-quoted the key used for BOTH the output label and the attribute access, emitting invalid AQL like `"my-key":doc."my-key"` (an attribute with special characters cannot be reached with double quotes ‚Äî confirmed rejected by ArangoDB with a syntax error). The attribute access now uses backticks (`"my-key":doc.`my-key``), the valid form; only the output label is double-quoted. Cases pairing `Field::QUOTED` with `Field::NAME` or `Field::ALTERS` are unchanged. Characterized then fixed in `AqlFieldsTest`.
- AQL `like()` helper: its third parameter was named `$caseSensitive` but maps to ArangoDB's `LIKE(text, search, caseInsensitive)` third argument, so it did the opposite of its name (`true` produced a case-INSENSITIVE match). Renamed to `$caseInsensitive` (default `false` = case-sensitive, matching AQL). The emitted AQL is unchanged for every call path ‚Äî `?search` stays case-insensitive and the `like` filter-function alter keeps its output ‚Äî only the (now honest) parameter name and the docs change. Added the previously missing direct unit test, and fixed the stale `SearchTraitTest` comment that described the predicates as "case-sensitive".
- `command:arangodb` document `upsert` now honors the `removeKeys` exclusion list: it was passed under the `excludes` key, which the upsert query never reads, so the exclusion silently had no effect.
- `ArangoCommand` registered the wrong signals for event dispatch: it passed `(int) [SIGINT, ‚Ä¶]` (which casts to `1`) to the variadic `setSignalsToDispatchEvent()`. The intended signal list is now passed.
- Removed the unused `ignoreCollection` arangodump option (`ArangoDumpOptions::$ignoreCollection` / `ArangoDumpOption::IGNORE_COLLECTION`): no such flag exists in `arangodump`, and exclusion is resolved client-side as a `--collection` complement. The `--ignore-collection` CLI flag is unchanged.
- `ArangoConfigTrait` now validates the connection settings with exceptions instead of `assert()` (which is compiled out when `zend.assertions=-1`), so a missing database/endpoint/username never silently flows on as null. Properties are initialized; an empty password remains valid.
- Declare the previously-undeclared `ext-pcntl` requirement in `composer.json` ‚Äî `ArangoCommand`'s signal handling (`getSubscribedSignals()` / `handleSignal()` / `setSignalsToDispatchEvent()`) relies on the `SIGINT` / `SIGTERM` / `SIGUSR1` / `SIGUSR2` / `SIGALRM` constants provided by the extension.
- Security: `command:arangodb dump` / `restore` no longer run `arangodump` / `arangorestore` through a shell. The command was assembled as a single string (option values quoted only with `json_encode`) and passed to `system()`, so a value containing shell metacharacters (password, database, endpoint, directory, collection name) could break the command or inject arbitrary shell. The processes are now executed via `proc_open()` with an explicit argument vector (no shell), so values are passed verbatim. New `ArangoProcessTrait` (`optionsToArguments()` + `runProcess()`), covered by `ArangoProcessTraitTest` including a metacharacter-stays-one-argument unit test and a no-shell probe.
- `command:arangodb restore --last` could never locate the archive: it set the input file to the bare filename returned by `findFiles()` (whose `FILTER` yields `$file->getFilename()`), so `assertFile()` failed unless the current working directory happened to be the dump directory. The `--last` branch now prepends the dump directory, matching the interactive branch.

- `command:arangodb restore --date <date>` could never locate the archive. The by-date suffix was built as `Char::DASH . $database . $shouldEncrypt ? FileExtension::TAR_GZ_ENCRYPTED : FileExtension::TAR`, but `.` binds tighter than the ternary, so PHP evaluated `(Char::DASH . $database . $shouldEncrypt)` as the condition ‚Äî always a non-empty (truthy) string. The suffix therefore always collapsed to `.tar.gz.enc` and silently dropped the `-{database}` segment, so a non-encrypted restore-by-date never matched its `{date}-{database}.tar.gz` file. The non-encrypted branch also used `FileExtension::TAR` (`.tar`) instead of `FileExtension::TAR_GZ` (`.tar.gz`), which the dump action actually produces. The suffix logic is now isolated in the testable `ArangoRestoreAction::getArchiveFileSuffix()` helper and covered by `ArangoRestoreActionTest`.

- `Facet::ARRAY_COMPLEX` was broken at query time: it iterated an undefined variable (`FOR doc_workshops IN resultworkshops`, with no separator) and built its filter on the out-of-scope `result` document, so the facet never ran. It now iterates the in-scope embedded array (`doc.<key>`) and returns `1`. Confirmed against a live ArangoDB.
- `Facet::EDGE_COMPLEX` used a malformed loop variable (`AQL::DOC_REF . $key` ‚Üí e.g. `docRefnumbers`) instead of the sibling convention `doc_<key>`; it now reads `doc_<key>` like the other builders, and gained per-field multi-value (OR) plus per-value inline negation (`!=`) on the same traversed vertex.
- Date filters with a valid `tz` always threw: the timezone-conversion path passed the bound placeholder (`@tz`) to `dateLocalToUTC()`, whose own timezone re-validation then rejected it (the literal timezone is already validated upstream by `isValidTimezone()`). The re-validation is now disabled there (`$throwable = false`), so `?filter={...,"tz":"Europe/Paris"}` emits `DATE_LOCALTOUTC(@v,@tz)` instead of crashing ‚Äî and the new date `between` honors `tz` on both bounds.

### Security

- `?sort=` in open mode (no `sortable` whitelist) interpolated the raw URL key into `doc.<key>` without validation ‚Äî an AQL injection vector (a model that exposes `?sort=` without configuring `$sortable`, the default). `SortTrait::prepareSort()` now validates each open-mode sort key with `assertAttributeName()`; the whitelisted path was already safe. Legitimate dotted paths (`address.city`) still sort; an unsafe key throws `ValidationException`.
- `aqlFields()` (the projection builder) interpolated the source attribute (`Field::NAME ?? key`) into `doc.<attr>` without validation. Although projection fields are server-config today (`?fields=` is not wired to any controller), the bare source is now guarded by `assertAttributeName()` as defense-in-depth, so a future URL-driven projection cannot inject AQL. The two legitimate non-identifier paths are preserved: the `Field::QUOTED` escape hatch (a special key reached via backticks, `doc.`my-key``) and numeric keys.
- HTTP facets enforce a strict trust contract: only bound values (`@bind`) are user-controlled. Operators are whitelisted (`getAlias`, falling back to the type default), facet keys are whitelisted against the model's `Arango::FACETS` (an unknown key is ignored), and the URL-provided sub-field names of complex facets are validated by `assertAttributeName` before being interpolated ‚Äî a dangerous name fails the facet (logged and skipped). `alt` function names are likewise whitelisted (an unknown function is a no-op), so no `?filter=` / `?facets=` input can inject AQL.
