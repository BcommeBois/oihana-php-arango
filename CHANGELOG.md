# Changelog

All notable changes to **oihana/php-arango** are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Fixed

- **`command:arangodb` now reads the `analyzers` (and `searchAliasViews`) init keys.** The `ArangoCommand` constructor wired only some of its declarative registries (`models`, `collectionIndexes`, …) but silently dropped the `analyzers` and `searchAliasViews` init keys: a host-declared custom-analyzer registry never reached `getAnalyzerDefinitions()` (it stayed `[]`), so `arango:analyzers --sync` reported « No analyzers configured » and created nothing, the `doctor` analyzers pass reconciled nothing, and a View referencing the analyzer stayed « invalid: analyzer not found ». The same omission left a host-declared `search-alias` view registry empty for the `views` action. Both keys are now read in the constructor like the other registries (`$this->analyzers = $init[ArangoCommandParam::ANALYZERS] ?? $this->analyzers`). **Strictly backward-compatible**: without the key the registry stays its empty default. The trait, property, getter and consuming actions all already existed — only the one constructor line per key was missing.
  - **Tests:** unit (`ArangoCommandTest` — each registry is wired from its init key, a lone `AnalyzerDefinition` is normalized, a missing key defaults to `[]`, plus a regression guard asserting the constructor wires *every* declared registry — `analyzers` / `collectionIndexes` / `models` / `searchAliasViews` — so the next added registry cannot slip through unwired) + live integration (`AnalyzerCommandIntegrationTest` — an analyzer declared through the **real command's** `analyzers` init key is reported missing, created by `--sync` on a real server, then in sync; previously this run printed « No analyzers configured »).

### Added

- **Federated search — type-aware model registry (foundation).** The `FederatedSearch` `models` registry now accepts, besides the direct `collection => 'model.id'` form, a **composite** form for a **polymorphic** collection — one served by several scoped models, e.g. `organizations` → `Customer` / `Provider` / `Subsidiary` with no generic model: `'organizations' => [ FederatedSearchParam::DISCRIMINATOR => 'additionalType', FederatedSearchParam::MAP => [ '<type>' => 'model.x', … ], FederatedSearchParam::FALLBACK => 'model.org' ]`. The registry is normalised on construction — the discriminator field defaults to `additionalType`; the map keeps only non-empty `type => model-id` pairs in **declaration order (= resolution priority)**; an entry with neither a map nor a fallback is dropped. New resolution helpers: `resolveModelId(spec, type)` (a scalar type **or** an array of types, map-order priority, fallback model-id or null) and `resolveModelInstance(modelId)` (split out of `resolveModel`). New `FederatedSearchParam` constants (`DEFAULT_DISCRIMINATOR` / `DISCRIMINATOR` / `MAP` / `FALLBACK`). **Strictly backward-compatible**: a fully direct registry behaves exactly as before. *(This lands the registry + resolution; the `rebuild()` routing that consumes it — reading each hit's type and grouping the rebuild per resolved model — follows in a subsequent change.)*
  - **Tests:** unit (`FederatedSearchTest` — composite normalisation incl. discriminator default and null fallback, map cleaning, the unresolvable-entry drop; `resolveModelId` — direct model, scalar-type mapping, multi-typed document resolved by **map order** not array order, fallback default, and the no-match-no-fallback drop).

- **Facet counts — count by an object-array sub-field (`offers[*].priceCurrency`).** A `?facetCounts=` dimension whose `Facet::PROPERTY` carries the `[*]` array-expansion marker (e.g. `offers[*].priceCurrency`) now counts a sub-field of an *embedded array of objects*: the count sub-query unwinds the array and projects the sub-field, so each element is tallied as its own bucket (`FOR item IN doc.offers COLLECT value = item.priceCurrency WITH COUNT INTO count …`). This brings facet counts to **notation parity** with `?filter=` and View search, which already accept array-expansion paths — counts still only tally (never restrict) and still inherit the list's filters. The `[*]` marker is the signal: it overrides the declared `Facet::FIELD` / `Facet::IN` type; **each `[*]` is one `FOR` hop**, so nested object arrays are counted per leaf element (`offers[*].prices[*].currency` → `FOR item IN doc.offers FOR item2 IN item.prices COLLECT value = item2.currency …`, with the path between two markers descending within the element, `a[*].b.c[*].d`); the container and sub-field are guarded by `assertAttributeName` (a dangerous path fails the facet, never reaching the AQL); `offers[*]` with no sub-field counts the element itself. **Strictly backward-compatible**: a property without `[*]` builds the exact same count sub-query as before.
  - **Tests:** unit (`FacetCountsQueryTraitTest` — sub-field unwind & projection, the `[*]` marker overriding a `FIELD` facet, the no-sub-field form projecting the element, filter inheritance, multi-level nested unwind, an intermediate path between two hops, dangerous sub-field guarded) + live integration (`CollectIntegrationTest` — counting `offers[*].priceCurrency` and the nested `offers[*].prices[*].currency` over a seeded object array yields the right per-element / per-leaf buckets on a real server).
  - **Docs:** FR/EN `db/facets.md` (new "Counting an object-array sub-field (`[*]`)" subsection: declaration, generated AQL, returned JSON, and the counts-never-restrict / notation-parity framing).

- **ArangoSearch — precise n-gram search by similarity threshold (`NGRAM_MATCH`).** A `Search::FIELDS` entry can now declare `Search::NGRAM` — an `ngram` Analyzer queried through `NGRAM_MATCH` (a tunable **similarity threshold**) instead of the loose `IN TOKENS` of `Search::ANALYZER`. Where the multi-analyzer `IN TOKENS` autocomplete matches on *any* shared fragment (and relies on `BM25` to rank), `NGRAM_MATCH` requires a **fraction** of the n-grams to match, excluding weak matches inside the `SEARCH` itself — so a partial term still matches while a whole word no longer drags in records that merely share fragments. Two forms: the analyzer name alone (server-default threshold `0.7`) or `[ Search::ANALYZER => …, Search::THRESHOLD => 0.6 ]` (a float in `[0.0, 1.0]`, validated). The ngram Analyzer is merged into the field's indexed analyzers (`buildViewLink`) and checked for existence (`viewDiff`); `prepareViewSearch` emits its own `ANALYZER(NGRAM_MATCH(path, @term, threshold, "analyzer"))` branch with the term kept bound and the field boost applied. `Search::NGRAM` is disjoint from `Search::ANALYZER` (the `IN TOKENS` recipes); `FUZZY` / `PHRASE` do not apply to it. **Strictly backward-compatible**: no `Search::NGRAM` → link and query unchanged.
  - **Tests:** unit (`ViewSearchTest` — ngram analyzer merged into the link, `NGRAM_MATCH` branch emitted, shorthand omits the threshold, field boost carried, out-of-range threshold rejected) + live integration (`MultiAnalyzerFieldIntegrationTest` — with a `min == max` trigram analyzer and a 0.5 threshold, the whole word « atelier » matches only « atelier » and excludes « ferronnerie », the precision the loose `IN TOKENS` combo lacks).
  - **Docs:** FR/EN `db/search/per-field-options.md` (new "Precise autocomplete (`NGRAM_MATCH`)" section) and `db/analyzers.md` (the `min == max` / `preserveOriginal: false` recommendation for `NGRAM_MATCH`).

- **ArangoSearch — multiple Analyzers per searched field.** A `Search::FIELDS` entry's `Search::ANALYZER` now accepts a **list** of Analyzers in addition to a single one, so the same field can be indexed (and queried) through several recipes at once — e.g. a `text` analyzer for whole-word search **plus** an `ngram` analyzer for autocomplete (`Search::ANALYZER => ['text_fr', 'autocomplete']`). The View link stores the field's full analyzers list, and the `SEARCH` query emits one `ANALYZER(…)` branch per analyzer (`OR`-ed); the field's other options (`BOOST` / `FUZZY` / `PHRASE`) apply to each branch. The View-level `Search::ANALYZER` stays a single value (the inherited default). **Strictly backward-compatible**: a single-string analyzer behaves as a one-element list and produces byte-for-byte the same link and query as before.
  - **Tests:** unit (`ViewSearchTest` — multi-analyzer link, one query branch per analyzer, and a one-element list identical to the single form) + live integration (`MultiAnalyzerFieldIntegrationTest` — a field indexed with `[text_fr, text_en]` matches a French-stemmed and an English-stemmed query, and a field indexed with `[text_en, <ngram>]` answers both a whole word and a partial term on a real Community server).
  - **Docs:** FR/EN `db/search-views.md` (new "Multiple Analyzers per field (autocomplete)" section: list declaration, generated link and SEARCH, the loose-fragment / scoring note, and the must-declare-the-ngram-first caveat).

- **ArangoSearch — `ngram` analyzer type (substring / autocomplete).** A custom n-gram analyzer can now be declared like the existing `text` / `norm` / `stem` / `identity` ones, via the new `AnalyzerType::NGRAM` and the `NgramAnalyzer` value object. An n-gram analyzer emits the substrings of its input between a `min` and `max` length (optionally keeping the original token, with `startMarker` / `endMarker` / `streamType` options) — the building block of substring / "as-you-type" autocomplete, typically paired with a `text` analyzer on the same field. `AnalyzerField` gains the `startMarker` / `endMarker` / `streamType` wire names. The `arango:analyzers` lifecycle (`--diff` / `--sync` / `--fix` / `--prune`) and `doctor` handle it with no change, since they consume `AnalyzerOptions` generically.
  - **Tests:** unit (`NgramAnalyzer` payload shapes; `AnalyzerType` now recognizes `ngram`; new `AnalyzerField` constants) + live integration (`NgramAnalyzerIntegrationTest` — the declared n-gram analyzer is created, round-trips `IN_SYNC` with no false drift on its properties, and powers a substring search where a partial term matches a longer value on a real Community server).
  - **Docs:** FR/EN `db/analyzers.md` (new `NgramAnalyzer` section: declaration, prefix-vs-substring difference with `edgeNgram`, the case-folding caveat, and the multiple-analyzers-per-field pairing).

- **ArangoSearch — search sub-fields of arrays of objects (`contactPoints[*].email`).** A `Search::FIELDS` path may now reach into a sub-field of an *array of objects* using the `[*]` expansion marker (e.g. `contactPoints[*].email`, `tags[*].label`, `members[*].name`), the same notation already used by `?filter=`. A `?search=` term then matches whenever it appears in **any** element of the array — a **non-correlated** search that works on ArangoDB **Community** and needs no Enterprise `nested` flag. The `[*]` marker is a developer-facing notation, stripped internally on **both** surfaces: the View link declares the flat nested path (`contactPoints` → `email`, so ArangoSearch descends into the array itself), and the `SEARCH` query emits the flat path too (`doc.contactPoints.email IN TOKENS(...)`) — the `SEARCH` grammar rejects array-expansion expressions, while the flat form already matches every element. Every marker is removed whatever the nesting depth (`employees[*].contactPoints[*].email`). Because `buildViewLink()` is the single place that turns a declared path into a link shape — and the one `viewDiff()` compares against the server — the stripped form is declared, created and compared everywhere, so no permanent false drift is reported after `viewSync()`. Declared paths are validated on their stripped form (`assertAttributeName()`) at both stages, turning a malformed declaration into a clear failure instead of a silently broken index. Per-field options (`Search::ANALYZER` / `FUZZY` / `PHRASE` / `BOOST` / `LANG` / `REQUIRES`) keep working on a `[*]` field; `trackListPositions` stays off (the default suits a non-correlated search). New `stripArrayExpansion()` helper (`db/helpers`, reusing the filter-side `Operator::ARRAY_EXPANSION` marker). **Strictly backward-compatible**: a field without `[*]` (`name`, `description.fr`) builds the exact same link and query as before.
  - **Tests:** unit (`ViewSearchTest` — link stripping incl. multi-level and per-field analyzer, query emitting the flat path, per-field options on an expanded field, guardrail rejection on a malformed path) + the `stripArrayExpansion()` helper's own test + live integration (`ViewArraySubfieldIntegrationTest` — a token inside a flat array of objects and a token inside a two-level nested array both match on a real Community server, and `viewDiff()` stays `IN_SYNC` across a `viewSync()` no-op, proving the stripped link never drifts).
  - **Docs:** FR/EN `db/search-views.md` (new "Object-array fields" section: declaration with `[*]`, generated link and SEARCH, the non-correlated/Community/no-Enterprise note, and the multi-level behavior).

## [1.4.0] - 2026-06-21

### Added

- **Relation quantifiers — `quant` on edge & join filters.** The `?filter=` array quantifier (`any` / `all` / `none` / an integer `n`) now also applies to `Filter::EDGE(S)` / `Filter::JOIN(S)` hierarchical filters, so a relation filter can express more than the hardcoded existence `LENGTH(...) > 0`: `none` → « no linked match » (`LENGTH(...) == 0`), an integer `n` → « at least n linked » (`LENGTH(...) >= n`, counted without `LIMIT`), and `all` → « every linked vertex satisfies the leaf » i.e. no vertex violates it (`LENGTH(FOR … FILTER !(<leaf>) LIMIT 1 RETURN 1) == 0`, vacuously true with no relation — consistent with the array surface and AQL `ALL`; combine with `any` for « all and at least one »). The same vocabulary and the same `ValidationException` as the array surface are reused — no new term, no new filter type. Pure existence/absence on a relation key **with no leaf condition** (`members[*]` [+ `quant`]) is now supported too — single-segment relation keys, previously silently dropped, are routed to the hierarchical builder. On a join, the structural key condition (`j._key == doc.x`) always stays positive; only the leaf is negated for `all`. Guardrails: `all` without a leaf condition and `n < 1` are rejected (`ValidationException`; use `none` for « no match »). **Strictly backward-compatible**: an absent `quant` keeps the historical `LENGTH(...) > 0`. New `TraversalQuantifier` value object and `resolveTraversalQuantifier()` helper (relation counterpart of `resolveQuantifier`; `n` inlined as an int, injection-safe).
  - **Tests:** unit (`HasHierarchicalFilterTest` — edge & join `none` / `n` / `all` / default `any` with and without a leaf, pure existence/absence, and the guardrails: `n=0`, unknown `quant`, `all` without a leaf rejected) + live (`FilterQuantRelationsIntegrationTest` — over a seeded organizations→members edge and a customers→companies join, each quantifier selects the right documents on a real server, including the vacuously-true `all` and dangling/absent references classified as absence).
  - **Docs:** FR/EN `db/filter.md` (new "Quantifiers on edges/joins" subsection of "Filtering through relations": semantics table, URL + generated AQL, guardrails, and the cross-reference to the array `quant` section).

### Documentation

- Custom phpDocumentor template (`.phpdoc/template/`) extending the default one with a README-based landing page (logo, badges, quick links) and the namespace table of contents moved to the bottom of the index.
- A GitHub Actions workflow (`.github/workflows/docs.yml`) that builds the API documentation and deploys it to GitHub Pages on every push to `main`; the README now links to the published reference (https://bcommebois.github.io/oihana-php-arango).
- The landing-page logo is referenced from `assets/images/` via its raw GitHub URL, so the generated `docs/` no longer duplicates the repository assets.

### Changed

- Dependencies: replaced `oihana/php-system` with the focused packages actually consumed — `oihana/php-controllers`, `oihana/php-routes`, `oihana/php-init`, `oihana/php-logging`, `oihana/php-models` and `oihana/php-traits` (namespaces `oihana\controllers`, `oihana\routes`, `oihana\init`, `oihana\logging`, `oihana\models`, `oihana\traits`). Drops the heavy Slim/Twig/Symfony stack that `php-system` pulled in. No code or public-API change.
- Tests: the relocated `DateTrait` is now imported from `oihana\traits\DateTrait` instead of `oihana\date\traits\DateTrait` (moved out of `php-system` into `php-traits`) in `ArangoDumpActionTest`, `ArangoRestoreActionTest` and `DumpRestoreIntegrationTest`.

## [1.3.0] - 2026-06-20

### Added

- **Federated search — the read-only HTTP triplet (Lot C5).** The federated engine is now reachable over HTTP through a read-only triplet mirroring the documents one (`route → controller → model`), except the controller holds a `FederatedSearch` engine instead of a single-collection model. New `FederatedSearchController` (new `oihana\arango\controllers\` member) extends `Controller` and exposes a **single** `search()` action: it reads `search` / `limit` / `offset` / `skin` from the request (reusing the standard `PrepareParam` chain and the capability-aware overrides from `DocumentsControllerCapabilitiesTrait`), resolves the engine from a DI service id (`FederatedSearchController::ENGINE`, or an instance) and renders `{ result: [ { collection, score, document }, … ], count, total, url }` — the `total` coming from `FederatedSearch::foundRows()` for "X results, page Y". **Security is already wired by the engine (lot C4):** exactly like `DocumentsController`, the controller resolves the request enforcer + subject resolver from the container, builds a request-scoped `Closure(string $subject): bool` (`PermissionAuthorizerTrait::buildPermissionAuthorizer()`) and poses it under `Arango::AUTHORIZER` in the engine `$init`; the engine consults it on its own to gate the searchable collections per request (fail-open without an enforcer). New generic `SearchRoute` (a thin `GetRoute` binding `GET` to a `search` action — reusable by any controller exposing one). New `FederatedSearchController::ENGINE` / `SEARCH` constants. The enforcer + subject-resolver container resolution, previously duplicated in `DocumentsController`, is factored into a shared `AuthorizationContextTrait` (`initializeAuthorizationContext()`) now used by both controllers — no behaviour change.
  - **Tests:** unit (`FederatedSearchControllerTest` — the action passes the prepared `search` / `limit` / `offset` / `skin` to the engine, reads them from the request query string, returns the engine rows through the null-response branch, wraps the PSR-7 response with the `total` from `foundRows()`, injects the authorizer under `Arango::AUTHORIZER` only when both the enforcer and the resolver are registered, resolves the engine from a service id and from an instance, returns an empty set without an engine, and fails gracefully when the engine throws; `SearchRouteTest` — registers the single `GET /search` bound to the `search` action, defaults the bound method to `search`, skips when the controller is unregistered).
  - **Docs:** FR/EN `db/federated-search.md` (new "Exposing it over HTTP" section — the triplet, the `GET /search` request/response, the DI wiring and the "permissions already wired like `DocumentsController`" note).

- **Federated search — per-collection permission gate (Lot C4).** A federated `search()` now only returns the collections the request is authorized to see. Each collection declares the permission `subject`(s) it requires via the new `FederatedSearchParam::REQUIRES` registry (`collection => subject` or an OR-list, mirroring `Field::REQUIRES`); a collection absent from the map is **public**. The gate is evaluated by the existing `isAuthorized()` helper against the request `Arango::AUTHORIZER` (fail-open without one) and — crucially — applied **at the *find* stage** via the search-alias view's `SEARCH … OPTIONS { collections: [...] }`: forbidden collections are excluded *before* the `LIMIT`, so the page, the ranking and `foundRows()` only ever cover allowed collections (no holes, correct total). `rebuild()` keeps a defensive gate. Field-level permissions stay enforced by each model at rebuild time. New `FederatedSearchParam::REQUIRES` init key.
  - **Tests:** unit (`FederatedSearchTest` — `find()` restricts the `OPTIONS { collections }` to the granted + public collections and excludes the denied ones, is fail-open without an authorizer, returns empty when none is allowed; `rebuild()` defensively drops an unauthorized collection; the `requires` registry keeps string / OR-list entries and drops the rest) + live (`SearchAliasViewIntegrationTest` — the same `search('shared')` under two different authorizers returns two different result sets, `foundRows()` honouring the restriction, on a real server).
  - **Docs:** FR/EN dedicated page `db/federated-search.md` (the engine, the two stages, configuration, pagination/total, skin, and the permission model with examples).

- **Federated search — the *rebuild* stage (Lot C3).** The engine's second stage completes the two-phase search: `FederatedSearch::rebuild()` takes a *find* page (`{ collection, key, score }`), groups the keys by collection, and re-hydrates each group **in one `list()` call per collection** (a `_key IN […]` filter — never one query per result) through the model that owns it (resolved via the collection → model registry), reusing that model's own projection pipeline (fields, joins, skin, permissions). The rebuilt documents are merged back **in score order**, each wrapped as `{ collection, score, document }`. A match whose collection is unregistered, whose service is not a `Documents`, or whose document the model does not return (filtered by its own rules) is dropped — the model stays authoritative. `search()` now runs *find* then *rebuild* end to end. The projection **skin is configurable**: the request `Arango::SKIN` overrides the engine default (`FederatedSearchParam::SKIN`), itself defaulting to `Skin::DEFAULT`. Pagination stays in the *find* stage (the LIMIT applies once to the global BM25 ranking), and the **total** number of matches before the LIMIT is exposed by `FederatedSearch::foundRows()` (the federated counterpart of the model `foundRows()`, via the search query's `fullCount`) for "X results, page Y". New `FederatedSearch::DOCUMENT` constant and `FederatedSearchParam::SKIN` init key.
  - **Tests:** unit (`FederatedSearchTest` — rebuild groups by collection / orders by score / wraps `{ collection, score, document }` / issues one `_key IN` `list()` per collection with the resolved skin; drops an unregistered collection, a non-`Documents` service, a key-less or model-omitted document; reads the key from array and object documents; skin request-overrides-default and default-fallback; `find()` requests `fullCount` and `foundRows()` exposes the total; `search()` runs find then rebuild) + live (`SearchAliasViewIntegrationTest` — `search('shared')` re-hydrates the ranked documents through real `Documents` models across both collections, `foundRows()` reports the total).

- **Wrapped vertices can carry joins too — `Filter::WRAP` + `Field::JOINS`.** Symmetric follow-up to the wrapped sub-edges: a `Filter::WRAP` field can also nest **joins** (stored-reference resolutions to another collection's documents) under the wrapped key, with the same top-level grammar — a `Filter::JOIN` / `Filter::JOINS` marker in the wrapped `Field::FIELDS` and a companion `Field::JOINS` map of resolutions. The join reads the stored attribute **from the wrapped vertex** (`vertex.role`), and `Field::EDGES` / `Field::JOINS` combine freely under the same key. `FieldsTrait::normalizeFieldDefinition` now also preserves `Field::JOINS` on the WRAP branch, `buildVariables()` threads the wrapped joins through the same recursion, and `Field::RAW` is rejected when combined with `Field::EDGES` **or** `Field::JOINS`. Purely additive and fully retro-compatible.
  - **Tests:** unit (`BuildVariablesTest` — a WRAP recurses into its edges and into its joins with the wrapped vertex as reference, a RAW wrap skips its declared relations; `BuildEdgeVariableTest` — a WRAP nests a join resolved from the wrapped vertex) + live (`WrapRelationsIntegrationTest` — a wrapped person exposes its `role` via a join resolved from the wrapped vertex, on a real server).
  - **Docs:** FR/EN `edges-joins-projection.md` ("joins too" note in the wrapped-relations section).

- **Wrapped vertices can carry their own relations — `Filter::WRAP` + `Field::EDGES`.** A `Filter::WRAP` field — which nests the traversal vertex under a named key (e.g. `subject`) instead of flattening it — can now also declare **sub-edges that start from the wrapped vertex** and nest **inside** the wrapped object, in a single query. A consumer projecting a list of links as `[{ subject: <vertex> }]` where the wrapped vertex is itself linked to a third entity by another edge (often reached `INBOUND`) can now expose `subject.worksFor` directly, with no flattening and no second round-trip. The declaration reuses the **exact top-level grammar**: the cardinality marker (`Filter::EDGE` / `Filter::EDGES` / `Filter::EDGES_COUNT`) lives in the wrapped `Field::FIELDS`, and the companion `Field::EDGES` map carries the sub-traversal definitions (`AQL::MODEL`, `AQL::DIRECTION` — both `OUTBOUND` and `INBOUND`, `AQL::FIELDS`, …). The whole existing edge machinery therefore applies verbatim — permission gating (`Field::REQUIRES`), sorts, counts and arbitrarily deep sub-projections all work, because each sub-edge is an ordinary `buildEdgeVariable` traversal whose backing `LET` is emitted by `buildVariables()` with the wrapped vertex as the traversal root. Purely additive and fully retro-compatible: a `Filter::WRAP` without `Field::EDGES` is unchanged. `Field::RAW` (verbatim reference) and `Field::EDGES` are mutually exclusive — there is no projected object to nest into — and combining them throws. `FieldsTrait::normalizeFieldDefinition` now preserves `Field::EDGES` through the model → query path on the WRAP branch.
  - **Tests:** unit (`BuildEdgeVariableTest` — a WRAP nests an `OUTBOUND` and an `INBOUND` sub-edge under the wrapped key, the sub-edge `LET` is emitted in the enclosing `FOR` scope and traverses from the wrapped vertex, a wrapped `Filter::EDGES_COUNT` nests the sub-traversal cardinality, a denied sub-edge (`Field::REQUIRES`) is dropped from both the `LET` and the projection, a wrapped sub-edge projects its own nested edge (`subject.worksFor.locatedIn`), a WRAP without edges is unchanged; `BuildVariablesTest` — a denied wrapped sub-edge emits no `LET`; `AqlFieldWrapTest` — `Field::RAW` combined with `Field::EDGES` throws) + live (`WrapRelationsIntegrationTest` — a wrapped person exposes its organization via an `INBOUND` sub-edge nested under the wrapped key, the generated nested `LET` parses and runs on a real server).
  - **Docs:** FR/EN `edges-joins-projection.md` (new "carrying the wrapped vertex's relations" section, with the neutral `account_has_identity` / `org_has_member` example and the generated AQL).

- **Federated search — the *find* stage (Lot C2).** The federated engine's first stage is wired: `FederatedSearch::find()` runs one scored SEARCH over the `search-alias` view and returns the matches as a flat list ranked by relevance, each row holding only its provenance and score — `{ collection, key, score }` — paginated. The query term is **bound** (never inlined) and tokenized per the declared spec: every `searchable` field is matched as `doc.<field> IN TOKENS(@search, "<analyzer>")`, OR-combined and wrapped in `ANALYZER(…)`, ranked by `BM25`, and the rows are projected through `MERGE(PARSE_IDENTIFIER(doc._id), { score })` so each carries its source collection and `_key`. The engine now resolves its `ArangoDB` façade (`Arango::DATABASE` — an instance or a container id) to run the query; an absent view, spec, database or term yields an empty result set. `search()` delegates to `find()` for now — the *rebuild* stage (per-collection re-hydration + score merge) lands in Lot C3. New `FederatedSearch::SCORE` / `FederatedSearch::DEFAULT_LIMIT` constants. Reuses the existing search helpers (`aqlScoredSearch`, `tokens`, `key`, `aqlBind`, `parseIdentifier`) — no query logic reinvented.
  - **Tests:** unit (`FederatedSearchTest` — `find()` builds the expected scored query (view, tokenized + analyzer-wrapped match, BM25 ranking, LIMIT, provenance return) with the bound term and returns the cursor rows, the default page size applies, an absent term / view / database / field yields an empty set, the database resolves from a container id, `search()` delegates) + live (`SearchAliasViewIntegrationTest` — `find('shared')` returns the ranked `{ collection, key, score }` of both collections on a real server).

- **Federated multi-collection search — engine skeleton & registry (Lot C1).** First piece of the federated search engine: one search bar over several collections (customers, products, sellers, places, …) returning a single list ranked by relevance. New standalone, container-aware `FederatedSearch` service (new `oihana\arango\search\` namespace) — deliberately **not** a `Documents` subclass, since it owns no single collection — carrying the engine configuration: the `search-alias` view to query, the federated search spec (fields + analyzer), and the **collection → model registry** (`collection => model-service-id`, the directory telling the engine which model rebuilds the documents of which collection, normalized to keep only non-empty string ↔ string entries). New `FederatedSearchParam` init-key enum (`VIEW` / `SEARCHABLE` / `MODELS`). - **Conditional field projection — `Field::WHEN` / `Field::ELSE`.** A scalar field can now guard its value behind a condition resolved **at query time** from the document's own attributes, rendered as an AQL ternary `<condition> ? <then> : <else>` (the SQL `CASE WHEN … THEN … ELSE …` shape). The key is always present — only the value switches; an absent value is `null`, never a removed key (no `MERGE`, intentionally out of scope). The condition mirrors the recursive `?filter=` grammar — a leaf (`'active'` truthiness, `[ 'visibility', 'public' ]` equality, `[ 'stock', 'gt', 0 ]` comparator, or the associative `FilterParam` `key`/`op`/`val`/`alt` form) combined with AND/OR/NOT groups and arbitrary nesting — but compiles **inline** (no bind variables, since the projection layer carries none): values are developer-declared literals escaped by `aqlValue()` (a value that looks like `doc.<attr>` is kept raw, enabling attribute-vs-attribute comparison), and attribute names are validated by `assertAttributeName()`. Only the infix comparators (`eq`/`ne`/`ge`/`gt`/`le`/`lt`/`in`/`nin`/`like`/`nlike`/`match`/`nmatch`) are supported; a function-form operator (`contains`/`sw`/`ew`/`regex`/…) throws. A leaf may carry an `alt` chain wrapping its operands (same `"lower"` / `{ key, val }` / `{ key, val:true }` mirror vocabulary as the flat filters). `Field::ELSE` is `null` by default, a literal, or an attribute reference (`[ Field::PROPERTY => 'basePrice' ]`); it composes with `Field::ALTERS` (which decorates the *then* branch) and `Field::NAME`. Applies to the default scalar projection only — `Field::WHEN` on a typed/structural filter (`EDGES`/`JOINS`/`DOCUMENT`/`MAP`/`URL`/…) throws an `UnsupportedOperationException`. New `Field::WHEN`/`Field::ELSE` options, the `aqlFieldConditional()` field builder and the `buildWhenCondition()` / `buildWhenLeaf()` / `resolveWhenElse()` helpers, dispatched from `aqlFields()` next to `Field::ALTERS`; `FieldsTrait::normalizeFieldDefinition` preserves `WHEN`/`ELSE` through the model → query path.

- **Per-type URL routing — `Field::PATHS`.** A `Filter::URL` field can now choose its path **at query time** from the document's type, instead of a single build-time `Field::PATH`. Placeholders (`{param}`) and the *base URL* are applied to **each** branch and to the fallback alike (the route values are pre-rendered into an AQL object literal). Declaring `Field::PATHS` without a `Field::PATH` fallback — or with an empty / non-associative map — throws an `UnsupportedOperationException` at build time, and the discriminant attribute is validated by `assertAttributeName` (injection guard). - **View-level search permission in ArangoSearch Views (Lot VF4c).** `Search::REQUIRES` can now also be declared on the `AQL::VIEW` block to gate the **whole** search (every field), on top of the per-field gate (Lot VF4b). The two levels combine with **AND** — a field is searched when the View-level requirement *and* its own are both satisfied (within a list, subjects combine with OR).
  - A denied View-level subject makes the whole `SEARCH` match nothing (`false`), whatever the per-field declarations.
  - `Search::REQUIRES` is the only **additive** facet: unlike boost/fuzzy/analyzer/lang/phrase (a field overrides the View), permissions accumulate (most restrictive wins).
  - **Tests:** unit (`ViewSearchTest` — View-level granted/denied, AND with the field level, OR over View-level subjects) + live (`ViewSearchIntegrationTest::testViewLevelRequiresGatesTheWholeSearch`).
  - **Docs:** FR/EN `db/search-views.md` (search-permissions section extended with the two levels + AND note).

- **Per-field search permissions in ArangoSearch Views (Lot VF4b).** A searched field can declare `Search::REQUIRES` (a permission subject or a list, OR semantics) — mirroring `Field::REQUIRES` for projections. A gated field joins the `SEARCH` only when the request authorizer (`Arango::AUTHORIZER`, already injected by the controller and consulted by `isAuthorized()`) grants at least one subject; an ungated field is always searchable.
  - **No leak by default:** if permissions remove every searched field, the emitted `SEARCH` is `false` (zero results) — it never falls back to searching everything or to the `LIKE` sweep, which would bypass the gate.
  - **Fail-open** without an authorizer (authorization layer disabled), consistent with projection/edge/join gating. `count()` / `facetCounts()` inherit the gating.
  - **Reuses existing plumbing:** no controller code — the same authorizer already injected for projection gates the search. Query-time only; the View still indexes every field (`buildViewLink()` / `viewDiff()` untouched).
  - **Tests:** unit (`ViewSearchTest` — authorized/denied, OR over subjects, deny-all → `SEARCH false`, fail-open, descriptor shape) + live (`ViewSearchIntegrationTest::testPerFieldRequiresGatesTheSearchableField`).
  - **Docs:** FR/EN `db/search-views.md` (new "search permissions" section + overview row) + `getting-started/arangosearch.md`. Backward-compatible — with no `REQUIRES` the AQL is unchanged.

- **Per-field exact-phrase bonus in ArangoSearch Views (Lot VF4a).** `Search::PHRASE` can now be declared **per field** inside a `Search::FIELDS` array entry, mirroring `Search::BOOST` / `Search::FUZZY`. The `PHRASE()` bonus can be enabled where it matters (a title) and left off elsewhere (`'code' => [ Search::PHRASE => false ]`).
  - **Resolution:** a field declaring `Search::PHRASE` wins (an explicit `false` opts that field out); a field with no `PHRASE` key inherits the View-level flag; with no global value the bonus is off.
  - **Code:** new `Search::PHRASE` key in the per-field spec; `prepareViewSearch()` resolves the bonus per field (composing with the per-field boost and Analyzer grouping).
  - **Tests:** unit (`ViewSearchTest` — per-field enable without global, opt-out under a positive global, descriptor shape) + live (`ViewSearchIntegrationTest::testPerFieldPhraseBoostsTheAdjacentMatch`).

- **Localized View search driven by `?lang=` (Lot VF3).** A searched field can be tagged with `Search::LANG` to mark it as a localized variant of an i18n object attribute (`'intro.fr' => [ Search::ANALYZER => 'text_fr', Search::LANG => 'fr' ]`). When the request carries an active language (`?lang=`, already used for the `TRANSLATE()` projection), only fields whose locale matches — plus locale-agnostic fields — take part in the `SEARCH`; without `?lang=` every field is searched.
  - **Guard:** an active language matching no field is ignored (all fields are searched) — the `SEARCH` is never emptied.
  - **No View change:** all locale sub-fields stay indexed (with their per-field Analyzer, Lot VF2); `?lang=` only narrows the query. `buildViewLink()` / `viewDiff()` / `viewSync()` are untouched.

  - **Docs:** FR/EN `db/search-views.md` (new "localized search" section) + `getting-started/arangosearch.md`.

- **Per-field Analyzer in ArangoSearch Views (Lot VF2).** `Search::ANALYZER` can now be declared **per field** inside a `Search::FIELDS` array entry, mirroring `Search::BOOST` / `Search::FUZZY`. A single View can index (and query) a `text_fr` field and a `text_en` field side by side (`'summary' => [ Search::ANALYZER => 'text_en' ]`).
  - **Resolution:** a field declaring `Search::ANALYZER` wins; otherwise it inherits the View-level Analyzer (`identity` by default).
  - **Both sides:** since the Analyzer is fixed at indexing time, `buildViewLink()` indexes each field with its Analyzer, and `prepareViewSearch()` groups expressions by Analyzer — one `ANALYZER(…)` wrap per group, `OR`-ed. With a single Analyzer the output is byte-for-byte the former AQL.

- **Per-field fuzzy tolerance in ArangoSearch Views (Lot VF1).** `Search::FUZZY` can now be declared **per field** inside a `Search::FIELDS` array entry, mirroring `Search::BOOST`.   - **Tests:** unit (`ViewSearchTest` — per-field override, opt-out under a positive global, inheritance, descriptor shape) + live (`ViewSearchIntegrationTest::testPerFieldFuzzyKeepsCodeExactWhileNameStaysTolerant`).
  - **Docs:** FR/EN `db/search-views.md` (new "per-field typo tolerance" section) + `getting-started/arangosearch.md`.

### Changed

- **Masking engine extracted to `oihana/php-masking`.** The portable PHP masking engine that lived under `src/oihana/arango/maskings/` is now provided by the dedicated, storage-agnostic [`oihana/php-masking`](https://github.com/BcommeBois/oihana-php-masking) library (added to `require`); the in-tree copy (the 10 maskers, `maskValue`/`maskDocument`/`maskDocumentNode`/`maskDocumentList`, `resolveMaskingRule`, `randomAlphaNumeric` and the `Masker`/`MaskingMode` enums) is removed. **No behaviour change.**

### Fixed

- **A field declared with the `identity` analyzer no longer reports a permanent false drift.** `buildViewLink()` emitted an explicit `{"analyzers":["identity"]}` for such a field, but `identity` is the link's default analyzer, so the server stores the field as `{}` (the redundant mention is dropped) — `viewDiff()` then compared declared `["identity"]` to server `{}` and reported `DRIFTED` forever (noise on every deploy, `doctor --apply` "repairing" in a loop). The link builder now omits the `analyzers` key when a field's resolved analyzer equals the link default (computed explicitly, not hard-coded), so the declared shape matches the stored one and `viewSync()` stops sending redundant data. Empty field nodes are serialized as JSON objects (`{}`) on the wire — `json_encode([])` would emit `[]`, which the server rejects as an invalid link definition.
  - **Tests:** unit (`ViewSearchTest` — identity field emits no `analyzers`, default-analyzer fields too, `text_fr` keeps it; `ViewTest` — empty field node serialized as `{}`) + live (`ViewSyncIntegrationTest::testIdentityFieldDoesNotDrift` — round-trip idempotent, no drift on re-sync, the identity field stays token-exact searchable).
  - **Docs:** FR/EN `db/search-views.md` (token-exact "code" field note in the per-field analyzer section).

- **`doctor` no longer reports the migrations tracking collection as an orphan.** No model declares the tracking collection (`migrationsCollection`, default `migrations`), yet both `migrate` and `doctor --apply` write their journal there — so after a single `migrate` or `doctor --apply`, every following `doctor` flagged it as an orphan (and offered it to `--prune`). `doctorOrphans()` now excludes it by its **configured name**, which also keeps it out of the `--prune` selection. Views orphan detection is unchanged.

### Documentation

- **`?filter=` relation traversal is now documented (edges / joins / nested documents).** Hierarchical filter keys (`address.city`, `company.name`, `employee[*].salary`) — already supported by `HasHierarchicalFilter` — were only mentioned in passing. A new "Filtering through relations" section in FR/EN `db/filter.md` covers the mental model, the DI declaration of the three flavours (`Filter::DOCUMENT`, `Filter::JOIN`/`JOINS`, `Filter::EDGE`/`EDGES` with their `AQL::JOINS`/`AQL::EDGES` maps), the generated AQL (existence check `LENGTH(… LIMIT 1 RETURN 1) > 0`), the cardinality ↔ `[*]` notation rule, multi-level traversal, AND/OR combination, `alt` on leaves, and performance/leniency — plus the key scope caveat (the filter targets the document **at the far end** of the relation, never the edge's own metadata).
- **Corrected the misleading "relations → `?facets` only" guidance.** FR/EN `db/search-and-filtering.md` (comparison table, "when to use which" bullets, and the "same intent" examples) stated relation filtering was facets-only; it now reflects that `?filter=` can traverse relations via a path, reserving `?facets=` for UI facets and aggregates, with cross-links to the new filter section.

#### ArangoSearch & AQL search (Lot S)

- **"Understanding ArangoSearch" wiki primer.** A new `wiki/{fr,en}/getting-started/arangosearch.md` — the pedagogical entry point of the search stack, closing the Lot S documentation: why `LIKE` hits four ceilings (relevance, tolerance, words, index), the four building blocks (Analyzer → View → `SEARCH` → scorers) with examples, a "what becomes possible" table, how a View lives server-side (eventual consistency, initial indexing, cost, dump/restore), the library layer map (function helpers → `aqlSearch()` → `aqlScoredSearch()` → the model `AQL::VIEW` block → the clients), good practices and pitfalls (the `==`-vs-`TOKENS` stemming trap, Analyzer features, post-`SEARCH` `FILTER`, declaration drift), and the planned evolutions. Linked from the main READMEs and referenced as the primer by `db/search-views.md` and `aql/aql-functions-search.md`.

- **Facet counts now follow the View search (Lot S4b).** When a model's View search is active (`AQL::VIEW` block + `?search=` term), every `?facetCounts=` counting sub-query iterates the View with the same `SEARCH` segment as the list (and the search leaves the shared `FILTER`), closing the interim limitation of S4a: the list, `count()` and the facet buckets now always agree on the matched set — including stemming, boosts and fuzzy matches the previous `LIKE` restriction could not see. The search expression and its binds are evaluated once and shared; relevance scoring is irrelevant for counting (`COLLECT` ignores order), so the sub-queries carry no scorer. Without an active View search the emitted facet-counts query is byte-for-byte unchanged. Covered by `ViewSearchTest` facet-counts cases + live `ViewSearchIntegrationTest::testFacetCountsFollowTheViewSearch` (also proving the View `SEARCH` is accepted inside `LET` sub-queries). Documented in `db/facets.md` and `db/search-views.md`.

- **Model-level ArangoSearch View search (Lot S4a).** A `Documents` model can now declare an ArangoSearch View through the new `AQL::VIEW` block (keys in the new `models/enums/Search` class): `Search::NAME` (required), `Search::ANALYZER` (default `identity`), `Search::FIELDS` (`field => boost` map, numeric shorthand or `[ Search::BOOST => n ]`, dotted paths, falls back to `AQL::SEARCHABLE` with boost 1), `Search::PHRASE` (exact-phrase bonus, `boost × 2`) and `Search::FUZZY` (Levenshtein distance, `0` = off).
  - **Behavior:** when the block is present and `?search=` carries a term, the list query switches automatically from the `LIKE` sweep to an index-accelerated, relevance-ranked `SEARCH` against the View — `FOR doc IN @@view SEARCH ANALYZER(BOOST(doc.f IN TOKENS(@search_N, "az"), w) OR BOOST(PHRASE(doc.f, @search_N), w×2) OR LEVENSHTEIN_MATCH(doc.f, @search_N, d) …, "az")` — keeping the `?search=` contract (comma-separated terms, OR everywhere, bound values) and the whole pipeline (`?filter`/`?facets`/conditions stay as a post-`SEARCH` `FILTER`, pagination/skins/projections untouched). The base expression uses `TOKENS()` so both sides are analyzed. Without the block, the emitted AQL is byte-for-byte unchanged.
  - **Score & count:** an active search exposes the synthetic `score` sort key (`Search::SCORE`, mirroring `?near=`/`distance`): default `score DESC`, composable (`?sort=-score,name`), dropped without an active search. `count()` follows the same branch as `list()`; `?facetCounts=` keeps the `LIKE` restriction for now (lifted by S4b).
  - **Provisioning:** the View is lazily provisioned at model initialization (mirroring `initializeCollection()`/`AQL::INDEXES`): created from the declaration when missing, never altered when it exists (no automatic drift resync — lifted by M1). New façade trait `db/traits/ViewManagementTrait` (`viewCreate`/`viewDrop`/`viewExists`) used by `ArangoDB`; new `BindTrait::bindView()`; new `AQL::VIEW` key.
  - **Tests:** unit (`ViewSearchTest`, `ViewManagementTraitTest`) + live (`ViewSearchIntegrationTest` — real provisioning at construction, boosted ranking, `list()`/`count()` agreement, explicit-sort override, fuzzy typo matching, `LIKE` fallback without the block).
  - **Docs:** new `db/search-views.md` recipe page, wired into `db/README`, `db/search.md`, `db/search-and-filtering.md`.

- **New `aqlScoredSearch()` operation (`db/operations/`).** Builds a complete, relevance-ranked search query over an ArangoSearch View in the canonical form `FOR … SEARCH … [OPTIONS { … }] LET score = BM25(…)|TFIDF(…) SORT score DESC LIMIT … RETURN …` — the ArangoSearch counterpart of `aqlVectorSearch()`. Explicit named arguments: `view`, `search`, `limit`, plus optional `analyzer` (wraps in `ANALYZER(expr, "name")`), `options` (the `SEARCH … OPTIONS { … }` object), `scorer` (`SearchScorer::BM25` default / `TFIDF`), the tunings `k` / `b` (BM25) and `normalize` (TFIDF), `offset`, and `docRef` / `scoreRef` / `return` overrides. Both scorers rank better matches higher, so the sort is always `DESC`; the score is bound to a `LET` so a custom `return` can expose it. Guards throw on an unknown scorer, `k`/`b` with TFIDF, or `normalize` with BM25. Composed from existing building blocks. Covered by `AqlScoredSearchTest` + live `SearchIntegrationTest::testAqlScoredSearchBuilder`. Documented in `aql/aql-operations.md` (new *Scored search* section).

- **Completed the `aqlSearch()` operation.** The `SEARCH` clause now supports an Analyzer wrap and the `OPTIONS` object. Two new `$init` keys next to `AQL::SEARCH`: `AQL::ANALYZER` (`'analyzer'`) wraps the expression in `ANALYZER(expr, "name")`, and `AQL::SEARCH_OPTIONS` (`'searchOptions'`) emits `SEARCH … OPTIONS { … }` — accepted as an associative array (hydrated into the new `SearchOptions` value-object), a `SearchOptions` instance, any plain/`JsonSerializable` object, or a pre-encoded JSON string. `SearchOptions` (new, `db/options/`, extends `QueryOptions`) carries `collections`, `conditionOptimization`, `countApproximate`, `parallelism`; the string values get `ConditionOptimization` (`AUTO`/`NONE`) and `CountApproximate` (`EXACT`/`COST`) constant classes. `AQL::SEARCH_OPTIONS` is deliberately distinct from `AQL::OPTIONS` (the `FOR`-level options, whose hydration silently dropped the SEARCH-specific keys before this). `aqlFor()` forwards its whole `$init` to `aqlSearch()`. Without the new keys the AQL is byte-for-byte identical. Covered by `AqlSearchTest` + live `SearchIntegrationTest::testAqlForWithSearchOptions`. Documented in `aql/aql-operations.md`.

- **New `db/functions/search/` AQL function family.** The native AQL *ArangoSearch functions* for querying Views, via a new `SearchFunction` enum and one helper per function: context `analyzer()` / `boost()`; filtering `phrase()` (string or array form mirroring the official syntax — integer `skipTokens` wildcards, object tokens like `['STARTS_WITH' => ['ips']]`), `levenshteinMatch()` (typo tolerance; omitted earlier positional options filled with the server defaults `transpositions = true` / `maxTerms = 64`), `ngramMatch()` / `minhashMatch()` (PHP signature takes the mandatory analyzer **third** and the optional threshold **last** — a required PHP parameter cannot follow an optional one — the helper re-orders the emitted AQL), `minMatch()` and `exists()` (`exists($path, analyzer: 'text_en')` fills the `"analyzer"` literal), plus `inRange()`; scoring `bm25()` (`b` without `k` fills `k = 1.2`) and `tfidf()`. `minMatch()` / `inRange()` reuse the existing `MiscFunction` constants (the functions also exist outside `SEARCH`); the new enum holds the ten ArangoSearch-only names incl. `OFFSET_INFO` (constant only, helper deferred). String-literal arguments (Analyzer names, search targets, `exists` types, the `phrase` content) are quoted via `json_encode`; paths/expressions/`doc` stay raw. Covered by `SearchFunctionsTest` (100% on the new files) + live `SearchIntegrationTest` (phrase order/adjacency, skip-token wildcards, Damerau vs pure Levenshtein, prefix arrays with `minMatchCount`, `MIN_MATCH`, `EXISTS` + `IN_RANGE`, unicode-escaped literals, `BM25` ranking flipped by `BOOST`). Documented on the new `aql/aql-functions-search.md` page.

- **`startsWith()` now supports the ArangoSearch form `STARTS_WITH(path, prefixes, minMatchCount)`.** Pass an array of prefixes (emitted as a JSON array, so plain strings are quoted) and an optional minimum number of matching prefixes. The legacy string form is unchanged (the prefix stays a raw AQL expression callers quote themselves), so existing call sites are unaffected.

The object form `alt:{ key:<chain>, val:<chain> }` gives one chain per side, and `val:true` mirrors the key-side chain (e.g. `LOWER(doc.email) == LOWER(@v)`); an array value maps the chain over each element (`@v[* RETURN LOWER(CURRENT)]`). - New AQL array "question mark" helper `oihana\arango\db\functions\arrays\arrayContains()` (`array[? <quantifier> FILTER cond]`): the boolean array-contains operator, the direct/idiomatic form of `LENGTH(array[* FILTER cond]) > 0`, with optional `ANY` / `ALL` / `NONE` / `AT LEAST (n)` quantifiers (omitted = at least one). Provided as a primitive (with direct unit tests) — notably to express quantified conditions over arrays of objects (e.g. `doc.reviews[? AT LEAST (3) FILTER CURRENT.rating >= @v]`); the existing `LENGTH(...) > 0` existentials are left unchanged.
- Opt-in live ArangoDB integration suite (`#[Group('integration')]`, run via `vendor/bin/phpunit -c phpunit-integration.xml`): builds a disposable database from `configs/config.toml`, seeds it, executes the generated AQL for real, then drops it — and skips cleanly (never fails) when no instance is reachable. Covers the facet builders, the `alt` transformations, and the embedded array operations (`DocumentsArrayTrait`: set dedup, sorted-set ordering, empty `[]`, move, object deep-equality, `[]` seeding, `arrayPurgeRef`) end-to-end, complementing the server-free unit suite that only freezes the AQL string.

### Changed

- `Facet::FIELD` is now operator-enabled (`FilterComparator`): it accepts an `op` (in the definition via `Facet::OP`, or per request via an `{op, val}` object), defaulting to `match` (`=~`) for backward compatibility, with generic negation. Numeric scalars keep their type, so `doc.price >= 100` never binds a string.
- The list facet family is unified under `Facet::IN` (op-driven membership); `Facet::LIST`, `LIST_FIELD` and `LIST_FIELD_SORTED` become thin aliases, and the broken `length` branch was removed. The facet dispatcher was hardened (nullable type lookup, null-safe logger).

### Removed

- `ListItemTrait` / `MultiFieldTrait` (`models/traits/`) — orphaned, untested legacy traits composed by `Documents` but never called, and generating invalid AQL (a bind parameter used as an object attribute name; a hardcoded `doc.hasPart` filter). Replaced by `DocumentsArrayTrait` (see Added): `deleteListItem`/`insertInMultiField`/… map to `arrayRemove`/`arrayInsert`/…, `deleteReverseInMultiField` → `arrayPurgeRef`, and `updateDateParentMultiField` is dropped in favour of an application-level `afterUpdate` listener (documented in `wiki/{fr,en}/db/arrays.md`).
- `PositionTrait` (`models/traits/aql/`) — dead, half-written trait: it was composed by no model, `preparePosition()` was never called, and its only AQL form had a hardcoded `@value` placeholder (no `$binds`/value parameter) plus an unfiltered `FOR doc_coll IN <collection>` scan. Removed the trait and its characterization test rather than freeze unused, broken output; a position-ranking feature, if ever needed, should be designed fresh with proper binding.
- `Facet::THESAURUS` — it was exactly an `EDGE` facet with `AQL::FIELDS = '_key,name,alternateName'` and `op = like`, so it is now expressed as that configuration rather than a dedicated type.
- Magic `id → _key` rewriting in facets — key/property aliases are now declared explicitly via `Facet::PROPERTY` everywhere, with no hidden remapping.

### Fixed

- `aqlCollect()` could emit invalid AQL combining `AGGREGATE` and `WITH COUNT INTO`, which are mutually exclusive: when both keys were supplied it appended both clauses. `AGGREGATE` now takes precedence and `WITH COUNT INTO` is dropped (to count alongside aggregates, express the count as a `LENGTH(1)` aggregate). The only caller (`EdgesCountTrait`, which passes `WITH COUNT` alone) is unaffected. The companion `aqlCollectReturn()` mirrors the same rule.
- `aqlUpsertExpression()` never rejected an `init` that defined **both** `FILTER` and `SEARCH`: the second guard repeated the first one's condition (`!isset($filter) && !isset($search)`) instead of detecting the both-defined case, so it was dead code and a conflicting init silently fell through to `$search ?? $filter` (SEARCH wins). The guard is now `isset($filter) && isset($search)` and throws `FILTER and SEARCH cannot be defined at the same time.`, matching the documented exclusive `UPSERT [ searchExpression | FILTER filterExpression ]` syntax. The two guards are split into independent `if` clauses. Calls passing exactly one of the two (every existing usage) are unchanged. Characterized in the new `AqlUpsertExpressionTest`.
- `removeValue()` array helper emitted the plural `REMOVE_VALUES` instead of `REMOVE_VALUE`, so the single-value `FilterFunction::REMOVE` (`remove`) silently no-op'd on a scalar (its second argument must be an array) and raised an AQL error when an occurrence limit was passed; it now emits `REMOVE_VALUE`, and the optional limit works as documented. The `$limit` parameter also accepts `int|string|null` and ignores non-numeric/non-positive values instead of raising a `TypeError` at call time (e.g. a `remove` filter built from a query string). `FilterFunction::REMOVES` (`removes`) keeps using `REMOVE_VALUES`.
- `ArangoCommand` was uninstantiable whenever its `init` array omitted the `directory` key: the constructor's fallback `$this->directory = $init[…] ?? $this->directory` read the `$directory` property *before initialization* (declared `public ?string $directory ;` with no default in `ArangoDumpAction` / `ArangoRestoreAction` / `ArangoListDumpsAction`), throwing `Error: Typed property … must not be accessed before initialization`. The triplicated declaration is extracted into a single `oihana\arango\commands\traits\DirectoryTrait` (property defaulted to `null` + an `initializeDirectory()` initializer matching the `initializeEncrypt()` / `initializePassphrase()` pattern), the three actions now `use DirectoryTrait`, and the constructor calls `$this->initializeDirectory( $init )`. Behavior for configs that already supplied a directory is unchanged.
- `ArangoDB` façade hardening: `getCursor()` was typed `: Cursor` (non-nullable) but `$cursor` is null before the first `execute()` and after `streamDocuments()` resets it, so calling it in those states threw a `TypeError` — it now returns `?Cursor`. `getExtra()` / `getFoundRows()` likewise dereferenced a possibly-null cursor and now return `[]` / `0` when no query has run. The constructor caught only `ArangoException`, but the client builder throws `InvalidArgumentException` for an invalid config (e.g. empty endpoint), so build failures were never logged; the `catch` is widened to `Throwable` (logged, then rethrown unchanged).
- `CollectionManagementTrait` (`ArangoDB` façade) called an **undefined** `toLegacyException()` in the `catch` blocks of `collectionRename()`, `getIndex()` and `getIndexes()` — a leftover shim from the migration off the official ArangoDB-PHP fork. On any client error those three methods crashed with `Error: Call to undefined method` instead of surfacing the failure. The dead `try/catch` wrappers are removed so the `oihana\arango\clients\exceptions\ArangoException` propagates directly, and the redundant `ArangoException as ArangoClientException` import alias (both names resolved to the same class) is dropped. The boolean / log-and-swallow methods are unchanged.
- `PropertyController` PATCH in `raw` mode (`init[Arango::RAW] = true`) returned `null` instead of the submitted property value: the handler read the prepared payload with object-property syntax (`$payload->{$this->property}`), but `PayloadsTrait::propertyPayload()` yields an **array** (`[ property => value ]`), so the access always resolved to `null`. It now reads the value with `oihana\core\accessors\getKeyValue( (array) $payload , $this->property )`, which handles arrays and objects uniformly. Non-`raw` mode (which reloads via `model->get()`) is unchanged. Characterized then fixed in `PropertyControllerTest`.
- `aqlFields` `Field::QUOTED`: it double-quoted the key used for BOTH the output label and the attribute access, emitting invalid AQL like `"my-key":doc."my-key"` (an attribute with special characters cannot be reached with double quotes — confirmed rejected by ArangoDB with a syntax error). The attribute access now uses backticks (`"my-key":doc.`my-key``), the valid form; only the output label is double-quoted. Cases pairing `Field::QUOTED` with `Field::NAME` or `Field::ALTERS` are unchanged. Characterized then fixed in `AqlFieldsTest`.
- AQL `like()` helper: its third parameter was named `$caseSensitive` but maps to ArangoDB's `LIKE(text, search, caseInsensitive)` third argument, so it did the opposite of its name (`true` produced a case-INSENSITIVE match). Renamed to `$caseInsensitive` (default `false` = case-sensitive, matching AQL). The emitted AQL is unchanged for every call path — `?search` stays case-insensitive and the `like` filter-function alter keeps its output — only the (now honest) parameter name and the docs change. Added the previously missing direct unit test, and fixed the stale `SearchTraitTest` comment that described the predicates as "case-sensitive".
- `command:arangodb` document `upsert` now honors the `removeKeys` exclusion list: it was passed under the `excludes` key, which the upsert query never reads, so the exclusion silently had no effect.
- `ArangoCommand` registered the wrong signals for event dispatch: it passed `(int) [SIGINT, …]` (which casts to `1`) to the variadic `setSignalsToDispatchEvent()`. The intended signal list is now passed.
- Removed the unused `ignoreCollection` arangodump option (`ArangoDumpOptions::$ignoreCollection` / `ArangoDumpOption::IGNORE_COLLECTION`): no such flag exists in `arangodump`, and exclusion is resolved client-side as a `--collection` complement. The `--ignore-collection` CLI flag is unchanged.
- `ArangoConfigTrait` now validates the connection settings with exceptions instead of `assert()` (which is compiled out when `zend.assertions=-1`), so a missing database/endpoint/username never silently flows on as null. Properties are initialized; an empty password remains valid.
- Declare the previously-undeclared `ext-pcntl` requirement in `composer.json` — `ArangoCommand`'s signal handling (`getSubscribedSignals()` / `handleSignal()` / `setSignalsToDispatchEvent()`) relies on the `SIGINT` / `SIGTERM` / `SIGUSR1` / `SIGUSR2` / `SIGALRM` constants provided by the extension.
- Security: `command:arangodb dump` / `restore` no longer run `arangodump` / `arangorestore` through a shell. The command was assembled as a single string (option values quoted only with `json_encode`) and passed to `system()`, so a value containing shell metacharacters (password, database, endpoint, directory, collection name) could break the command or inject arbitrary shell. The processes are now executed via `proc_open()` with an explicit argument vector (no shell), so values are passed verbatim. New `ArangoProcessTrait` (`optionsToArguments()` + `runProcess()`), covered by `ArangoProcessTraitTest` including a metacharacter-stays-one-argument unit test and a no-shell probe.
- `command:arangodb restore --last` could never locate the archive: it set the input file to the bare filename returned by `findFiles()` (whose `FILTER` yields `$file->getFilename()`), so `assertFile()` failed unless the current working directory happened to be the dump directory. The `--last` branch now prepends the dump directory, matching the interactive branch.

- `command:arangodb restore --date <date>` could never locate the archive. The by-date suffix was built as `Char::DASH . $database . $shouldEncrypt ? FileExtension::TAR_GZ_ENCRYPTED : FileExtension::TAR`, but `.` binds tighter than the ternary, so PHP evaluated `(Char::DASH . $database . $shouldEncrypt)` as the condition — always a non-empty (truthy) string. The suffix therefore always collapsed to `.tar.gz.enc` and silently dropped the `-{database}` segment, so a non-encrypted restore-by-date never matched its `{date}-{database}.tar.gz` file. The non-encrypted branch also used `FileExtension::TAR` (`.tar`) instead of `FileExtension::TAR_GZ` (`.tar.gz`), which the dump action actually produces. The suffix logic is now isolated in the testable `ArangoRestoreAction::getArchiveFileSuffix()` helper and covered by `ArangoRestoreActionTest`.

- `Facet::ARRAY_COMPLEX` was broken at query time: it iterated an undefined variable (`FOR doc_workshops IN resultworkshops`, with no separator) and built its filter on the out-of-scope `result` document, so the facet never ran. It now iterates the in-scope embedded array (`doc.<key>`) and returns `1`. Confirmed against a live ArangoDB.
- `Facet::EDGE_COMPLEX` used a malformed loop variable (`AQL::DOC_REF . $key` → e.g. `docRefnumbers`) instead of the sibling convention `doc_<key>`; it now reads `doc_<key>` like the other builders, and gained per-field multi-value (OR) plus per-value inline negation (`!=`) on the same traversed vertex.
- Date filters with a valid `tz` always threw: the timezone-conversion path passed the bound placeholder (`@tz`) to `dateLocalToUTC()`, whose own timezone re-validation then rejected it (the literal timezone is already validated upstream by `isValidTimezone()`). The re-validation is now disabled there (`$throwable = false`), so `?filter={...,"tz":"Europe/Paris"}` emits `DATE_LOCALTOUTC(@v,@tz)` instead of crashing — and the new date `between` honors `tz` on both bounds.

### Security

- `?sort=` in open mode (no `sortable` whitelist) interpolated the raw URL key into `doc.<key>` without validation — an AQL injection vector (a model that exposes `?sort=` without configuring `$sortable`, the default). `SortTrait::prepareSort()` now validates each open-mode sort key with `assertAttributeName()`; the whitelisted path was already safe. Legitimate dotted paths (`address.city`) still sort; an unsafe key throws `ValidationException`.
- `aqlFields()` (the projection builder) interpolated the source attribute (`Field::NAME ?? key`) into `doc.<attr>` without validation. Although projection fields are server-config today (`?fields=` is not wired to any controller), the bare source is now guarded by `assertAttributeName()` as defense-in-depth, so a future URL-driven projection cannot inject AQL. The two legitimate non-identifier paths are preserved: the `Field::QUOTED` escape hatch (a special key reached via backticks, `doc.`my-key``) and numeric keys.
- HTTP facets enforce a strict trust contract: only bound values (`@bind`) are user-controlled. Operators are whitelisted (`getAlias`, falling back to the type default), facet keys are whitelisted against the model's `Arango::FACETS` (an unknown key is ignored), and the URL-provided sub-field names of complex facets are validated by `assertAttributeName` before being interpolated — a dangerous name fails the facet (logged and skipped). `alt` function names are likewise whitelisted (an unknown function is a no-op), so no `?filter=` / `?facets=` input can inject AQL.
