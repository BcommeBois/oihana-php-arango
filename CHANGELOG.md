and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Added

- **`Field::WHERE` — filter the elements of a projected array (`Filter::MAP`) by a condition, including against a runtime bind.** A `Filter::MAP` projection could map a nested array but never *restrict* its elements: `aqlFieldMap()` emitted no `FILTER`, and the projection layer carried no bind variables, so `Field::WHEN` conditions could only compare against a literal frozen in the model definition. `Field::WHERE` closes both gaps at once. It reuses the `Field::WHEN` condition grammar (leaves, `AND`/`OR`/`NOT` groups, `alt`), compiles it against the array element, and inserts a `FILTER` between the `FOR` and the `RETURN`:
  ```php
  'addresses' =>
  [
      Field::FILTER => Filter::MAP ,
      Field::WHERE  => [ 'region' , 'in' , aqlBindRef( 'allowedRegions' ) ] ,
      Field::FIELDS => [ 'street' => Filter::DEFAULT , 'city' => Filter::DEFAULT ] ,
  ]
  // addresses: ( FOR item IN doc.addresses FILTER item.region IN @allowedRegions RETURN { street: item.street, city: item.city } )
  ```
  It is **fail-closed**: a bind bound to `[]` yields `IN []` (no element); a bind missing from the final map fails the AQL query (never silently "no filter"). Out-of-scope elements are never read, so filter/sort/facet can infer nothing from them. `Field::WHERE` (which elements) stays distinct from `Field::WHEN` (a field's value), composes with `Field::REQUIRES`, and requires no change to the model query path — `get()` and `list()` already propagate `AQL::BINDS` into the final `bindVars`.
  - **Tests:** new `Field::WHERE` cases in `AqlFieldMapTest` (bind reference, `OR` group with a boolean bind, inline literal, and the no-`WHERE` non-regression). FR/EN wiki `db/conditional-fields.md` gains a `Field::WHERE` section.

### Changed

- **Test suite upgraded to PHPUnit 13 (`^12` → `^13`, 13.2.4).** `composer update -W` pulled `php-code-coverage` 14 and the matching `sebastian/*` packages; `composer.lock` is not tracked, so this is a `require-dev` bump only. Three mock call sites PHPUnit 13 flagged for removal in PHPUnit 14 were modernized, leaving the suite deprecation-free (4163 tests, 7823 assertions):
  - `UserMaxLevelResolverTest` — the shared `getResult()` mock dropped its `->expects($this->any())` invocation expectation and is now a plain stub (`->method('getResult')`), clearing the *"any() invoked count expectation is deprecated"* notice across all 17 tests that use it.
  - `ArangoDBTest` / `FederatedSearchTest` — `with*()` argument matchers previously used without `expects()` now declare an explicit `->expects($this->once())` expectation, resolving *"Using with*() without expects() is deprecated"*.

- **PHPStan wired into the toolchain (level 0).** `phpstan/phpstan ^2.2` added as a dev dependency with a `phpstan.neon.dist` pinned at level 0 over `src/`, a `composer analyse` script, and a static-analysis step in CI (before the test suite). Level 0 is green across the 772 source files; stricter levels are left for a later incremental pass.

- **PHPStan raised to level 1.** The two remaining level-1 findings were resolved without behaviour change: `PrepareTraversalTrait::prepareTraversal` gains a `@param-out array<string,mixed> $init` annotation (it takes `$init` by reference and only *adds* keys, but PHPStan had sealed the by-ref out-type to the documented input shape, wrongly collapsing later `EdgesGetTrait` reads of `AQL::TARGET` / `Arango::SORT` / `Arango::RETURN` to `null`); `HasFacetAggregateConditions` drops a dead `isset( $this->container )` guard, since the only consumer family (`Documents`, via `FacetTrait`) always sets the non-nullable container in its constructor. `phpstan.neon.dist` is bumped `level: 0 → 1`.

- **PHPStan raised to level 2.** The remaining level-2 findings were six stale PHPDoc annotations, corrected to match the native signature they document — no behaviour change: `DocumentsCommandTrait::$excludes` / `$removeKeys` drop a redundant `mixed` from their `@var array|mixed|null` (the native type is `?array`); `PayloadsTrait::propertyPayload()`'s `@param mixed $property` becomes `@param string|null $property`; `ArangoTrait::getResult()`'s `@return object|null` becomes `@return array|null` (its sibling `getObject()`, genuinely `?object`, was already correct and untouched); `HasFilterBoolean::prepareFilterBooleanValue()`'s `@return mixed` becomes `@return string`; `DocumentsTruncateTrait::truncate()`'s `@param array{ collection:null|string } $init` shape marks `collection` optional (`collection?:`), matching both the `[]` default and the `$init[...] ?? $this->collection` fallback read. `phpstan.neon.dist` is bumped `level: 1 → 2`.

- **`aqlBindRef()` / `AqlBindReference` — reference an AQL bind by name from a static definition, without inlining a value.** Where `aqlValue()` writes a value straight into the query text, `aqlBindRef('name')` returns a validated value object rendered as the `@name` token only; the value is contributed by the caller through the existing top-level bind mechanism (`AQL::BINDS`), which merges into the query's single `bindVars` map. A dedicated `instanceof` marker (not a marker string) is used because on the value side a plain string is already a legitimate literal — a value may start with `@` — so a string convention would be ambiguous. `buildWhenLeaf()` honors a bind reference on **both** the value (right) and the attribute/truthy (left) side, so a bound boolean like `[ aqlBindRef('unrestricted') ]` compiles to a bare `@unrestricted`. The bind name is validated by `assertBindVariable()` (an invalid name throws `BindException`).

### Fixed

- **`?metaOnly=` / `?facetsOnly=` bypassed the permission authorizer — every gate fell open on the sidebar path.** `DocumentsControllerListTrait::list()` builds `$modelInit` *without* an `Arango::AUTHORIZER` key, and the only place that injects one is `beforeModelCall()` — which was called **inside the `else` branch**, i.e. only on the document-fetch path. In meta-only mode the flow jumps straight to `count( $modelInit )`, then `facetCounts([ ...$modelInit ])` and `bounds([ ...$modelInit ])`, all on a `$modelInit` the hook never mutated. With no authorizer present, `isAuthorized()` falls open (returns `true`), so **every** `Field::REQUIRES` / `Facet::REQUIRES` / bound gate was silently disabled on exactly this path. The consequence was a client-triggerable inference oracle: `?facetsOnly=true&facetCounts=<hidden dimension>` returned the distinct values and counts of a field masked from reading, `?metaOnly=true&filter=<hidden field>` leaked set membership through the exact `total`, and `?bounds=<hidden field>` leaked a real min/max — the meta-only mode being precisely the faceted-search sidebar, the most exposed route. `beforeModelCall()` is now invoked **before** the `if( $metaOnly )` branch, so `$modelInit` carries the authorizer on every model call (count / facetCounts / bounds *and* list); it stays idempotent (an authorizer already in `$modelInit` is kept), so the normal path is unchanged. No new model-level configuration is introduced — meta-only now inherits every gate already declared on the model, and the fix is pure controller-side propagation (no new AQL).
  - **Tests:** three new `DocumentsControllerGatingTest` cases drive a real `list()` over an authorizer-capturing `MockDocuments` — `?metaOnly=true` propagates the authorizer to `count()` / `facetCounts()` / `bounds()`, the deprecated `?facetsOnly=true` alias does the same, and the normal (non meta-only) path still propagates it to `list()` (with `count()` untouched). Reverting the hook back into the `else` fails the two meta-only cases (`count() did not receive the authorizer`) while the normal path stays green, proving the tests pin the leak. FR/EN wiki `db/facets.md` (`?facetsOnly=`) and `db/bounds.md` (`?metaOnly=`) gain a "permissions are enforced" note.

- **A filter could not be gated on its own definition — the `?filter=` permission check was not symmetric with facets.** The library already keeps a filter behind `Field::REQUIRES`, but only the one **inherited from the projection**: `FilterTrait::prepareFilter()` read `isAttributeAuthorized( $rootSegment, $this->fields, … )` and never the `Field::REQUIRES` declared on the filter's own definition (`$this->filters[ $rootSegment ]`). `FacetTrait::prepareFacets()` gates on **both** (`!isAttributeAuthorized( … ) || !isAuthorized( $facet )`), so a facet could be locked independently of its projection while the sibling filter could not. The consequence: an attribute whose projection is guarded by *another* mechanism (or deliberately always projected, hence carrying no `Field::REQUIRES`) could not have its filter gated — a caller could filter on a dimension they should not be able to probe and infer set membership (which documents come back) even when the dimension's content is masked in the projection. `prepareFilter()` now applies a second gate (Gate B) right after the inherited one: it reads the `Field::REQUIRES` on the filter definition itself and, when refused, **neutralises** the predicate to `Boolean::FALSE` (never drops it — so it still composes safely in `and`/`or`/`not`, exactly like the existing gate and like `FacetTrait`). The gate is guarded by `is_array( $rootFilterDef )`, so a plain string (`FilterType::STRING`, …) or callable definition — which carries no `Field::REQUIRES` to read — is left untouched, and a definition without `Field::REQUIRES` behaves exactly as before. It reuses the existing `isAuthorized()` helper (already imported by `FacetTrait`); the request-level `Arango::AUTHORIZER` already flows in via `$auth`, so no host wiring changes. The `Field::REQUIRES` is declared at the root of the filter definition, including for nested shapes (`AQL::TYPE => Filter::ARRAY_EXPANSION`, `AQL::FILTERS => [ … ]`).
  - **Tests:** new `FilterDefinitionRequiresGateTest` — refused definition neutralises to `false` (no membership leak, no value bound), granted definition expands normally, fail-open with no authorizer, a `string` definition is ignored by Gate B, a definition without `Field::REQUIRES` is unaffected, and refused composition empties an `and` branch. FR/EN wiki `db/filter.md` gains an *"on the filter definition itself"* section under *Permission (`REQUIRES`)*.

- **`Field::WHERE` was silently dropped by skin normalization (bind declared-but-unused).** `Field::WHERE` is consumed by the `Filter::MAP` renderer (`aqlFieldMap()`), but `FieldsTrait::normalizeFieldDefinition()` rebuilds every field definition from a closed whitelist (`clean([...])`) — and that list omitted `Field::WHERE`. Because `prepareQueryFields()` runs on *every* query (every real model), the key was stripped before reaching the renderer: no `FILTER` was emitted. When the `WHERE` referenced a runtime bind (`aqlBindRef`), the bind was still contributed to `bindVars` but never used in the query text, so ArangoDB rejected it with *"bind parameter 'x' was not declared in the query"*. `Field::WHERE` is now preserved by the whitelist alongside `Field::WHEN` / `Field::REQUIRES` (it is read only by the MAP renderer, so preserving it unconditionally is inert for other field types). The trait docblock's preserved-keys list is corrected to enumerate the full declarative set.
  - **Tests:** `FieldsTraitTest::testMapWhereSurvivesSkinNormalization` exercises the full chain — a `Filter::MAP` field carrying `Field::WHERE` (with a bind reference) through `prepareQueryFields()` under a requested skin — and asserts the key survives normalization intact.

- **A conditional bind on a skinned-away field left an orphan `bindVars` entry (query rejected).** A `Field::WHERE` / `Field::WHEN` may compare against a runtime bind via `aqlBindRef()`, but the carrying field is *projected conditionally*: the active skin (or an explicit `?fields`) can drop it, so the final query never references `@name`. The caller cannot know in advance whether the field survives, so it supplies the bind value through `AQL::BINDS` regardless — and ArangoDB then rejects the query (*"bind parameter 'x' was not declared in the query"*) because a declared bind is unused. `prepareAndExecute()` — the single execution chokepoint every query method funnels through (`get()`, `list()`, `count()`, `exist()`, edges…) — now drops the binds the query text does not reference, so the safeguard applies everywhere uniformly:
  - The pruning is **opt-in and bounded**: it touches only binds declared *optional*. A new `?array $optionalBinds` parameter (4th position) names them; when `null` (the default) the list is **derived from the model's own field definitions** — every `AqlBindReference` found anywhere in `$fields` / `$skinFields` (`collectOptionalBindNames()`, a depth-agnostic `array_walk_recursive` scan). A bind whose name is not listed is *never* removed, so a query using no conditional bind behaves exactly as before; passing `[]` disables the pruning explicitly.
  - A listed bind is kept only when the query references it, matched on the whole `@name` token (negative lookbehind + word-boundary lookahead) so `@offers` is not seen inside `@offersScope` and a `@@col` collection reference never matches a value bind. Safe by construction: only surplus binds are removed; a referenced-but-missing bind still fails as before. No host wiring is needed — the source of truth is the `aqlBindRef()` already declared in the field.
  - **Tests:** nine `ArangoTraitTest` cases cover kept-when-referenced, dropped-when-absent, non-optional-untouched, prefix-collision, collection-bind-untouched, derivation from `$fields` and from `$skinFields`, explicit `[]` disabling, and derived-yet-referenced. FR/EN wiki `db/conditional-fields.md` gains a *"skinned field: the orphan bind is pruned"* section.

- **`Documents::__construct()` / `ArangoTrait::initializeCollection()` declared their `$type` parameter as `string` while the property it feeds (`protected int $type`) and every actual value (`CollectionType::DOCUMENT` / `EDGE`, both `int`) are numeric.** No caller ever passed anything but the class-constant default or `Edges`'s explicit `CollectionType::EDGE`, so PHP's weak-typing coercion happened to round-trip the value back to the right `int` on the property write — but the declared parameter type was dishonest, and depended on that coercion always canceling out. Both signatures are now `int $type = CollectionType::DOCUMENT`, matching the property and the enum. Behaviour is unchanged, the round-trip coercion is simply no longer needed. `ArangoTrait::initializeCollection()` / `initializeIndexes()` / `initializeDatabase()` also read their `Arango::COLLECTION` / `TYPE` / `OPTIONS` / `INDEXES` / `DATABASE` keys via the base `AQL` enum instead of its `Arango` subclass — `Arango extends AQL`, so the constants are byte-identical; this only reduces the trait's coupling to the higher-level parameter enum.

- **`DocumentsController` list flow called model methods absent from the typed contract.** `DocumentsControllerListTrait` calls `$this->model->foundRows()`, `->facetCounts()` and `->bounds()`, but `$model` is typed (via the shared `ModelTrait`) as the generic `?DocumentsModel`, which does not declare them — the calls only resolved at runtime because the concrete `Documents` model happens to provide them (through `ArangoTrait` / `DocumentsFacetCountsTrait` / `DocumentsBoundsTrait`). A new `ArangoDocumentsModel` interface extends `DocumentsModel` with those three ArangoDB-specific operations (kept off the generic contract so other models are unaffected); `Documents` now implements it, and `DocumentsController` refines its `$model` type to it via a `@property` tag. Behaviour is unchanged — the calls already worked — the contract is now honest.

- **Property access on a possibly-string signal payload (PHP 8.4 warning).** `Payload::$data` is typed `mixed`, but two edge callbacks accessed document properties on it directly: `CasbinPolicySyncEdgeTrait::onEdgeInsert()` read `$edge->_from` / `->_to`, and `OnDeleteVertex::onDeleteVertex()` read `$doc->_key`. When the payload carried a bare string (an ArangoDB `_id`) instead of a document object, PHP 8.4 emitted an *"attempt to read property on string"* warning and the value silently resolved to `null`. Both now guard with `is_object()` before the access — behaviour is unchanged (a non-object payload was already a no-op), only the warning is gone and the type is honest.

- **Two malformed PHPDoc tags that PHPStan could not parse.** `DocumentsCommand` carried a spurious `@extends Kernel` tag — `@extends` documents *generic* parent instantiation (`@extends Foo<T>`) and `Kernel` is not generic, so the tag was both meaningless and a parse error; it is removed (the real `extends Kernel` lives in code). In `aqlTraversal()`, the `@param array{…} $init` shape closed its `}` and its `$init` variable on two separate lines, which the PHPDoc parser rejects; `$init` now follows the closing brace on the same line (matching the working shape in `PrepareTraversalTrait`). Documentation only — no code change.

- **The config-level `clear` console setting was never honored (and warned on PHP 8.4).** `ArangoCommand` and `DocumentsCommand` read `$this->commandOptions?->clearable`, but `CommandOptions` has no `clearable` property — the actual flag is `$clear` (`clearable` only exists as an option *name* constant and as `clearConsole()`'s parameter name). So the fallback always resolved to `null` (the terminal was cleared only when the `--clear` CLI option was passed, never from the command's configured options), and reading the undefined property emitted a PHP 8.4 warning whenever `commandOptions` was set. Both call sites now read `$this->commandOptions?->clear`.

- **Three static-analysis findings surfaced by the new PHPStan pass.** `Collection::__construct` is annotated `@final`, making `rename()`'s `new static( $database, $name )` provably safe while keeping `EdgeCollection`'s covariant return (a `new self()` would have downcast it). `DocumentsControllerUpdateTrait` now composes `ModelCallTrait`, so its `beforeModelCall()` / `afterModelCall()` hooks resolve from the trait itself instead of the host class: `PropertyController` pulls this trait (via `PropertyControllerPatchTrait`) without extending `DocumentsController`, so its inherited `update()` used to reference undefined hooks. Behaviour is unchanged — `DocumentsController`'s own hook override still wins over the trait default.

## [1.5.0] - 2026-07-18

### Added

- **`ConceptSchemeController` / `TraversalController` — the response envelope now carries `count` and `total`.** The SKOS `/scheme` used to return only the `ConceptScheme` object, and the transitive traversal routes (children / ancestors / descendants) carried `count` but no `total` — so a client could not read "how many top concepts / vertices" from the envelope without walking the payload. Both now pass `Output::COUNT` **and** `Output::TOTAL` to `success()`. Neither route is paginated (every root / every traversed vertex ships in one response), so `count == total` and no `limit` / `offset` is emitted: on `/scheme` the top-level `total` is the size of `hasTopConcept` (the "N families" an UI shows), where `result` is the scheme object rather than a list. Shared by every consumer of `ConceptSchemeController` (e.g. a catalog `/scheme` and a thesaurus `/scheme` gain it at once); no host code, just the new envelope keys.
  - **Tests:** `ConceptSchemeControllerTest` gains an envelope case (drives `get()` with a real request/response and asserts `count == total == count(hasTopConcept)`); `TraversalControllerTest` gains the plural-envelope case (`count == total == number of traversed vertices`). FR/EN wiki `controllers/README.md` notes the `count`/`total` on both cards.

- **`InjectFilterTrait::injectFilterGroup()` — a server-side scope can now be a disjunction.** Injected filters stack as *siblings*, and siblings combine with AND, so `injectFilter()` could only ever express a conjunctive scope. A whole class of perimeters is disjunctive — "targeted at me **or** at everyone", "mine **or** public", "my tenant **or** shared" — and had no supported expression: the filter DSL accepts a logic group (`[ 'or', c1, c2 ]`), but the injector could not produce one, and a caller hand-building the group would be writing the trait's internal storage shape. The new helper closes that gap: the group is stored as **one** injected entry, so `prepareFilter()` merges it as a single operand and yields `(a || b) && <the caller's filter>` — never `a || b || <the caller's filter>`, which would degrade the restriction into an alternative (the failure mode fixed in this same cycle). Operands are filter definitions or nested groups, so an arbitrary tree can be built; an empty operand list injects nothing, since a group with no operand would widen the set to everything — never what a scope means.
  ```php
  $this->injectFilterGroup( $init , FilterLogic::OR ,
  [
      [ FilterParam::KEY => 'ownerId' , FilterParam::VAL => $userKey ] ,
      [ FilterParam::KEY => 'public'  , FilterParam::VAL => true     ] ,
  ]) ;
  // -> FILTER ( ( doc.ownerId == @a || doc.public == true ) && <url filter> )
  ```
  - **Tests:** `InjectFilterTraitTest` gains the group cases — stored as one entry (not two siblings), empty list injects nothing, stacking beside a plain injected filter, a nested group, the merge as a single operand against a URL filter, and the lone-group-unwrapped path when no URL filter is present. The `InjectFilterHost` mock gains its `callInjectFilterGroup()` proxy.

- **`Arango::SOURCE` — a relation can now read its anchor at an absolute path in the document, decoupled from its output name.** Until now a regular relation derived its anchor **from its output field name**: a join named `provider` matched on `doc.provider`, an edge named `supplier` traversed from `doc` itself — the output label and the place the data lives were welded together, so a `provider` join whose foreign key sits at `doc.selector.providerId` was impossible (only the polymorphic builder, which fed `Arango::PROPERTY` as the match root, could decouple them). The new **opt-in** `Arango::SOURCE` key declares that anchor as an **absolute path from `doc`**; absent, the generated AQL is byte-identical. The anchor's nature follows the relation: on a **join** it is the foreign key value (`FILTER doc_join._key == doc.<source>`), on an **edge** it is the traversal start vertex (`FOR … IN OUTBOUND doc.<source> …`, whose value must be a full `_id`, not a bare `_key`). `SOURCE` composes with `Arango::PROPERTY`, which stays a relative suffix (`doc.<source>.<property>`), so the historical `substitutesSegment` pattern is unchanged. On a **polymorphic edge** the two references are intentionally distinct: `SOURCE` moves the traversal start while the discriminator stays resolved on the parent document (`doc.<discriminator>` still selects the edge collection).
  - **Internals.** `buildJoinSubquery()` gains a dedicated `$keyPath` parameter that anchors the match, decoupled from `$name` — which keeps prefixing the generated variable names of nested relations. Routing `buildPolymorphicJoinVariable()` through this `$keyPath` (instead of overloading `$name` with the dotted key path) fixes a latent bug: a polymorphic branch carrying a sub-edge/join used to leak a dot into an AQL variable name (`selector.areaServed_manager_j…` → invalid query); it now emits a clean `area_manager_…`. `buildEdgeVariable()` / `buildPolymorphicEdgeVariable()` compose the start vertex via `key()`; `buildEdgeSubquery()` is unchanged (it already receives the start vertex as a parameter).
  - **Tests:** `BuildJoinVariableTest` / `BuildJoinSubqueryTest` (absolute anchor decoupled from the output name, composition with `PROPERTY`, the `Filter::JOINS` array `IN` form, `UNIQUE` / `KEY` / `docRef` independence, and the byte-identical fallback when `SOURCE` is absent); `BuildPolymorphicJoinVariableTest` (a branch with a nested join no longer leaks the dotted key path into variable names); `BuildEdgeVariableTest` (the traversal start moves to `doc.<source>`, composes with a custom start vertex, byte-identical without `SOURCE`); `BuildPolymorphicEdgeVariableTest` (the traversal start moves while the discriminator stays on the parent). FR/EN wiki `edges-joins-projection.md` gains an *Anchoring a relation elsewhere* section (join and edge examples, the generated AQL, the full-`_id` caveat on edges, and the `PROPERTY` composition), added to both tables of contents.

- **Polymorphic edges — live validation and bilingual docs (lot 3 of 3, closing the feature).** A `PolymorphicEdgeIntegrationTest` drives the real `buildVariables()` dispatch (`isPolymorphic` → the polymorphic edge builder → the shared assembler → `APPEND` of guarded traversals) against a seeded, disposable ArangoDB: a `warehouse` node traverses `warehouse_edges` to its warehouse, a `company` node `company_edges` to its subsidiary, an undeclared `city` node resolves to `null` without a fallback and to its region with one, and a permission-denied branch is dropped live (fail-closed → `null`) — proving the generated `APPEND( ( FOR v,e IN OUTBOUND … ) , … )` parses AND routes to the right edge collection on a real server. FR/EN wiki `edges-joins-projection.md` gains a *Polymorphic edge* section (the discriminator / `MAP` / `FALLBACK` declaration, the `APPEND`-of-guarded-traversals AQL, the array-`LET`-unwrapped-by-the-projection note, per-branch direction/depth, the shared fail-closed gating + fallback `NOT IN` no-oracle guarantees, and the **`Filter::EDGES_COUNT` unsupported in v1** note), added to both tables of contents.

- **Polymorphic edges — per-branch gating, the `FALLBACK` branch, and a shared branch assembler (lot 2 of 3: the security layer).** Each branch of a polymorphic edge is now permission-gated on its own (`Field::REQUIRES` / `AQL::REQUIRES` via `isAuthorized()`): a denied branch is dropped from the `APPEND` — its edge collection is never traversed, so neither a value nor an existence bit of the hidden type can leak (fail-closed). A new optional `Arango::FALLBACK` branch catches discriminator values matching none of the declared types, guarded by `NOT IN [ … ]` over all map keys (gated or not, so a document of a denied type routes to nothing, never to the fallback — no oracle). When every branch is dropped the `LET` holds `[]`, so the projection resolves to `null` (`Filter::EDGE`) / `[]` (`Filter::EDGES`) rather than a broken clause. **The branch machinery is no longer duplicated:** the validation + per-branch gating + fallback + `APPEND` combine — until now living in the join builder — is extracted into a shared `buildPolymorphicRelationVariable()` that both `buildPolymorphicJoinVariable()` and `buildPolymorphicEdgeVariable()` delegate to; a `fn($branch, $guard): string` callback is the only relation-specific part (a join wraps `buildJoinSubquery()`, an edge delegates to the already-parenthesized `buildEdgeSubquery()`). The anti-oracle gate is therefore written and tested once. The join output is byte-identical (its 17 characterization tests pass unchanged).
  - **Tests:** new `BuildPolymorphicRelationVariableTest` exhaustively covers the shared assembler with a stub branch builder (validation, guards, ordering, per-branch gating, fallback `NOT IN`, denied fallback, denied type still excluded from the fallback, every-branch-denied → `[]`); `BuildPolymorphicEdgeVariableTest` gains the edge-level gating / fallback cases against real `MockEdges` (denied branch dropped, all denied → `[]`, fallback guarded by `NOT IN`, denied fallback dropped, non-array fallback throws). 100% line coverage on the shared assembler and both thin adapters.

- **Polymorphic edges — an edge can now pick its traversed collection from a discriminator field of the start vertex (lot 1 of 3: the engine).** The edge counterpart of the polymorphic join: a regular edge traverses one fixed collection (`AQL::MODEL`); a *polymorphic* edge instead declares `Arango::DISCRIMINATOR` (the start-vertex field that decides — e.g. `kind`) and `Arango::MAP` (a `type => edge-definition` table, one branch per value — e.g. `warehouse => warehouse_edges`, `company => company_edges`), so the traversed collection is chosen at query time from a value of the parent document. AQL forbids a computed collection in `FOR … IN <dir> … <collection>`, so the edge is compiled as an **`APPEND` of guarded static traversals**: one regular traversal per branch, each guarded by an equality on the discriminator (`FILTER doc.kind == "warehouse"`) so only the matching branch yields rows. The resulting `LET` holds an **array** exactly like a regular edge, so the projection layer unwraps it with `FIRST()` for a `Filter::EDGE` or keeps the whole array for a `Filter::EDGES` — no change to the projection. Each branch is a full edge definition, so it may declare its own `AQL::DIRECTION`, `AQL::MAX_DEPTH`, etc. No new `Filter` marker: an edge definition routes to the polymorphic builder as soon as it carries `Arango::MAP` + `Arango::DISCRIMINATOR` (via the shared `isPolymorphic()`), and the two existing composable gates (field-level `Field::REQUIRES`, definition-level `AQL::REQUIRES`) still gate the whole edge. Builds on lot 0 (the `buildEdgeSubquery` extraction): `buildPolymorphicEdgeVariable()` calls it per branch with the injected guard. Per-branch permission gating and the `FALLBACK` branch land in lot 2; the integration test and FR/EN docs in lot 3. **`Filter::EDGES_COUNT` on a polymorphic definition is out of scope** (the count builder uses `LENGTH(traversal)`, not an `APPEND`) — a count entry stays on the count builder.

### Changed

- **`Arango` now extends `AQL` — one vocabulary, each key declared once.** The two enums had drifted into two views of the same vocabulary: **59** constants were declared in *both* (with, verified, **zero** divergent value), while each still held keys the other lacked. A single container definition therefore had to interleave the two classes to be writable at all — a polymorphic join spelled `AQL::KEY` / `AQL::PROPERTY` / `AQL::MODEL` / `AQL::FIELDS` next to `Arango::DISCRIMINATOR` / `Arango::MAP` / `Arango::FALLBACK`, not by preference but because neither class alone carried the whole declaration. `Arango extends AQL` writes down what was already true: the 59 duplicate declarations are deleted from `Arango` and inherited from `AQL`, and the four remaining declarative keys — `DISCRIMINATOR`, `MAP`, `FALLBACK`, `FIELD` — move to `AQL`, where the rest of the declaration vocabulary already lives. The split that falls out is **declarative vs runtime**: `AQL` carries what one writes in a model / join / edge definition, `Arango` keeps the ~101 keys a controller or a model receives at execution time (`ARGS`, `PARAMS`, `SKIN`, `AUTHORIZER`, …) — and inherits the declarative half, so `Arango::` addresses everything while `AQL::` stays the definition vocabulary.
  - **Strictly additive for consumers.** Verified by reflection against the previous state: every constant that resolved before still resolves, at the same value — **0 lost, 0 changed**. `Arango` goes from 162 to **223** resolvable keys (+61 inherited) while its own declarations drop from 162 to **101**; `AQL` declares **122** (was 118). **No duplicate declaration remains** across the two. `PaginationTrait` being used by both classes is harmless: a trait constant redeclared with an identical value is legal under inheritance.
  - **Tests:** the suite stays green unchanged (4137 tests, 7782 assertions) — which is the point: the refactor re-homes declarations, no behaviour is attached to them.

- **The edge traversal sub-query body is now a single reusable source (internal refactor, no behaviour change).** `buildEdgeVariable()`'s traversal body — the parenthesized `( FOR v, e IN <dir> startVertex edge_collection … RETURN … )` — is extracted into a new `buildEdgeSubquery()`; `buildEdgeVariable()` becomes a thin wrapper that resolves the `LET` variable name and prefixes the body. Output is byte-for-byte identical (the 30+ characterization tests pass unchanged). `buildEdgeSubquery()` gains an `$extraConditions` param emitted as a `FILTER` right after the traversal (empty → no `FILTER`, so the legacy output is unchanged). This mirrors the join split (`buildJoinVariable` / `buildJoinSubquery`) and prepares a forthcoming *polymorphic edge* — one whose traversed edge collection is chosen from a discriminator field of the start vertex — that wraps several guarded traversals in an `APPEND`. Guarded additionally by a new `BuildEdgeSubqueryTest`.

- **The shared `FOR` + `FILTER` scope of the list family is now a single source (internal refactor, no behaviour change).** `buildListQuery()`, `buildCountQuery()`, `buildFacetCountsQuery()` and `buildBoundsQuery()` each re-built the same `FOR` (View + `SEARCH`, or collection) and the same conjunctive `FILTER` (conditions + active + facets + filter + search + runtime conditions) — four byte-for-byte copies that had to be kept in lockstep, or `count()` / a facet bucket / a bound would frame a set the list never shows. Extracted into a new `FilteredScopeTrait::buildFilteredScope()` returning the `[ $for, $filter ]` pair, composed by the four builders; each call site drops to a one-liner. `CountQueryTrait` read the runtime conditions under `Arango::CONDITIONS` where the others used `AQL::CONDITIONS` — both resolve to `'conditions'`, so the unification is a no-op. The full suite is unchanged (the four builders emit identical AQL), guarded additionally by a new `FilteredScopeTraitTest`.

- **`?filter=` / `match` — three malformed spellings now fail loud instead of silently returning `0`.** Each of them built a *valid but always-false* AQL (`total: 0`), indistinguishable from a genuine empty result — a debugging trap, and the opposite of the recent fail-closed turn (sort / group / facets). To be distinguished from an **unknown key**, which stays silently ignored (loosens, never restricts). The three forms, all detected at build time:
  - **Nested object leaf after `[*]`** (`offers[*].seller.id`, where `seller` is an object) — *auto-corrected*. The dotted sub-path was unrecognised and fell through to `doc.offers[*].seller.id == @v` (an array projection compared to a scalar, never true); `prepareFilterArray()` now widens the sub-field capture to a **dotted** path and builds the correct inline `LENGTH(doc.offers[* FILTER CURRENT.seller.id == @v]) > 0`. A **direct** sub-field (`offers[*].priceCurrency`) already worked; the greedy base keeps the multi-level existential form (`employee[*].contactPoint[*].verified`) unchanged. `assertAttributeName()` still guards the dotted name against injection.
  - **Operator object in the simple `match` form** (`{"price":{"op":"gt","val":0}}`) — *rejected*. The `{op,val}` object was bound as the compared value (`CURRENT.price == @{op,val}`, never true); `buildCombinedInlineFilter()` now throws a `ValidationException` when a simple-form value is **non-scalar**, pointing to the explicit `all` form (where an object equality is legitimate).
  - **Unsupported operator inside a `match`** (`between`, and generically any operator outside the inline alias set — `contains`, `sw`, `regex`, or a typo) — *rejected*. It degraded to `==` against the raw value (`CURRENT.price == @[10,100]`, never true); `buildInlineFilterCondition()` now throws a `ValidationException` for any operator not in `FilterComparator::__ALIAS__` (`eq, ne, gt, ge, lt, le, in, nin, like, nlike, match, nmatch`) instead of the silent `==` fallback. Message points to two conditions (`ge` + `le`) for a range. `in` / `nin` stay supported (array value legitimate).
  - **Backward compatibility:** additive — legitimate forms are unchanged (direct `x[*].field`, simple `match` of scalar equalities, the `all` / `any` / `none` form, the multi-level existential form, and the silently-ignored unknown key). Only genuinely malformed spellings — that nobody writes on purpose and that returned `0` — now raise.
  - **Tests:** `BuildInlineFilterConditionTest` (`between` / unknown operator throw, `in` still builds `IN @v`), `BuildCombinedInlineFilterTest` (operator-object and list values throw, object equality via `all` and `null` value still accepted), `HasHierarchicalFilterTest` (a nested object leaf builds the correct inline `CURRENT.seller.id`). 100% line coverage on the three touched files.
  - **Docs:** FR/EN `db/filter.md` gains a *two `match` forms + three rejected traps* section (simple vs `all`, each trap with ✗ written / ✓ correct).

- **Aggregate facets now gate the aggregated field — closes the `agg:max` bound oracle, and a URL can no longer aggregate an arbitrary field (BREAKING).** An aggregate facet (`Facet::EDGE_AGGREGATE` / `Facet::JOIN_AGGREGATE`) computes a statistic (`avg` / `max` / `sum` / …) over a related field and compares it to a threshold; the field was URL-overridable (`{ "field": "…" }`) and only injection-checked, so a client could aggregate **any** related field — hidden or not — and reconstruct a bound of its value by dichotomy on the threshold. Two layers now guard it: **(1, fail-closed whitelist)** the URL may only pick a field the facet declares as aggregatable in `AQL::FIELDS` (a string is the single allowed field, a list is the allowed set with its first entry the default); a field outside the declared set — or any field when none is declared — neutralises the facet to `Boolean::FALSE`. **(2, opt-in per-user gate)** when the facet declares its target model (`AQL::MODEL`), the aggregated field inherits that model's `Field::REQUIRES` per request through the shared `isPathAuthorized()` walker; a refused field neutralises the facet. Absent `AQL::MODEL`, layer 2 is skipped — the whitelist still bounds the surface. **Backward compatibility:** an aggregate facet that let the URL choose the field must now declare the allowed set in `AQL::FIELDS` — exactly the `?sort=` / `?groupBy=` fail-closed migration; a facet already declaring `AQL::FIELDS` and querying that same field is unchanged. Relation-facet locking stays explicit (`Field::REQUIRES` on the facet definition — no inheritance from a homonymous `$edges` / `$joins` relation).
  - **Tests:** `FacetAggregateRequiresGateTest` (Levier 2 — a refused / granted target field, an ungated field, the no-`AQL::MODEL` skip, the unknown-model skip, and fail-open without an authorizer); `FacetAggregateTest` gains the Levier 1 cases (a URL field outside the whitelist, and with no whitelist declared, both neutralised) and its facets now declare `AQL::FIELDS`.

- **`?filter=` array-of-objects leaves are now gated too — closes the `contactPoint[*].email` / `match` sub-field oracle.** Completes the deep `?filter=` gate for the object-array expansion path (a distinct builder from the scalar / document / relation leaves): a single-field expansion (`contactPoint[*].email`) is gated in `buildArrayTraversal()` via `isPathAuthorized()` on the array element's sub-field, and every field referenced by a `match` (the simple `{ propertyID, value }` form or the explicit `{ all | any | none: [ { key, … } ] }` form) is gated in `prepareFilter()` through the new `collectMatchFields()` extractor. A refused sub-field neutralises the whole predicate to `Boolean::FALSE` — never dropped, which under a `none` match would loosen into an existence oracle. **Backward compatibility:** additive — an object array whose element sub-fields carry no `Field::REQUIRES` filters byte-for-byte as before.
  - **Tests:** `FilterDeepRequiresGateTest` gains object-array cases (a refused / granted `contactPoint[*].email` and its fail-open without an authorizer; a refused / granted simple `match`, and a refused explicit-`all` `match`).

- **`?filter=` deep leaves are now gated at the exact sub-field, across nested documents and relations — closes the `employee[*].salary` / `address.city` filter oracle.** The permission gate on `?filter=` resolved `Field::REQUIRES` from the **root segment only**, so a leaf hidden deeper leaked through: a sub-field of a nested document (`address.city` — `address` readable, `city` locked) or a leaf inside an *authorized* relation (`employee[*].salary` — the `employee` edge allowed, but `salary` locked in the employee model) turned an include / exclude (or a bindable comparison) into an oracle on the hidden value. The hierarchical filter builder now threads the **current model's projection** — switched to the target model's `$fields` when an edge / join is crossed — and the **relative field path** (reset at each crossing) down to every leaf, gated through the shared `isPathAuthorized()` walker (`Field::FIELDS` + fail-closed `SKIN_FIELDS` union, every level of the sub-path). A refused leaf is neutralised to `Boolean::FALSE` and — the security crux — sinks the **whole traversal** *before* the `all` / `none` quantifier negation, so a refused leaf under `quant:all` can never become `NOT(false) = true`, an existence oracle (« Option B »). When the target model of a relation is unresolvable, the leaf fails open (nothing to inherit). **Backward compatibility:** additive — a model with no nested / target-model `Field::REQUIRES` filters byte-for-byte as before. **Array-of-objects leaves** (`contactPoint[*].email`, `match`) flow through a distinct builder and are gated too — see the *array-of-objects* entry above.
  - **Tests:** unit (`FilterDeepRequiresGateTest` — a refused / granted nested-document leaf (`address.city`), a refused edge leaf neutralising the whole traversal and a granted one traversing, the `all`-quantifier leaf staying `false` (no existence oracle), a fail-open on an unresolved target model, a refused / granted join leaf, and deep leaves failing open without an authorizer).

- **`?groupBy=` deep paths are now gated at the exact sub-field, not just the root — closes the `address.city` group-by oracle.** The permission gate on a grouping dimension / aggregate resolved `Field::REQUIRES` from the **root segment only** (`explode('.', $field)[0]`), so a sub-field locked in the `Field::FIELDS` of a structural field (`address.city` — `address` unlocked, `city` carrying a `REQUIRES`) leaked its distinct values (dimension) or a bound (aggregate) while the root stayed open. A new shared helper **`isPathAuthorized()`** walks the whole dotted path, reading `Field::REQUIRES` at **every** level and descending through the declared sub-fields — `Field::FIELDS` **and**, fail-closed, every `AQL::SKIN_FIELDS` bucket (a sub-field locked in *any* projection stays gated, since a permission is skin-independent). `GroupTrait` now gates the dimension and the aggregated field through it instead of the root-only `isAttributeAuthorized()`. A single-segment path behaves exactly as before — `isPathAuthorized()` is a strict superset of `isAttributeAuthorized()`, and each segment is stripped of the `[*]` marker so the same helper serves the hierarchical filter leaves next. **Backward compatibility:** additive — a model with no nested `Field::REQUIRES` groups byte-for-byte as before; the gate only adds refusals on sub-fields that were previously leaking.

- **`?groupBy=` is now fail-closed and permission-gated — no more open-mode grouping, no group-by oracle (BREAKING).** Grouping was the last read parameter still running in *open mode*: `$groupable === null` (the default) let a client `COLLECT` on **any** field, returning its distinct values and per-value counts in clear — a direct oracle — and aggregates (`Group::AGG`) leaked a **bound** of a field (`MAX/MIN/AVG/SUM`) with no whitelist at all. ### Fixed

- **An injected scope filter can no longer be absorbed by a caller's own `or` group — a URL filter group used to turn a server-side restriction into an alternative.** `InjectFilterTrait::prepareFilter()` merges the URL filter with the filters a controller injects server-side (`injectFilter()`) — the mechanism every scoped surface relies on to force a perimeter the caller must never widen (`seller._key IN [ … ]` on a portfolio, an owner key, a tenant). The merge asked one binary question — `isset( $urlFilter[ FilterParam::KEY ] )`, "a single filter or a list?" — of a **ternary** reality: the filter DSL also accepts a *logic group* (`[ 'or' , c1 , c2 ]`), which carries no top-level `KEY` and was therefore mistaken for a plain list and **spliced**. Its operator stayed in head position and swallowed the injected filters as extra operands: `?filter=["or",A,B]` compiled to `(A || B || scope)` instead of `((A || B) && scope)` — **the scope became a third alternative rather than a restriction**, so any caller sending an `or` group over a whitelisted key read the whole collection. (`[ 'not' , c ]` was mangled the same way: the splice broke the arity check that routes the negation, and the `not` degraded into a join operator.) The merge no longer decides: the URL filter always enters as **one operand** of an explicit `and` group — `[ FilterLogic::AND , $urlFilter , ...$injected ]` — which is shape-agnostic, because the filter parser recurses and resolves the URL filter whole, in its own parentheses, where it can never meet the injected ones. The `AQL::FILTERS` whitelist was no defense here: it gates *which keys* are filterable, never *which logic* assembles them.
  - **Compatibility.** The two branches carrying the bulk of the traffic are untouched — nothing injected (`return $urlFilter`) and no URL filter (`return $injected[0]` / `$injected`), the latter covering every `prepareFilter( null , … )` membership probe. Only a request that carries **both** a `?filter=` and an injected filter changes, and only in shape: a single filter and a plain list produce the same AQL as before (the implicit `and` of a bare list becomes the explicit `and` of a group — both already covered by `HasFilterConditionsTest::testImplicitAndConditions` / `testExplicitAndConditions`), while a group is repaired. **Consumers asserting the returned array shape must update:** the merge now returns `[ 'and' , … ]` where it returned a flat list.
  - **Tests:** `InjectFilterTraitTest` gains the missing third shape — `testPrepareFilterKeepsInjectedOutOfAUrlFilterGroup` (an `or` group stays a nested operand, never absorbs the scope), `testPrepareFilterKeepsInjectedOutOfAUrlFilterNegation` (a `not` group keeps its single operand), `testPrepareFilterKeepsInjectedOutOfAUrlFilterGroupWithMultipleInjected` (every injected filter stays a sibling of the `and`) — and the two merge tests are rewritten / tightened onto the explicit `and` shape. The suite covered the two shapes that worked and never the one that did not, which is why this shipped.

- **`match` sub-field names are now injection-guarded unconditionally.** A `?filter=` `match` on an object array (`contactPoint[* FILTER …]`) interpolates each sub-field name into `CURRENT.<field>`; the field was only validated when the base key declared an `AQL::FILTERS` whitelist, so a `match` on a bare `FilterType::ARRAY` key left a client sub-field name unguarded. `buildInlineFilterCondition()` — the single choke point every sub-field flows through — now calls `assertAttributeName()` on the field as defense in depth: a nested path (`address.city`) still passes, an injection-looking name (`x) || 1==1`) throws `ValidationException`. A companion security invariant is documented on `Edges::getVertices()` (its verbatim `AQL::RETURN` / `AQL::SORT` / `AQL::EDGE_COLLECTION` / `AQL::GRAPH` knobs are server-only — never wire a client parameter into them; a client sort must travel through the whitelisted, permission-gated `?sort=`).
  - **Tests:** unit (`BuildInlineFilterConditionTest` — a dotted sub-field path is accepted, an unsafe field name throws).


- **`returnFields()` anchors relation `LET` sub-queries on the projected document when projecting through a vertex traversal.** When a model's declared fields include a relation (`Filter::EDGES` / `EDGE` / `EDGES_COUNT` / `JOIN` / `JOINS`) and that model is projected as the **target of a vertex traversal** (`Edges::getVertices()` and its inbound/outbound variants, which call `returnFields()` with `DOC_REF` = the traversal loop variable, e.g. `vertex`), the generated relation `LET` sub-queries started from an **unbound** `doc_`-prefixed alias (`doc_vertex`) while the `RETURN` projection used the bare loop variable (`vertex`) — the two diverged, and ArangoDB raised `collection or view not found: doc_vertex`. The cause: the prepared-fields branch of `returnFields()` prefixed the document reference with `AQL::DOC_PREFIX` before handing it to `buildVariables()`, but nothing binds a `doc_<ref>` loop in that path (unlike the facet helpers, which open their own correlated `FOR doc_<key>`), and the sibling `RETURN` projection (`aqlFields()`) used the un-prefixed reference. The branch now passes `$docRef` verbatim to both, so the relation `LET`s and the projection anchor on the **same** variable — exactly as the `*` (whole-document) branch already did. In a main query (`DOC_REF` = `doc`) the reference is unchanged, so `list()` / `get()` output is identical.
  - **Tests:** unit (`FieldsTraitTier2Test` — an `EDGES_COUNT` and an `EDGES` field projected through a traversal (`DOC_REF: 'vertex'`) each emit a `LET` whose inner `FOR … IN OUTBOUND vertex …` starts from the outer loop variable, with no `doc_vertex` alias; plus a non-regression check that the same `EDGES_COUNT` in a main query still anchors on `doc`).

- **`Filter::EDGES_COUNT` now composes inside a vertex traversal.** A model declaring a `Filter::EDGES_COUNT` field worked in `list()` / `get()` but raised an `AQL: variable 'vertex' is assigned multiple times` error the moment that model was the **target** of a traversal (`Edges::getVertices()` / `getOutboundVertices()` / `getInboundVertices()`). The reason: the count sub-query, built by `buildEdgeCountVariable()`, hard-coded its inner loop as `FOR vertex IN <dir> <start> <edge> RETURN vertex`. In `list()` / `get()` the outer model loop is `doc`, so no clash — but a vertex traversal's outer loop is already named `vertex` (`AQL::VERTEX`), and when the target model re-emitted its count `LET` over that traversed vertex, the inner `FOR vertex` collided with the outer one. The inner count loop now uses a **unique** variable derived from the `LET` name (`<name>_v`, itself a `randomKey` in the live flow) instead of the shared `vertex`; the `LET` variable name — i.e. the projected field's public API — is unchanged, and the count semantics are identical. The two sibling builders were already safe (`buildEdgeVariable()` / `buildJoinVariable()` derive their loop names from `randomKey()`); only the count helper reused the bare `vertex`.
  - **Tests:** unit (`BuildEdgeCountVariableTest` — a count projected with the outer traversal loop name (`startVertex: 'vertex'`) yields two distinct `FOR` loops, the inner one **not** reusing `vertex`; the default and unique-name snapshots updated to the `<name>_v` inner loop; `BuildVariablesTest` — the `EDGES_COUNT` field snapshot updated likewise).

- **Analyzer diff — a declared `pipeline` no longer drifts on its sub-analyzer defaults.** `analyzerDiff()` compares a declaration to the server with **subset** semantics: it iterates only the declared keys, so the server defaults a declaration omits (a `norm` declared `{ locale }` reads back `{ locale, case, accent }`) are ignored — but only because the top-level loop walks declared keys one by one. A `pipeline` analyzer nests its chain inside a **list** (`properties.pipeline`), and the list was compared by deep equality, so each sub-analyzer the server fills with all its defaults (`norm` → `case` / `accent`; `ngram` → `startMarker` / `endMarker` / `streamType`) diverged from the declaration and the pipeline showed up as a **permanent false drift** — the same bug the `identity` analyzer hit on View links. The pipeline list now gets its own declaration-oriented comparison (`comparePipeline()`): members are matched by **position** (the chain order is significant — a reordered chain *is* a real drift) and by type, and only the **declared** sub-properties are checked, server defaults ignored. `normalizeAnalyzerValue()` is untouched, so other list properties (e.g. `stopwords`) keep their strict equality. A `RawAnalyzer( 'pipeline', … )` rebuilt from a server dump (all defaults present) stays a valid round-trip.
  - **Tests:** unit (`AnalyzerManagementTraitTest` — a declared `norm` → `ngram` pipeline IN_SYNC against the server-defaulted chain; real drift still caught on a changed sub-analyzer type, a reordered chain, a different chain length, and a changed sub-property) + live (`PipelineAnalyzerSyncIntegrationTest` — on a real server the created pipeline re-diffs IN_SYNC despite the server filling every sub-analyzer default, and a `RawAnalyzer` dumped from that server round-trips too).

- **Federated search — the rebuild now restricts to the matched keys.** `FederatedSearch::rebuild()` asked each model for its page through the public `?filter=` channel (`Arango::FILTER` with `_key IN […]`), which only applies to a model's **whitelisted** filterable fields — and `_key` is almost never one, so the filter was silently dropped and `list()` (called without a `LIMIT`) rebuilt the **entire collection**, leaving the match-driven re-merge to trim the page. Correct, but it loaded a whole collection per model on every federated search (invisible on the tiny test collections, severe at scale). The key restriction now travels through the **trusted internal condition** channel (`AQL::CONDITIONS` + `AQL::BINDS` → `FOR doc IN coll FILTER doc._key IN @keys …`), which bypasses the field whitelist and always applies, so each model fetches **only** the page's keys. Behaviour is otherwise identical (the re-merge still orders by score); this also benefits the type-aware composite routing (one trusted condition per resolved model).
  - **Tests:** unit (`FederatedSearchTest` — the rebuild restriction asserted through the bound key set on both the direct and composite paths) + live (`SearchAliasViewIntegrationTest` — on a real server each model's `list()` returns **only** the keys it was asked to rebuild, not the whole collection).

- **`command:arangodb` now reads the `analyzers` (and `searchAliasViews`) init keys.** The `ArangoCommand` constructor wired only some of its declarative registries (`models`, `collectionIndexes`, …) but silently dropped the `analyzers` and `searchAliasViews` init keys: a host-declared custom-analyzer registry never reached `getAnalyzerDefinitions()` (it stayed `[]`), so `arango:analyzers --sync` reported « No analyzers configured » and created nothing, the `doctor` analyzers pass reconciled nothing, and a View referencing the analyzer stayed « invalid: analyzer not found ». The same omission left a host-declared `search-alias` view registry empty for the `views` action. Both keys are now read in the constructor like the other registries (`$this->analyzers = $init[ArangoCommandParam::ANALYZERS] ?? $this->analyzers`). **Strictly backward-compatible**: without the key the registry stays its empty default. The trait, property, getter and consuming actions all already existed — only the one constructor line per key was missing.
  - **Tests:** unit (`ArangoCommandTest` — each registry is wired from its init key, a lone `AnalyzerDefinition` is normalized, a missing key defaults to `[]`, plus a regression guard asserting the constructor wires *every* declared registry — `analyzers` / `collectionIndexes` / `models` / `searchAliasViews` — so the next added registry cannot slip through unwired) + live integration (`AnalyzerCommandIntegrationTest` — an analyzer declared through the **real command's** `analyzers` init key is reported missing, created by `--sync` on a real server, then in sync; previously this run printed « No analyzers configured »).

## [1.3.0] - 2026-06-20

### Added

- **Federated search — the *rebuild* stage (Lot C3).** The engine's second stage completes the two-phase search: `FederatedSearch::rebuild()` takes a *find* page (`{ collection, key, score }`), groups the keys by collection, and re-hydrates each group **in one `list()` call per collection** (a `_key IN […]` filter — never one query per result) through the model that owns it (resolved via the collection → model registry), reusing that model's own projection pipeline (fields, joins, skin, permissions). - **Conditional field projection — `Field::WHEN` / `Field::ELSE`.** A scalar field can now guard its value behind a condition resolved **at query time** from the document's own attributes, rendered as an AQL ternary `<condition> ? <then> : <else>` (the SQL `CASE WHEN … THEN … ELSE …` shape). The key is always present — only the value switches; an absent value is `null`, never a removed key (no `MERGE`, intentionally out of scope). The condition mirrors the recursive `?filter=` grammar — a leaf (`'active'` truthiness, `[ 'visibility', 'public' ]` equality, `[ 'stock', 'gt', 0 ]` comparator, or the associative `FilterParam` `key`/`op`/`val`/`alt` form) combined with AND/OR/NOT groups and arbitrary nesting — but compiles **inline** (no bind variables, since the projection layer carries none): values are developer-declared literals escaped by `aqlValue()` (a value that looks like `doc.<attr>` is kept raw, enabling attribute-vs-attribute comparison), and attribute names are validated by `assertAttributeName()`. Only the infix comparators (`eq`/`ne`/`ge`/`gt`/`le`/`lt`/`in`/`nin`/`like`/`nlike`/`match`/`nmatch`) are supported; a function-form operator (`contains`/`sw`/`ew`/`regex`/…) throws. A leaf may carry an `alt` chain wrapping its operands (same `"lower"` / `{ key, val }` / `{ key, val:true }` mirror vocabulary as the flat filters). `Field::ELSE` is `null` by default, a literal, or an attribute reference (`[ Field::PROPERTY => 'basePrice' ]`); it composes with `Field::ALTERS` (which decorates the *then* branch) and `Field::NAME`. Applies to the default scalar projection only — `Field::WHEN` on a typed/structural filter (`EDGES`/`JOINS`/`DOCUMENT`/`MAP`/`URL`/…) throws an `UnsupportedOperationException`. New `Field::WHEN`/`Field::ELSE` options, the `aqlFieldConditional()` field builder and the `buildWhenCondition()` / `buildWhenLeaf()` / `resolveWhenElse()` helpers, dispatched from `aqlFields()` next to `Field::ALTERS`; `FieldsTrait::normalizeFieldDefinition` preserves `WHEN`/`ELSE` through the model → query path.

### Fixed

A global `max_total` size cap (`"5G"` / `"500M"` / a byte count) applies last, dropping the oldest archives across buckets until the total fits.
  - **Safety:** at least one archive per bucket is always kept, the freshly created archive is never pruned, `--dry-run` lists without deleting. Two triggers: `dump --prune` (prune-only, no new dump) and `[arango.dump.retention] auto = true` (prune after each successful dump).
  - **Code:** new `ArangoRotationTrait` — pure `planRotation()` engine (count / age / conservative combo / floor / `max_total` / current-exclusion, no I/O, no clock), `pruneDumps()`, `resolveRetentionPolicy()`, `retentionEnabled()`; new `RetentionOption` keys (`keep` / `max_age` / `max_total` / `auto` / `buckets`) and `ArangoCommandParam::RETENTION` (stripped from the dump options). The archive-name regexp duplicated across listing / restore / rotation was consolidated into the single `ArchivePattern::REGEXP` (trait constants alias it — a trait constant can no longer be reached directly, per PHP 8.4).
  - **Tests:** unit (`ArangoRotationTraitTest` — every `planRotation` rule, the policy resolver incl. invalid `max_age`, the size parser, `pruneDumps` delete / dry-run / nothing-to-prune / current-exclusion; `ArangoDumpActionTest` — `--prune` prune-only, no-policy warning, dry-run, post-dump `auto`, off-by-default, key-never-leaks) + live (`testRotationPrunesRealArchivesOnDisk`).
  - **Docs:** `commands/arangodb.md` (new "Archive rotation" section with `P30D`/`P6M`/`P1Y` examples) + `config.example.toml`.

- **Portable PHP masking engine (Lot D5b).** Makes the convenient masking of Lot D5 work on **every ArangoDB edition**, Community included — native `arangodump` masking is Enterprise-only, so the engine anonymizes the dump as a post-processing pass over the `*.data.json` files (the Community binary dumps the data fine, only its masking is gated).
  - **Engine:** a dependency-free, framework-agnostic `oihana\arango\maskings` namespace (file-per-function, ready to move to `oihana/php-core` or a dedicated masking library when a second consumer appears): one pure function per masker (`maskEmail`, `maskPhone`, `maskCreditCard`, `maskZip`, `maskRandom`, `maskRandomString`, `maskXifyFront`, `maskDatetime`, `maskInteger`, `maskDecimal`), a `maskValue()` dispatcher (array-aware: a masker on a JSON array masks each element, nested objects untouched), the `maskDocument()` path engine (the `arangodump` path semantics: `name` top-level leaf, `a.b` exact, `.name` any-depth, `*` all leaves, backtick-quoted literal keys, arrays per element, first-matching-rule-wins, and the system attributes `_key`/`_id`/`_rev`/`_from`/`_to` never masked), and the `Masker` / `MaskingMode` enums. The goal is semantic equivalence (PII removed as valid, typed values), not byte-compatibility with the Enterprise binary.

The native `--maskings` / `[arango.dump] maskings` file remains the Enterprise escape hatch and, when present, takes over (PHP engine off); only `masked` mode is supported (collection exclusion stays the job of `--collection` / the profile — a non-`masked` mode raises).
  - **Tests:** unit (`MaskingsTest` — every masker's type rules and edge cases, the dispatcher, the full path engine incl. the reference `xifyFront`/`.name` examples; `ArangoMaskingTraitTest` — `maskDumpDirectory` matching, wildcard default, filename fallback, skips, non-`masked` rejection; `ArangoDumpActionTest` — PHP-engine path, profile-over-default, CLI-native-wins, dry-run) + live (`testPhpEngineAnonymizesARealDump` — removes email and a nested phone from the dumped data, running on Community, no skip).
  - **Docs:** `commands/arangodb.md` (the "Masking" section now leads with the PHP engine) + `config.example.toml`.

- **Dump-time masking (Lot D5).** Anonymize a dump **as it is written**, so the archive itself is clean and can travel/restore without exposing PII (the natural complement of the staging→local profiles). The latent `ArangoDumpOptions::$maskings` plumbing was unreachable (no CLI, no config, no doc); D5 makes it usable two ways.
  - **Native file (escape hatch):** a new `--maskings <file.json>` CLI flag and the `[arango.dump] maskings = "…"` config default pass an `arangodump` maskings JSON file verbatim.
  - **Convenient TOML form (common case):** a `masking` table — in a profile (`[arango.profiles.<name>.masking]`) or in the dump defaults (`[arango.dump.masking]`) — compiled to a native maskings file in a temp (cleaned up) and fed to the binary. Flat dotted-key syntax: a key **without a dot** is a collection (or `*`) set to a document-level mode (`exclude` / `structure` / `masked` / `full`); a key **with a dot** is `<collection>.<path>` set to a masker — a name string (`email`, `phone`, `creditCard`, `zip`, `random`, `randomString`, `xifyFront`, `datetime`, `integer`, `decimal`) or an inline table with `type` + parameters (`{ type = "xifyFront", unmaskedLength = 4 }`); a dotted key implies the collection's `masked` mode. An unknown mode/masker raises a clear error.

  - **Code:** new `ArangoMaskingTrait` (`compileMaskings()` + `materializeMaskings()`), `ArangoCommandOption::MASKINGS`, `ArangoCommandParam::MASKING` (stripped from the dump options); `--dry-run` previews the masking source.

- **Restore guard-rails (Lot D4).** `restore` is the only destructive action of the dump tooling and had no safety net; four guard-rails now frame it, reusing the proven `migrate` idiom.
  - **(1) Protected collections:** a new `[arango.restore] protected = […]` config key (a deployment policy, never an `arangorestore` option — stripped before the binary) lists collections the restore refuses to overwrite unless `--force`; the check runs on the *effective* set the restore will write (the explicit `--collection` / profile selection, or — for a whole-archive / exclude-only restore — every collection in the untarred archive), so a stray full restore can no longer clobber the auth collections.

  - **(4) Selection validation:** a requested collection absent from the archive triggers a non-blocking warning.

- **Surgical complete backup (Lot D3).** New `--complete` flag on `dump` (also a `[arango.dump] complete = true` default) closing the Lot S completeness gap: a default dump backs up the user collections, their indexes and the View definitions, but not the custom analyzers (`_analyzers`) or named graphs (`_graphs`) — system collections that, missing on a restore into a fresh server, can break a View referencing a custom analyzer.
  - **Behavior:** includes every user collection plus `_analyzers` and `_graphs`, and *only* those two system collections (never `_users`, `_jobs`, `_queues`, …): enumerates via the HTTP API and passes `<all user> + _analyzers + _graphs` with `includeSystemCollections = true`. - **Dump / restore profiles + `--dry-run` (Lot D2).** Reusable, named selections for `dump` and `restore` — the heart of the dump tooling. A profile names *what* to extract (and optionally *from where*) so a recurring extraction is not retyped as a long `--collection …` line; the motivating workflow pulls a subset of staging into the local database to test against real data without overwriting the local auth collections.
  - **Profile shape:** `collections` / `edges` (positive selection, merged), `exclude` (set subtraction), and optionally `endpoint` / `database` / `user` / `password` (a source connection). Selection resolves to `(collections + edges) − exclude`; an exclude-only profile means "everything minus exclude" (the universe is the server collections for `dump`, the archive's collections for `restore`, read from the untarred `*.structure.json`).
  - **`--profile <value>`:** accepts either a named `[arango.profiles.<name>]` section or a path to a standalone `.toml` file (absolute/relative, via `oihana\files\toml\resolveTomlConfig` — portable). New `ArangoProfileTrait`; profiles captured via `ArangoCommandParam::PROFILES` (keys `collections` / `edges` / `exclude` get `PROFILE_*` constants).
  - **Safety by design:** a profile's connection is the source — `dump` uses it, `restore` ignores it and always writes to the local target (a profile can never push its data back to its origin server); `--profile` is mutually exclusive with `--collection` / `--ignore-collection`; precedence stays `binary default → [arango.dump]/[arango.restore] → profile → CLI`.
  - **`--dry-run`** (on `dump` and `restore`): reports the resolved plan (connection, archive, exact collection list) and runs nothing.

- **Configurable dump/restore option defaults (Lot D1).** The foundation of the dump/restore tooling. The `dump` / `restore` actions already wrapped the full `arangodump` / `arangorestore` option surface, but only the connection settings reached them — every other option had to be retyped each run, and most were not even exposed.
  - **Config sections:** two optional `config.toml` sections, `[arango.dump]` and `[arango.restore]`, hold persistent defaults for any option (keys are the camelCase property names of `ArangoDumpOptions` / `ArangoRestoreOptions`; unknown keys ignored). #### Maintenance commands — views, doctor, migrate (Lots M1–M3)

- **Versioned data migrations (Lot M3).** The imperative counterpart of the declarative `doctor`: where `doctor` reconciles the *structure* declared in the DI definitions, `migrate` runs hand-written PHP migrations that transform *data already in the database* (a backfill, a field reshape, a one-off cleanup) — operations that cannot be expressed as configuration.
  - **`oihana\arango\migrations` namespace:** the abstract `Migration` (contract — mandatory `up()`, optional no-op `down()`, `description()`; receives the `ArangoDB` façade, so free AQL via `query()`, the collection CRUD and the doctor primitives, plus `renameField()` / `dropField()` / `setDefault()` built on pure `migrations\helpers\*Query()` builders); the `MigrationRunner` engine (`discover()` scans `Version<timestamp>_<Label>.php` and orders by version, `status()`, `pending()`, `apply()` records each run `active` → `completed` or `failed`-and-stops-net, `down( n )` LIFO, `forget()`); the `MigrationGenerator` (`--create` shell, no database); the `MigrationStore` persistence; and the `MigrationAction` tracking document — a schema.org value-object (`extends org\schema\actions\UpdateAction`) carrying `_key` = version, `actionStatus`, `startTime`/`endTime`, `agent` (`user@host`), `error`, and the custom `gitCommit` hash (the link between the database and the source tree).

- **Structure doctor (Lot M2).** The global declarations ↔ server health check, extending the Lot M1 View diff & sync to the whole declared structure. The motivating bug: lazy provisioning only creates the `AQL::INDEXES` when the collection is created, so an index added to an existing model is silently never created anywhere (queries fall back to full scans with no error) — and the model did not even retain its index declaration after construction.
  - **Façade:** `ArangoTrait` keeps the declaration in the new `$indexes` property; `collectionDiff( $name , $type )` on `CollectionManagementTrait` (existence + document/edge type — a drifted type is never repaired, that is a migration); the index surface extracted to the new `IndexManagementTrait` — the legacy `createIndex` / `dropIndex` / `getIndex` / `getIndexes` (`createIndex` / `dropIndex` now typed `string|Collection`, like the promoted `resolveCollectionName()`), plus `indexesDiff( $collection , $indexes )` (declared indexes matched by `name` or `type`+`fields` signature, subset-aware field comparison with **order-sensitive** `fields`, `inBackground` excluded, automatic `primary`/`edge` out of scope, undeclared server indexes reported) and `indexesSync( $collection , $indexes , $force )` (missing created; drifted rebuilt by drop + recreate only when forced — an index is immutable).
  - **Model:** new `DoctorTrait` (composed by `Documents`, inherited by `Edges`): `diagnose()` returns one `DiffReport` per declared object (collection → indexes → View, via the M1 `viewDiff()`) and `repair( force )` reconciles.
  - **`doctor` action:** report by default with the orphan footnote (collections/Views no configured model declares — never failing, never touched), `--apply` to repair, `--force` for index rebuilds, `--prune` for interactive orphan removal; the exit code is a health check (CI-friendly); same `ArangoCommandParam::MODELS` wiring and container lazy kill-switch as `views`. The M1 report classes were generalized: `ViewDiffReport` → `DiffReport` (with `kind` + `DiffKind` constants) and `ViewDiffStatus` → `DiffStatus`; the shared `models` property moved to `ArangoModelsTrait`; new `ArangoClientTrait::resolveDatabase( $input )`. New `arango:doctor` composer script.

- **View diff & sync (Lot M1).** Closing the declaration-drift reservation of the model-level View search: provisioning is create-if-missing, so editing an `AQL::VIEW` block used to leave the server View silently out of date. The gap can now be inspected and repaired explicitly, at three levels.
  - **Façade (`ViewManagementTrait`):** `viewDiff( $name , $links )` compares the desired per-collection link map with the server `properties()` — declaration-oriented and subset-aware (undeclared server defaults ignored, `analyzers` compared order-insensitively, fields indexed but no longer declared reported as drift) — and `viewSync( $name , $links )` creates a missing View or repairs a drifted one via `updateProperties()` (verified live: PATCH replaces each declared collection link wholesale while leaving View options and other collections' links untouched, the View staying queryable during the background re-index). Both return the new typed `DiffReport` (`db/results/`: `name`, `status`, human-readable `changes`, `applied`) with `DiffStatus` constants (`IN_SYNC` / `MISSING` / `DRIFTED` / `INVALID` / `UNREACHABLE`).
  - **Model (`SearchTrait`):** no-argument `viewDiff()` / `viewSync()` use the model's own declaration (new `getViewName()` / `getViewLinks()`), adding coherence checks (malformed declaration, Analyzer/collection unknown to the server, same-name entity of another type → all `INVALID`, never created nor synchronized). New façade trait `AnalyzerManagementTrait` (`analyzerExists()`).
  - **`views` action:** default lists the Views, `--diff` reports every configured model (plus the orphan-views footnote), `--sync[=a,b]` applies, `--drop[=a,b]` removes targeted or interactive multi-select; models supplied as container ids via the new `ArangoCommandParam::MODELS`; the exit code fails when a model is unreachable.   - **Docs:** `commands/arangodb.md` (new `views` section) + the drift notes of `db/search-views.md` / `getting-started/arangosearch.md` now point to the remedy.

- **Container-aware lazy provisioning switch.** The model lazy mode now resolves through the new `oihana\traits\LazyTrait` (from `oihana/php-system`): `ArangoTrait` dropped its own `$lazy` property and composes the trait, `initializeCollection()` / `initializeView()` read `$this->initializeLazy( $init )->isLazy()`, resolution order container entry `lazy` > init key `Arango::LAZY` > property default (`true`). #### ArangoSearch & AQL search (Lot S)

- **"Understanding ArangoSearch" wiki primer.** A new `wiki/{fr,en}/getting-started/arangosearch.md` — the pedagogical entry point of the search stack, closing the Lot S documentation: why `LIKE` hits four ceilings (relevance, tolerance, words, index), the four building blocks (Analyzer → View → `SEARCH` → scorers) with examples, a "what becomes possible" table, how a View lives server-side (eventual consistency, initial indexing, cost, dump/restore), the library layer map (function helpers → `aqlSearch()` → `aqlScoredSearch()` → the model `AQL::VIEW` block → the clients), good practices and pitfalls (the `==`-vs-`TOKENS` stemming trap, Analyzer features, post-`SEARCH` `FILTER`, declaration drift), and the planned evolutions. Linked from the main READMEs and referenced as the primer by `db/search-views.md` and `aql/aql-functions-search.md`.

- **Facet counts now follow the View search (Lot S4b).** When a model's View search is active (`AQL::VIEW` block + `?search=` term), every `?facetCounts=` counting sub-query iterates the View with the same `SEARCH` segment as the list (and the search leaves the shared `FILTER`), closing the interim limitation of S4a: the list, `count()` and the facet buckets now always agree on the matched set — including stemming, boosts and fuzzy matches the previous `LIKE` restriction could not see. The search expression and its binds are evaluated once and shared; relevance scoring is irrelevant for counting (`COLLECT` ignores order), so the sub-queries carry no scorer. Without an active View search the emitted facet-counts query is byte-for-byte unchanged. Covered by `ViewSearchTest` facet-counts cases + live `ViewSearchIntegrationTest::testFacetCountsFollowTheViewSearch` (also proving the View `SEARCH` is accepted inside `LET` sub-queries). Documented in `db/facets.md` and `db/search-views.md`.

- **Model-level ArangoSearch View search (Lot S4a).** A `Documents` model can now declare an ArangoSearch View through the new `AQL::VIEW` block (keys in the new `models/enums/Search` class): `Search::NAME` (required), `Search::ANALYZER` (default `identity`), `Search::FIELDS` (`field => boost` map, numeric shorthand or `[ Search::BOOST => n ]`, dotted paths, falls back to `AQL::SEARCHABLE` with boost 1), `Search::PHRASE` (exact-phrase bonus, `boost × 2`) and `Search::FUZZY` (Levenshtein distance, `0` = off).
  - **Behavior:** when the block is present and `?search=` carries a term, the list query switches automatically from the `LIKE` sweep to an index-accelerated, relevance-ranked `SEARCH` against the View — `FOR doc IN @@view SEARCH ANALYZER(BOOST(doc.f IN TOKENS(@search_N, "az"), w) OR BOOST(PHRASE(doc.f, @search_N), w×2) OR LEVENSHTEIN_MATCH(doc.f, @search_N, d) …, "az")` — keeping the `?search=` contract (comma-separated terms, OR everywhere, bound values) and the whole pipeline (`?filter`/`?facets`/conditions stay as a post-`SEARCH` `FILTER`, pagination/skins/projections untouched). The base expression uses `TOKENS()` so both sides are analyzed. Without the block, the emitted AQL is byte-for-byte unchanged.
  - **Score & count:** an active search exposes the synthetic `score` sort key (`Search::SCORE`, mirroring `?near=`/`distance`): default `score DESC`, composable (`?sort=-score,name`), dropped without an active search. `count()` follows the same branch as `list()`; `?facetCounts=` keeps the `LIKE` restriction for now (lifted by S4b).
  - **Provisioning:** the View is lazily provisioned at model initialization (mirroring `initializeCollection()`/`AQL::INDEXES`): created from the declaration when missing, never altered when it exists (no automatic drift resync — lifted by M1). New façade trait `db/traits/ViewManagementTrait` (`viewCreate`/`viewDrop`/`viewExists`) used by `ArangoDB`; new `BindTrait::bindView()`; new `AQL::VIEW` key.
  - **Tests:** unit (`ViewSearchTest`, `ViewManagementTraitTest`) + live (`ViewSearchIntegrationTest` — real provisioning at construction, boosted ranking, `list()`/`count()` agreement, explicit-sort override, fuzzy typo matching, `LIKE` fallback without the block).
  - **Docs:** new `db/search-views.md` recipe page, wired into `db/README`, `db/search.md`, `db/search-and-filtering.md`.

- **New `aqlScoredSearch()` operation (`db/operations/`).** Builds a complete, relevance-ranked search query over an ArangoSearch View in the canonical form `FOR … SEARCH … [OPTIONS { … }] LET score = BM25(…)|TFIDF(…) SORT score DESC LIMIT … RETURN …` — the ArangoSearch counterpart of `aqlVectorSearch()`. Explicit named arguments: `view`, `search`, `limit`, plus optional `analyzer` (wraps in `ANALYZER(expr, "name")`), `options` (the `SEARCH … OPTIONS { … }` object), `scorer` (`SearchScorer::BM25` default / `TFIDF`), the tunings `k` / `b` (BM25) and `normalize` (TFIDF), `offset`, and `docRef` / `scoreRef` / `return` overrides. Both scorers rank better matches higher, so the sort is always `DESC`; the score is bound to a `LET` so a custom `return` can expose it. Guards throw on an unknown scorer, `k`/`b` with TFIDF, or `normalize` with BM25. Composed from existing building blocks. Covered by `AqlScoredSearchTest` + live `SearchIntegrationTest::testAqlScoredSearchBuilder`. Documented in `aql/aql-operations.md` (new *Scored search* section).

- **Completed the `aqlSearch()` operation.** The `SEARCH` clause now supports an Analyzer wrap and the `OPTIONS` object. Two new `$init` keys next to `AQL::SEARCH`: `AQL::ANALYZER` (`'analyzer'`) wraps the expression in `ANALYZER(expr, "name")`, and `AQL::SEARCH_OPTIONS` (`'searchOptions'`) emits `SEARCH … OPTIONS { … }` — accepted as an associative array (hydrated into the new `SearchOptions` value-object), a `SearchOptions` instance, any plain/`JsonSerializable` object, or a pre-encoded JSON string. `SearchOptions` (new, `db/options/`, extends `QueryOptions`) carries `collections`, `conditionOptimization`, `countApproximate`, `parallelism`; the string values get `ConditionOptimization` (`AUTO`/`NONE`) and `CountApproximate` (`EXACT`/`COST`) constant classes. `AQL::SEARCH_OPTIONS` is deliberately distinct from `AQL::OPTIONS` (the `FOR`-level options, whose hydration silently dropped the SEARCH-specific keys before this). `aqlFor()` forwards its whole `$init` to `aqlSearch()`. First stable release. All AQL high-level operations are supported (the last one, `WINDOW`, landed in this release), alongside the full document/edge model layer, the filter / facet / group / search engine, the HTTP controllers, the ArangoDB clients (Database, Collection, Graph, View, Analyzer, Transaction, indexes including vector), and the CLI commands. ### Added

- `WINDOW` operation builder (`aqlWindow()`) — sliding-window aggregation (running totals, rolling averages, and other statistics over neighbouring rows), completing the set of AQL high-level operations. Both forms are supported, selected by the presence of `AQL::RANGE_VALUE`: row-based (`WINDOW { preceding: N, following: M } AGGREGATE …`) and range-based (`WINDOW rangeValue WITH { preceding: …, following: … } AGGREGATE …`, where the `WITH` keyword belongs to the `WINDOW` syntax, unrelated to the collection-declaring `WITH` operation). `$init` keys: `AQL::AGGREGATE` (required; reuses the same aggregate machinery as `COLLECT` via `aqlAssignments()`), `AQL::PRECEDING`, `AQL::FOLLOWING`, `AQL::RANGE_VALUE`. Bounds are emitted as-is when numeric and single-quoted when given as strings (ISO 8601 durations like `PT1H`); a cumulative running total uses the string `'unbounded'` as `preceding` (verified live: a bareword is parsed as a collection name, the quoted string is accepted). Without an aggregate the builder returns an empty string. The bounds object is serialized by a dedicated `aqlWindowBounds()` helper (one file per helper, reusable on its own). New `AQL::PRECEDING` / `AQL::FOLLOWING` / `AQL::RANGE_VALUE` enum keys. Covered by unit tests (`AqlWindowTest`, `AqlWindowBoundsTest`) and a live integration test (`WindowIntegrationTest`). Documented in `wiki/{fr,en}/aql/aql-operations.md`.

- Automatic `WITH` declaration on anonymous graph traversals — the edge-traversal methods (`EdgesGetTrait::getVertices()` and its `getOutboundVertices` / `getInboundVertices` / `getAnyVertices` / `getFirst*` wrappers, plus `EdgesCountTrait::countVertices()` and its `countOutboundVertices` / `countInboundVertices` / `countAnyVertices` wrappers) now prefix the generated query with an AQL `WITH` clause when they traverse a raw edge collection (no named graph). In a cluster, collections reached dynamically by collection-set traversals must be locked at query start to avoid deadlocks; the reachable vertex collections are derived from the model's `_from` / `_to` vertex models by direction (`OUTBOUND` → the `_to` collection, `INBOUND` → the `_from` collection, `ANY` → both, de-duplicated). Nothing is emitted for named-graph traversals (collections already known), when the vertex models are absent, or on a single server where the clause is a harmless no-op. The declared collections can be overridden through the method's `AQL::WITH` option (a collection name or an array of names). The existing `aqlWith()` helper is now wired into the query pipeline via the new private `PrepareTraversalTrait::prepareTraversalWith()`. Results are unchanged — only the generated AQL gains the leading `WITH`. The standalone `countEdgesQuery()` (a plain `_from`/`_to` edge count, not a traversal) is untouched. Documented in `wiki/{fr,en}/aql/aql-operations.md` (`aqlWith()`) and `wiki/{fr,en}/clients/graphs.md`.

- Server-side grouping / aggregation over a list query (the AQL `COLLECT` clause) — exposed over HTTP via `?groupBy=field[,field]` (CSV shortcut, implies a per-group count) and `?group={...}` (full JSON spec with the short `by` / `agg` / `count` / `sort` / `alt` keys), or in PHP via the `Arango::GROUP` option and the new `Group` vocabulary. `ListQueryTrait::buildListQuery()` injects a `COLLECT` at the canonical position (`FOR / FILTER / LET / COLLECT / SORT / LIMIT / RETURN`); when present, the document-based `SORT` and projection are replaced by a grouped `SORT` (`Group::SORT`, over the group/aggregate variables — `doc` is out of scope after `COLLECT`) and a `RETURN` derived from the spec by the new `aqlCollectReturn()` helper (group keys + aggregate keys + the `WITH COUNT` variable; scalar for a pure count; overridable via `Arango::RETURN`). Aggregates reuse the `FacetAggregator` catalogue (`sum` / `avg` / `min` / `max`), and grouping-key transforms reuse the `alt` engine (`alterExpression`, e.g. `year = DATE_YEAR(doc.created)`); per-group counting uses `WITH COUNT INTO` alone and a `LENGTH(1)` aggregate when combined with other aggregates. The raw `Arango::COLLECT` spec stays accepted as a power-user passthrough. Grouping fields come from the URL, so they are validated against AQL injection with `assertAttributeName()`, plus an optional `groupable` model whitelist/mapping (`urlKey => field`, like `sortable`, declared via `Arango::GROUPABLE`) to restrict and rename groupable dimensions. The dimensions are keys of the model's `Arango::FACETS` whitelist (the filterable facets become the counted facets, an unknown key is ignored); v1 supports the scalar `Facet::FIELD` and the array-membership `Facet::IN` family (`LIST` / `LIST_FIELD` / `LIST_FIELD_SORTED`). `FacetCountsQueryTrait` builds one `LET` sub-query per dimension (`COLLECT value = … WITH COUNT INTO count SORT count DESC RETURN { value, count }`) over the **same conjunctive filter** as the list (counts reflect the currently filtered set), and `DocumentsFacetCountsTrait::facetCounts()` runs them in a single round-trip; the controller (`PrepareFacetCountsTrait`) exposes the buckets under `options.facets`, next to `options.total`. - HTTP facets (`?facets=`) overhaul — every facetable field is declared under `Arango::FACETS` and reuses the filter operator vocabulary (`FilterComparator` / `FilterArrayComparator` + `FilterParam::OP/VAL`), with new and generalised types:
  - `Facet::IN` — array-membership facet (op-driven, default `any.in`); the former `LIST`, `LIST_FIELD`, `LIST_FIELD_SORTED` are kept as aliases.
  - `Facet::JOIN` — simple key-join existential facet (the join counterpart of `Facet::EDGE`): keeps documents having at least one joined document whose field matches the requested value, with multi-value OR and `-` negation.
  - `Facet::JOIN_COMPLEX` — key-join existential facet matching SEVERAL fields (AND) on the same joined document (object `{field: condition}`), sharing the new `HasFacetComplexConditions` builder with `EDGE_COMPLEX`.
  - `Facet::EDGE` generalised — configurable operator (`Facet::OP`) and multi-field OR (`AQL::FIELDS`), multi-value CSV (OR) and `-` negation as exclusion (`LENGTH(…) == 0`). The multi-field `like` configuration replaces the former THESAURUS facet.
  - The URL-provided sub-field names of complex facets (interpolated into `doc.<field>`) are validated by the new `isAttributeName` / `assertAttributeName` helpers — a dangerous name fails the facet (logged and skipped), so no untrusted fragment ever reaches the AQL.
- `alt` symmetric transformations on both `?filter=` and `?facets=` — `alt` now wraps the comparison **value** (right side), not only the **field** (left). The object form `alt:{ key:<chain>, val:<chain> }` gives one chain per side, and `val:true` mirrors the key-side chain (e.g. `LOWER(doc.email) == LOWER(@v)`); an array value maps the chain over each element (`@v[* RETURN LOWER(CURRENT)]`). On `?filter=` it covers the flat filters (string/number/array/date) and the nested array-expansion filters (`field[*].sub` and `match` — `LOWER(CURRENT.email) == LOWER(@v)`, with `alt` applied globally to every `match` sub-field); edge/join/document leaves inherit it through the underlying flat filter, while the structural join key stays raw. On facets the chain may come from the model definition (new `Facet::ALT`, a default for every request) or from the URL request object (`{op,val,alt}`, which takes precedence). It is wired on `FIELD`, `EDGE`, `JOIN` and `IN` (membership — where the array field is projected element-wise, `doc.tags[* RETURN LOWER(CURRENT)]`); complex facets (`EDGE_COMPLEX` / `JOIN_COMPLEX` / `ARRAY_COMPLEX`) accept a global `Facet::ALT` applied to every sub-field. The legacy field-only string/list forms (`"lower"`, `["trim","lower"]`) are unchanged. Backed by the shared free helpers `oihana\arango\db\helpers\alterExpression()` / `resolveAltSides()`, called directly by the filter and facet builders; documented with per-case examples in `wiki/{fr,en}/db/{filter,facets}.md`.
- New `AT LEAST (n)` array quantifier for `?filter=`: declared with the array-form operator `["atLeast.<cmp>", n]` (element 0 the `atLeast.<cmp>` code, element 1 the threshold, defaulting to 1), it keeps documents where at least `n` array elements satisfy the comparison — e.g. `{"op":["atLeast.ge",2],"val":80}` → `doc.scores AT LEAST (2) >= @v`. The `<cmp>` suffix reuses the filter operators (eq/ne/gt/ge/lt/le/in/nin); the threshold is cast to an int (injection-safe) and the field stays `alt`-aware. New `FilterArrayComparator::AT_LEAST*` constants.
- New `between` (range) operator for `?filter=` (number/string/date) and the `?facets=` FIELD facet: an inclusive `key >= @min && key <= @max` driven by `min`/`max` keys. Bound omission is type-driven — number/string drop the omitted side (one-sided range), while a date resolves an omitted bound to "now" (always two-sided); a date `tz` applies to both bounds (`DATE_LOCALTOUTC(@min,@tz)`). The compared field stays `alt`-aware. Backed by the shared, unit-tested `oihana\arango\db\helpers\buildBetweenClauses()` helper.
- New `Field::ALTERS` output-side transformation on the field projection: an `alt`-style chain (same `FilterFunction` vocabulary — `'lower'`, `['trim','lower']`, `['substring',0,3]`) wrapping the projected value at `RETURN` time, e.g. `'name' => [ Field::ALTERS => ['trim','lower'] ]` → `name: LOWER(TRIM(doc.name))`. The filter-side `alt` counterpart for the output, it homogenizes filters ↔ projection. Opt-in per field, applied only to the default scalar projection (`key: doc.key`) — ignored on typed/structural filters (BOOL, DATETIME, EDGE, JOIN, MAP, …); `Field::NAME` still selects the source attribute so a transformed value can be exposed under another key. Documented with worked examples in `wiki/{fr,en}/edges-joins-projection.md`.
- New aggregate facets `Facet::EDGE_AGGREGATE` and `Facet::JOIN_AGGREGATE` for `?facets=`: instead of the existential `LENGTH(…) > 0` of `EDGE`/`JOIN`, they aggregate a numeric field over all linked documents and compare it to a threshold — `AGG(FOR doc_x IN <source> [FILTER <join>] RETURN doc_x.<field>) <op> @threshold`. Driven by the request object `{agg, field, op, val}` (a bare scalar = threshold), each piece overridable on the URL and falling back to the definition (`Facet::AGG`, `AQL::FIELDS`, `Facet::OP`). The aggregator (`avg`, `sum`, `min`, `max`, `count`) is whitelisted by the new `FacetAggregator` enum (unknown ⇒ facet skipped + logged); `count` is field-less (`RETURN 1`) and generalizes the existential. A URL-provided `field` is validated with `assertAttributeName`, the threshold is numeric-cast, and the comparison is guarded by `LENGTH(…) > 0` so a `lt`/`le` threshold never lets the AQL empty-set sentinel (`null` for avg/min/max, `0` for sum/count) match documents with no linked document. New keys `Facet::AGG`, `FilterParam::AGG`, `FilterParam::FIELD`; shared `HasFacetAggregateConditions` builder; documented with worked examples in `wiki/{fr,en}/db/facets.md`.
- New conditional `alt` functions `coalesce` / `notNull`: wrap the field in AQL `NOT_NULL(...)` to substitute a default when it is missing or null before comparison — e.g. `alt:[["coalesce",0]]` → `NOT_NULL(doc.discount, 0) == @v`. The default value(s) come from the URL and are inlined as strict AQL literals via `json_encode` (always quoted/escaped, never the raw passthrough of `aqlValue()`), so a default is always treated as literal data — injection-safe.
- New `pluck` `alt` function (and the `oihana\arango\db\functions\arrays\pluck()` helper): projects an array of objects onto a single sub-field via AQL's inline projection `array[* RETURN CURRENT.<field>]`. Chained with an aggregate it reduces over a property of embedded objects — e.g. `alt:[["pluck","price"],"avg"]` → `AVERAGE(doc.items[* RETURN CURRENT.price])`. The sub-field name comes from the URL and is validated with `assertAttributeName` before interpolation (injection-safe; an unsafe name throws `ValidationException`).
- New AQL array-expansion helpers `oihana\arango\db\functions\arrays\arrayMap()` (`array[* RETURN expr]`) and `arrayFilter()` (`array[* FILTER cond]`), centralizing the previously hand-built `[* … ]` string literals. The five existing sites now build through them — `pluck()` (a thin `arrayMap` wrapper over a validated sub-field), the array-valued `alt` projection in `FilterTrait`, the `Facet::IN` field-side projection in `HasFacetIn`, and the two array-expansion existentials in `HasFilterArray` (`LENGTH(arrayFilter(...)) > 0`). Pure refactor — the emitted AQL is unchanged — plus direct unit tests for both helpers.
- New AQL array "question mark" helper `oihana\arango\db\functions\arrays\arrayContains()` (`array[? <quantifier> FILTER cond]`): the boolean array-contains operator, the direct/idiomatic form of `LENGTH(array[* FILTER cond]) > 0`, with optional `ANY` / `ALL` / `NONE` / `AT LEAST (n)` quantifiers (omitted = at least one). Provided as a primitive (with direct unit tests) — notably to express quantified conditions over arrays of objects (e.g. `doc.reviews[? AT LEAST (3) FILTER CURRENT.rating >= @v]`); the existing `LENGTH(...) > 0` existentials are left unchanged.
- Opt-in live ArangoDB integration suite (`#[Group('integration')]`, run via `vendor/bin/phpunit -c phpunit-integration.xml`): builds a disposable database from `configs/config.toml`, seeds it, executes the generated AQL for real, then drops it — and skips cleanly (never fails) when no instance is reachable. Covers the facet builders, the `alt` transformations, and the embedded array operations (`DocumentsArrayTrait`: set dedup, sorted-set ordering, empty `[]`, move, object deep-equality, `[]` seeding, `arrayPurgeRef`) end-to-end, complementing the server-free unit suite that only freezes the AQL string.

### Changed

- `Facet::FIELD` is now operator-enabled (`FilterComparator`): it accepts an `op` (in the definition via `Facet::OP`, or per request via an `{op, val}` object), defaulting to `match` (`=~`) for backward compatibility, with generic negation. Numeric scalars keep their type, so `doc.price >= 100` never binds a string.
- The list facet family is unified under `Facet::IN` (op-driven membership); `Facet::LIST`, `LIST_FIELD` and `LIST_FIELD_SORTED` become thin aliases, and the broken `length` branch was removed. The facet dispatcher was hardened (nullable type lookup, null-safe logger).

### Removed

- `ListItemTrait` / `MultiFieldTrait` (`models/traits/`) — orphaned, untested legacy traits composed by `Documents` but never called, and generating invalid AQL (a bind parameter used as an object attribute name; a hardcoded `doc.hasPart` filter). Replaced by `DocumentsArrayTrait` (see Added): `deleteListItem`/`insertInMultiField`/… map to `arrayRemove`/`arrayInsert`/…, `deleteReverseInMultiField` → `arrayPurgeRef`, and `updateDateParentMultiField` is dropped in favour of an application-level `afterUpdate` listener (documented in `wiki/{fr,en}/db/arrays.md`).
- `PositionTrait` (`models/traits/aql/`) — dead, half-written trait: it was composed by no model, `preparePosition()` was never called, and its only AQL form had a hardcoded `@value` placeholder (no `$binds`/value parameter) plus an unfiltered `FOR doc_coll IN <collection>` scan. Removed the trait and its characterization test rather than freeze unused, broken output; a position-ranking feature, if ever needed, should be designed fresh with proper binding.
- `Facet::THESAURUS` — it was exactly an `EDGE` facet with `AQL::FIELDS = '_key,name,alternateName'` and `op = like`, so it is now expressed as that configuration rather than a dedicated type.
- Magic `id → _key` rewriting in facets — key/property aliases are now declared explicitly via `Facet::PROPERTY` everywhere, with no hidden remapping.

### Fixed

- `aqlCollect()` could emit invalid AQL combining `AGGREGATE` and `WITH COUNT INTO`, which are mutually exclusive: when both keys were supplied it appended both clauses. `AGGREGATE` now takes precedence and `WITH COUNT INTO` is dropped (to count alongside aggregates, express the count as a `LENGTH(1)` aggregate). The only caller (`EdgesCountTrait`, which passes `WITH COUNT` alone) is unaffected. The companion `aqlCollectReturn()` mirrors the same rule.
- `aqlUpsertExpression()` never rejected an `init` that defined **both** `FILTER` and `SEARCH`: the second guard repeated the first one's condition (`!isset($filter) && !isset($search)`) instead of detecting the both-defined case, so it was dead code and a conflicting init silently fell through to `$search ?? $filter` (SEARCH wins). The guard is now `isset($filter) && isset($search)` and throws `FILTER and SEARCH cannot be defined at the same time.`, matching the documented exclusive `UPSERT [ searchExpression | FILTER filterExpression ]` syntax. The two guards are split into independent `if` clauses. Calls passing exactly one of the two (every existing usage) are unchanged. Characterized in the new `AqlUpsertExpressionTest`.
- `removeValue()` array helper emitted the plural `REMOVE_VALUES` instead of `REMOVE_VALUE`, so the single-value `FilterFunction::REMOVE` (`remove`) silently no-op'd on a scalar (its second argument must be an array) and raised an AQL error when an occurrence limit was passed; it now emits `REMOVE_VALUE`, and the optional limit works as documented. The `$limit` parameter also accepts `int|string|null` and ignores non-numeric/non-positive values instead of raising a `TypeError` at call time (e.g. a `remove` filter built from a query string). `FilterFunction::REMOVES` (`removes`) keeps using `REMOVE_VALUES`.
- `ArangoCommand` was uninstantiable whenever its `init` array omitted the `directory` key: the constructor's fallback `$this->directory = $init[…] ?? $this->directory` read the `$directory` property *before initialization* (declared `public ?string $directory ;` with no default in `ArangoDumpAction` / `ArangoRestoreAction` / `ArangoListDumpsAction`), throwing `Error: Typed property … must not be accessed before initialization`. The triplicated declaration is extracted into a single `oihana\arango\commands\traits\DirectoryTrait` (property defaulted to `null` + an `initializeDirectory()` initializer matching the `initializeEncrypt()` / `initializePassphrase()` pattern), the three actions now `use DirectoryTrait`, and the constructor calls `$this->initializeDirectory( $init )`. Behavior for configs that already supplied a directory is unchanged.
- `ArangoDB` façade hardening: `getCursor()` was typed `: Cursor` (non-nullable) but `$cursor` is null before the first `execute()` and after `streamDocuments()` resets it, so calling it in those states threw a `TypeError` — it now returns `?Cursor`. `getExtra()` / `getFoundRows()` likewise dereferenced a possibly-null cursor and now return `[]` / `0` when no query has run. The constructor caught only `ArangoException`, but the client builder throws `InvalidArgumentException` for an invalid config (e.g. empty endpoint), so build failures were never logged; the `catch` is widened to `Throwable` (logged, then rethrown unchanged).
- `CollectionManagementTrait` (`ArangoDB` façade) called an **undefined** `toLegacyException()` in the `catch` blocks of `collectionRename()`, `getIndex()` and `getIndexes()` — a leftover shim from the migration off the official ArangoDB-PHP fork. On any client error those three methods crashed with `Error: Call to undefined method` instead of surfacing the failure. The dead `try/catch` wrappers are removed so the `oihana\arango\clients\exceptions\ArangoException` propagates directly, and the redundant `ArangoException as ArangoClientException` import alias (both names resolved to the same class) is dropped. The boolean / log-and-swallow methods are unchanged.
- `PropertyController` PATCH in `raw` mode (`init[Arango::RAW] = true`) returned `null` instead of the submitted property value: the handler read the prepared payload with object-property syntax (`$payload->{$this->property}`), but `PayloadsTrait::propertyPayload()` yields an **array** (`[ property => value ]`), so the access always resolved to `null`. It now reads the value with `oihana\core\accessors\getKeyValue( (array) $payload , $this->property )`, which handles arrays and objects uniformly. Non-`raw` mode (which reloads via `model->get()`) is unchanged. Characterized then fixed in `PropertyControllerTest`.
- `aqlFields` `Field::QUOTED`: it double-quoted the key used for BOTH the output label and the attribute access, emitting invalid AQL like `"my-key":doc."my-key"` (an attribute with special characters cannot be reached with double quotes — confirmed rejected by ArangoDB with a syntax error). The attribute access now uses backticks (`"my-key":doc.`my-key``), the valid form; only the output label is double-quoted. Cases pairing `Field::QUOTED` with `Field::NAME` or `Field::ALTERS` are unchanged. Characterized then fixed in `AqlFieldsTest`.
- AQL `like()` helper: its third parameter was named `$caseSensitive` but maps to ArangoDB's `LIKE(text, search, caseInsensitive)` third argument, so it did the opposite of its name (`true` produced a case-INSENSITIVE match). Renamed to `$caseInsensitive` (default `false` = case-sensitive, matching AQL). The emitted AQL is unchanged for every call path — `?search` stays case-insensitive and the `like` filter-function alter keeps its output — only the (now honest) parameter name and the docs change. Added the previously missing direct unit test, and fixed the stale `SearchTraitTest` comment that described the predicates as "case-sensitive".
- `command:arangodb` document `upsert` now honors the `removeKeys` exclusion list: it was passed under the `excludes` key, which the upsert query never reads, so the exclusion silently had no effect.
- `ArangoCommand` registered the wrong signals for event dispatch: it passed `(int) [SIGINT, …]` (which casts to `1`) to the variadic `setSignalsToDispatchEvent()`. The intended signal list is now passed.
- Removed the unused `ignoreCollection` arangodump option (`ArangoDumpOptions::$ignoreCollection` / `ArangoDumpOption::IGNORE_COLLECTION`): no such flag exists in `arangodump`, and exclusion is resolved client-side as a `--collection` complement. The `--ignore-collection` CLI flag is unchanged.
- `ArangoConfigTrait` now validates the connection settings with exceptions instead of `assert()` (which is compiled out when `zend.assertions=-1`), so a missing database/endpoint/username never silently flows on as null. Properties are initialized; an empty password remains valid.
- Declare the previously-undeclared `ext-pcntl` requirement in `composer.json` — `ArangoCommand`'s signal handling (`getSubscribedSignals()` / `handleSignal()` / `setSignalsToDispatchEvent()`) relies on the `SIGINT` / `SIGTERM` / `SIGUSR1` / `SIGUSR2` / `SIGALRM` constants provided by the extension.
- Security: `command:arangodb dump` / `restore` no longer run `arangodump` / `arangorestore` through a shell. The command was assembled as a single string (option values quoted only with `json_encode`) and passed to `system()`, so a value containing shell metacharacters (password, database, endpoint, directory, collection name) could break the command or inject arbitrary shell. The processes are now executed via `proc_open()` with an explicit argument vector (no shell), so values are passed verbatim. New `ArangoProcessTrait` (`optionsToArguments()` + `runProcess()`), covered by `ArangoProcessTraitTest` including a metacharacter-stays-one-argument unit test and a no-shell probe.
- `command:arangodb restore --last` could never locate the archive: it set the input file to the bare filename returned by `findFiles()` (whose `FILTER` yields `$file->getFilename()`), so `assertFile()` failed unless the current working directory happened to be the dump directory. The `--last` branch now prepends the dump directory, matching the interactive branch.

- `command:arangodb restore --date <date>` could never locate the archive. The by-date suffix was built as `Char::DASH . $database . $shouldEncrypt ? FileExtension::TAR_GZ_ENCRYPTED : FileExtension::TAR`, but `.` binds tighter than the ternary, so PHP evaluated `(Char::DASH . $database . $shouldEncrypt)` as the condition — always a non-empty (truthy) string. The suffix therefore always collapsed to `.tar.gz.enc` and silently dropped the `-{database}` segment, so a non-encrypted restore-by-date never matched its `{date}-{database}.tar.gz` file. The non-encrypted branch also used `FileExtension::TAR` (`.tar`) instead of `FileExtension::TAR_GZ` (`.tar.gz`), which the dump action actually produces. The suffix logic is now isolated in the testable `ArangoRestoreAction::getArchiveFileSuffix()` helper and covered by `ArangoRestoreActionTest`.

- `Facet::ARRAY_COMPLEX` was broken at query time: it iterated an undefined variable (`FOR doc_workshops IN resultworkshops`, with no separator) and built its filter on the out-of-scope `result` document, so the facet never ran. It now iterates the in-scope embedded array (`doc.<key>`) and returns `1`. Confirmed against a live ArangoDB.
- `Facet::EDGE_COMPLEX` used a malformed loop variable (`AQL::DOC_REF . $key` → e.g. `docRefnumbers`) instead of the sibling convention `doc_<key>`; it now reads `doc_<key>` like the other builders, and gained per-field multi-value (OR) plus per-value inline negation (`!=`) on the same traversed vertex.
- Date filters with a valid `tz` always threw: the timezone-conversion path passed the bound placeholder (`@tz`) to `dateLocalToUTC()`, whose own timezone re-validation then rejected it (the literal timezone is already validated upstream by `isValidTimezone()`). The re-validation is now disabled there (`$throwable = false`), so `?filter={...,"tz":"Europe/Paris"}` emits `DATE_LOCALTOUTC(@v,@tz)` instead of crashing — and the new date `between` honors `tz` on both bounds.

### Security

- `?sort=` in open mode (no `sortable` whitelist) interpolated the raw URL key into `doc.<key>` without validation — an AQL injection vector (a model that exposes `?sort=` without configuring `$sortable`, the default). `SortTrait::prepareSort()` now validates each open-mode sort key with `assertAttributeName()`; the whitelisted path was already safe. Legitimate dotted paths (`address.city`) still sort; an unsafe key throws `ValidationException`.
- `aqlFields()` (the projection builder) interpolated the source attribute (`Field::NAME ?? key`) into `doc.<attr>` without validation. Although projection fields are server-config today (`?fields=` is not wired to any controller), the bare source is now guarded by `assertAttributeName()` as defense-in-depth, so a future URL-driven projection cannot inject AQL. The two legitimate non-identifier paths are preserved: the `Field::QUOTED` escape hatch (a special key reached via backticks, `doc.`my-key``) and numeric keys.
- HTTP facets enforce a strict trust contract: only bound values (`@bind`) are user-controlled. Operators are whitelisted (`getAlias`, falling back to the type default), facet keys are whitelisted against the model's `Arango::FACETS` (an unknown key is ignored), and the URL-provided sub-field names of complex facets are validated by `assertAttributeName` before being interpolated — a dangerous name fails the facet (logged and skipped). `alt` function names are likewise whitelisted (an unknown function is a no-op), so no `?filter=` / `?facets=` input can inject AQL.
