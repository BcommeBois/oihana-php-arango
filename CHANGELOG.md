## [Unreleased]

### Fixed

- **`returnFields()` anchors relation `LET` sub-queries on the projected document when projecting through a vertex traversal.** When a model's declared fields include a relation (`Filter::EDGES` / `EDGE` / `EDGES_COUNT` / `JOIN` / `JOINS`) and that model is projected as the **target of a vertex traversal** (`Edges::getVertices()` and its inbound/outbound variants, which call `returnFields()` with `DOC_REF` = the traversal loop variable, e.g. `vertex`), the generated relation `LET` sub-queries started from an **unbound** `doc_`-prefixed alias (`doc_vertex`) while the `RETURN` projection used the bare loop variable (`vertex`) ŌĆö the two diverged, and ArangoDB raised `collection or view not found: doc_vertex`. The cause: the prepared-fields branch of `returnFields()` prefixed the document reference with `AQL::DOC_PREFIX` before handing it to `buildVariables()`, but nothing binds a `doc_<ref>` loop in that path (unlike the facet helpers, which open their own correlated `FOR doc_<key>`), and the sibling `RETURN` projection (`aqlFields()`) used the un-prefixed reference. The branch now passes `$docRef` verbatim to both, so the relation `LET`s and the projection anchor on the **same** variable ŌĆö exactly as the `*` (whole-document) branch already did. In a main query (`DOC_REF` = `doc`) the reference is unchanged, so `list()` / `get()` output is identical.
  - **Tests:** unit (`FieldsTraitTier2Test` ŌĆö an `EDGES_COUNT` and an `EDGES` field projected through a traversal (`DOC_REF: 'vertex'`) each emit a `LET` whose inner `FOR ŌĆ” IN OUTBOUND vertex ŌĆ”` starts from the outer loop variable, with no `doc_vertex` alias; plus a non-regression check that the same `EDGES_COUNT` in a main query still anchors on `doc`).

- **`Filter::EDGES_COUNT` now composes inside a vertex traversal.** A model declaring a `Filter::EDGES_COUNT` field worked in `list()` / `get()` but raised an `AQL: variable 'vertex' is assigned multiple times` error the moment that model was the **target** of a traversal (`Edges::getVertices()` / `getOutboundVertices()` / `getInboundVertices()`). The reason: the count sub-query, built by `buildEdgeCountVariable()`, hard-coded its inner loop as `FOR vertex IN <dir> <start> <edge> RETURN vertex`. In `list()` / `get()` the outer model loop is `doc`, so no clash ŌĆö but a vertex traversal's outer loop is already named `vertex` (`AQL::VERTEX`), and when the target model re-emitted its count `LET` over that traversed vertex, the inner `FOR vertex` collided with the outer one. The inner count loop now uses a **unique** variable derived from the `LET` name (`<name>_v`, itself a `randomKey` in the live flow) instead of the shared `vertex`; the `LET` variable name ŌĆö i.e. the projected field's public API ŌĆö is unchanged, and the count semantics are identical. The two sibling builders were already safe (`buildEdgeVariable()` / `buildJoinVariable()` derive their loop names from `randomKey()`); only the count helper reused the bare `vertex`.
  - **Tests:** unit (`BuildEdgeCountVariableTest` ŌĆö a count projected with the outer traversal loop name (`startVertex: 'vertex'`) yields two distinct `FOR` loops, the inner one **not** reusing `vertex`; the default and unique-name snapshots updated to the `<name>_v` inner loop; `BuildVariablesTest` ŌĆö the `EDGES_COUNT` field snapshot updated likewise).

- **Analyzer diff ŌĆö a declared `pipeline` no longer drifts on its sub-analyzer defaults.** `analyzerDiff()` compares a declaration to the server with **subset** semantics: it iterates only the declared keys, so the server defaults a declaration omits (a `norm` declared `{ locale }` reads back `{ locale, case, accent }`) are ignored ŌĆö but only because the top-level loop walks declared keys one by one. A `pipeline` analyzer nests its chain inside a **list** (`properties.pipeline`), and the list was compared by deep equality, so each sub-analyzer the server fills with all its defaults (`norm` ŌåÆ `case` / `accent`; `ngram` ŌåÆ `startMarker` / `endMarker` / `streamType`) diverged from the declaration and the pipeline showed up as a **permanent false drift** ŌĆö the same bug the `identity` analyzer hit on View links. The pipeline list now gets its own declaration-oriented comparison (`comparePipeline()`): members are matched by **position** (the chain order is significant ŌĆö a reordered chain *is* a real drift) and by type, and only the **declared** sub-properties are checked, server defaults ignored. `normalizeAnalyzerValue()` is untouched, so other list properties (e.g. `stopwords`) keep their strict equality. A `RawAnalyzer( 'pipeline', ŌĆ” )` rebuilt from a server dump (all defaults present) stays a valid round-trip.
  - **Tests:** unit (`AnalyzerManagementTraitTest` ŌĆö a declared `norm` ŌåÆ `ngram` pipeline IN_SYNC against the server-defaulted chain; real drift still caught on a changed sub-analyzer type, a reordered chain, a different chain length, and a changed sub-property) + live (`PipelineAnalyzerSyncIntegrationTest` ŌĆö on a real server the created pipeline re-diffs IN_SYNC despite the server filling every sub-analyzer default, and a `RawAnalyzer` dumped from that server round-trips too).

- **Federated search ŌĆö the rebuild now restricts to the matched keys.** `FederatedSearch::rebuild()` asked each model for its page through the public `?filter=` channel (`Arango::FILTER` with `_key IN [ŌĆ”]`), which only applies to a model's **whitelisted** filterable fields ŌĆö and `_key` is almost never one, so the filter was silently dropped and `list()` (called without a `LIMIT`) rebuilt the **entire collection**, leaving the match-driven re-merge to trim the page. Correct, but it loaded a whole collection per model on every federated search (invisible on the tiny test collections, severe at scale). The key restriction now travels through the **trusted internal condition** channel (`AQL::CONDITIONS` + `AQL::BINDS` ŌåÆ `FOR doc IN coll FILTER doc._key IN @keys ŌĆ”`), which bypasses the field whitelist and always applies, so each model fetches **only** the page's keys. Behaviour is otherwise identical (the re-merge still orders by score); this also benefits the type-aware composite routing (one trusted condition per resolved model).
  - **Tests:** unit (`FederatedSearchTest` ŌĆö the rebuild restriction asserted through the bound key set on both the direct and composite paths) + live (`SearchAliasViewIntegrationTest` ŌĆö on a real server each model's `list()` returns **only** the keys it was asked to rebuild, not the whole collection).

- **`command:arangodb` now reads the `analyzers` (and `searchAliasViews`) init keys.** The `ArangoCommand` constructor wired only some of its declarative registries (`models`, `collectionIndexes`, ŌĆ”) but silently dropped the `analyzers` and `searchAliasViews` init keys: a host-declared custom-analyzer registry never reached `getAnalyzerDefinitions()` (it stayed `[]`), so `arango:analyzers --sync` reported ┬½ No analyzers configured ┬╗ and created nothing, the `doctor` analyzers pass reconciled nothing, and a View referencing the analyzer stayed ┬½ invalid: analyzer not found ┬╗. The same omission left a host-declared `search-alias` view registry empty for the `views` action. Both keys are now read in the constructor like the other registries (`$this->analyzers = $init[ArangoCommandParam::ANALYZERS] ?? $this->analyzers`). **Strictly backward-compatible**: without the key the registry stays its empty default. The trait, property, getter and consuming actions all already existed ŌĆö only the one constructor line per key was missing.
  - **Tests:** unit (`ArangoCommandTest` ŌĆö each registry is wired from its init key, a lone `AnalyzerDefinition` is normalized, a missing key defaults to `[]`, plus a regression guard asserting the constructor wires *every* declared registry ŌĆö `analyzers` / `collectionIndexes` / `models` / `searchAliasViews` ŌĆö so the next added registry cannot slip through unwired) + live integration (`AnalyzerCommandIntegrationTest` ŌĆö an analyzer declared through the **real command's** `analyzers` init key is reported missing, created by `--sync` on a real server, then in sync; previously this run printed ┬½ No analyzers configured ┬╗).

### Added

- **`buildTree()` / `buildTreeAlter()` ŌĆö rebuild a nested `children[]` tree from a flat hierarchy projection.** Two new autoloaded helpers close the hierarchy feature: `oihana\arango\models\helpers\buildTree()` turns the flat parent-linked list produced by a depth-ranged edge projection (Lot A) into a nested `children[]` tree, and `buildTreeAlter()` wires that reshape as an `Alter::MAP` so the tree is delivered automatically in the response. `buildTree( $flat, $parentSource = AQL::_PARENT, $rootKey = null, $childrenKey = 'children', $keyField = Schema::_KEY )` is a **pure O(n)** reconstruction (no extra query): it groups nodes by parent and descends from the root, is **cycle-safe** (a node already on the current branch is never descended into again ŌåÆ pathological self-referential data cannot loop), and infers the roots when no `$rootKey` is given (a node whose parent is absent from the list). It reads the parent link from either the `AQL::WITH_PATH`-injected `_parent` (default) or a stored parent field (`parentSource: 'broader'`). `buildTreeAlter( $parentSource, $childrenKey, $keyField )` returns a callback matching the `Alter::MAP` signature that roots the tree on the enclosing document's `_key`, returns non-array values untouched, and reads the root from an array **or** object document. Declared together with `AQL::MAX_DEPTH` + `AQL::WITH_PATH` on the edge, a `Field::ALTERS => [[ Alter::MAP, buildTreeAlter() ]]` yields a fully nested hierarchy from a **single** traversal plus an in-memory reshape, at any depth. Single-parent per node (guaranteed under `AQL::WITH_PATH`); a polyhierarchy stays on the flat `?filter=` surface.
  - **Tests:** unit (`BuildTreeTest` ŌĆö explicit-root nesting, inferred roots, custom parent source / children key / identity field, the cycle guard terminating on `A Ōåö B`, non-array rows ignored, an orphan becoming a root, empty list; `BuildTreeAlterTest` ŌĆö the returned callable roots the tree on the document `_key` from an array and an object document, passes non-array values through unchanged, and honours a custom parent source / children key).
  - **Docs:** FR/EN `edges-joins-projection.md` *Hierarchical traversal* section gains a *Rebuilding the tree* sub-section (the `buildTree()` helper, the `buildTreeAlter()` `Alter::MAP` wiring, the end-to-end A+B+C declaration, and the single-parent/polyhierarchy scope note).

- **`AQL::WITH_PATH` injects reconstruction metadata (`_parent` / `_depth`) into a depth-ranged edge projection.** The depth traversal added above returns a **flat list**; rebuilding a nested tree needs each node's parent. When the document does **not** store its parent (the link lives only in the edges), declare `AQL::WITH_PATH => true` on the edge definition: `buildEdgeVariable()` then exposes the traversal `path` variable and injects two computed keys into each projected element ŌĆö `_parent` (`AQL::_PARENT`, the immediate parent's `_key` = `path.vertices[-2]._key`) and `_depth` (`AQL::_DEPTH` = `LENGTH(path.edges)`). It applies to both object shapes: a field projection gains the two keys in its `RETURN { ŌĆ” }`, and a whole-vertex projection grafts them with `MERGE(vertex, { _parent, _depth })`. A scalar `Arango::PROPERTY` projection has no object to carry them, so `AQL::WITH_PATH` is **ignored** there (and no `path` variable is emitted). New constants `AQL::WITH_PATH` / `AQL::_PARENT` (`_parent`) / `AQL::_DEPTH` (`_depth`). **Strictly backward-compatible**: off by default ŌåÆ no `path` variable, byte-for-byte unchanged AQL. A depth-1 node's `_parent` is the start-vertex key ŌĆö the root for tree reconstruction.
  - **Tests:** unit (`BuildEdgeVariableTest` ŌĆö `WITH_PATH` injects `_parent` / `_depth` into a field projection and exposes the `path` loop variable; the whole-vertex branch grafts them via `MERGE`; a scalar `PROPERTY` projection ignores it and emits no `path` variable; without `WITH_PATH` no metadata and no `path` variable appear).
  - **Docs:** FR/EN `edges-joins-projection.md` *Hierarchical traversal* section gains a *Reconstruction metadata* sub-section (the stored-parent vs path-computed choice, the injected keys, the three projection shapes).

- **`Filter::EDGES` projections accept a traversal depth (`AQL::MIN_DEPTH` / `AQL::MAX_DEPTH`).** An edge projection followed the relation **one level deep** only (direct children/parents). A self-referential relation ŌĆö a hierarchy such as a thesaurus, a category tree or an org chart ŌĆö can now be followed across **several levels in a single traversal** by declaring a depth on the edge definition: `AQL::EDGES => [ 'descendants' => [ AQL::MODEL => 'concept_links', AQL::DIRECTION => Traversal::OUTBOUND, AQL::MAX_DEPTH => 5 ] ]` generates `FOR vertex, edge IN 1..5 OUTBOUND doc concept_links ŌĆ”` and returns the **flat list** of all descendants (or, with `Traversal::INBOUND`, ancestors) up to depth N. `buildEdgeVariable()` now reads `AQL::MIN_DEPTH` / `AQL::MAX_DEPTH` from the edge definition and forwards them to `aqlTraversal()` (which already supported the range). **Rules:** `AQL::MAX_DEPTH` alone defaults the lower bound to `1` (`1..N`); `AQL::MIN_DEPTH` alone is **rejected** (`UnexpectedValueException`) because ArangoDB requires a bounded range and an unbounded self-referential traversal risks a runaway cycle. **Strictly backward-compatible**: with neither key the traversal stays at depth 1 and the generated AQL is byte-for-byte unchanged. Homogeneous (self-referential) relations only ŌĆö a heterogeneous `Type1 ŌåÆ Type2 ŌåÆ Type3` chain stays expressed as declared nested edges, one level per type.
  - **Tests:** unit (`BuildEdgeVariableTest` ŌĆö `MAX_DEPTH` alone renders `IN 1..5 OUTBOUND`; an explicit `MIN_DEPTH`+`MAX_DEPTH` renders the exact range; `INBOUND` ascends with a range; no depth keeps the legacy rangeless AQL; `MIN_DEPTH` without `MAX_DEPTH` throws).
  - **Docs:** FR/EN `edges-joins-projection.md` gains a *Hierarchical traversal* section (depth declaration, descend/ascend direction, the defaults/guard rules, the flat-list output and the homogeneous-only scope).

- **`AQL::SORTABLE` accepts a compact indexed notation (shorthand + alias).** A model's sort whitelist no longer has to be a fully associative `urlKey => fieldPath` map where, in the overwhelmingly common case, the token and the field are identical (`Prop::CREATED => Prop::CREATED`). Three interchangeable notations are now accepted and may be **mixed in the same array**: the **indexed shorthand** ŌĆö a plain `[ Prop::_FROM, Prop::_TO, Prop::CREATED ]` list where each token resolves to the field of the same name; the **indexed alias** ŌĆö a single-pair `[ Prop::NAME => Prop::GIVEN_NAME ]` element for the rare case where the public `?sort=` token differs from the AQL field (`?sort=name` ŌåÆ `doc.givenName`); and the **associative (legacy)** map, unchanged. The pair direction stays `[ urlKey => fieldPath ]` across every form, so the legacy map and the new indexed forms speak the same language. A new `oihana\arango\models\helpers\normalizeSortable()` helper (autoloaded via `composer.json` `files`) folds any of the three into the canonical `urlKey => fieldPath` map; `initializeSortable()` applies it once at construction, so the hot `prepareSort()` path is untouched. Field values may still be a multi-segment path (`[ 'address', 'city' ]` ŌåÆ `address.city`), in shorthand-alias form too. **Why it matters:** the redundant `field => field` map existed only to *leave room* for the very rare alias, bloating every model's config for no benefit; the shorthand removes the redundancy while the alias stays expressible. **Strictly backward-compatible**: `normalizeSortable()` is idempotent and returns an already-associative map untouched (and `null` open mode is preserved), so every existing `SORTABLE` declaration behaves exactly as before.
  - **Tests:** unit (`NormalizeSortableTest` ŌĆö the three notations and their hybrid mix folded to the canonical map; multi-segment field paths kept in both legacy and alias form; a pure-list indexed element with no token dropped; a non-string/non-array indexed value ignored; `null` preserved; idempotency. `SortTraitTest` ŌĆö `initializeSortable()` normalizes the indexed shorthand and the hybrid alias and the resulting whitelist resolves `?sort=` correctly, an out-of-whitelist token is still dropped, and the absent key keeps open mode `null`).

- **`Alter::MAP` callbacks now receive the request context (skin, locale, ŌĆ”).** The five fetch seams of `ArangoTrait` ŌĆö `getDocuments()` / `getFirstResult()` / `getObject()` / `getResult()` / `streamDocuments()` ŌĆö gain an optional trailing argument `array $context = []`, forwarded to `alter()` (php-models Ōēź Unreleased) and thus to every `Alter::MAP` callback as its **6th argument**. The three read entry points hand their whole `$init` through as the context: `list()` ŌåÆ `getDocuments( ŌĆ” , context: $init )`, `get()` ŌåÆ `getObject( ŌĆ” , context: $init )`, `stream()` ŌåÆ `streamDocuments( ŌĆ” , context: $init )`. A `MAP` callback can therefore read `$context[ Arango::SKIN ] ?? Skin::DEFAULT` and pick its mapping strategy from the **current** request skin (or any other `$init` key) instead of a hard-coded one ŌĆö e.g. enriching a sub-document with the same skin the caller asked for. **Why it matters:** before this, an `Alter::MAP` enrichment had no way to know the skin of the request that produced the document, so it could only re-fetch related models with a fixed skin. The skin is consumed early (at query build, by `filterFieldsBySkin()`) and was lost before `alter()` ran; passing `$init` as the context re-exposes it ŌĆö and everything else in the request ŌĆö to the alteration stage, with no mutable shared state (the context is threaded by value).
  - **Backward compatibility:** the parameter defaults to `[]`, and a `MAP` callback that does not declare the trailing `$context` keeps working unchanged (PHP discards the surplus argument). **One caveat for overriders:** because the executor signatures grew an optional parameter, any subclass that **overrides** `getDocuments` / `getFirstResult` / `getObject` / `getResult` / `streamDocuments` must add the matching `array $context = []` to stay signature-compatible (PHP fatals otherwise) ŌĆö this only affects code that *reimplements* a seam (in practice, test doubles), never code that *calls* it. `Alter::CALL` deliberately does **not** receive the context (its callables are frequently native PHP functions that reject extra arguments); promote a `CALL` to `Alter::MAP` if it needs the context.
  - **Requires:** `oihana/php-models` with the `alter( ŌĆ” , array $context = [] )` signature.
  - **Tests:** unit (`ArangoTraitTest` ŌĆö every seam forwards its `$context` to the real `alter()`, captured through an `Alter::MAP` callback's 6th argument; `DocumentsListTraitTest` / `DocumentsGetTraitTest` / `DocumentsStreamTraitTest` ŌĆö `list()` / `get()` / `stream()` each forward the whole `$init` as the alteration context).

- **Typed `PipelineAnalyzer` ŌĆö compose analyzers into an ordered chain.** A new `oihana\arango\clients\analyzer\PipelineAnalyzer` value object (`readonly`, `implements AnalyzerOptions`) serialises ArangoDB's `pipeline` analyzer type, which runs an **ordered** chain of sub-analyzers, each fed the previous one's output. Its constructor takes a single `array $pipeline` of other {@see AnalyzerOptions} value objects in significant order (e.g. `[ new NormAnalyzer( locale: 'fr', case: 'lower', accent: false ), new NgramAnalyzer( min: 3, max: 5, preserveOriginal: true ) ]`) and `toArray()` emits `{ type: 'pipeline', properties: { pipeline: [ ŌĆ”each member->toArray() ] } }`. **Why it exists:** a standalone `ngram` analyzer normalises **neither case nor accents**, so a field stored in upper case (`"L'ABSIE"`) indexes upper-case n-grams while a lower-case query (`l'ab`) never matches ŌĆö case-/accent-insensitive autocomplete is silently broken. The clean server-side fix is a `pipeline` that runs `norm` (lower-case + accent fold) **before** `ngram`; until now the only way to express it was the untyped `new RawAnalyzer( 'pipeline', ŌĆ” )`. New enum entries `AnalyzerType::PIPELINE` (`'pipeline'`, promoted from the deferred list to V1) and `AnalyzerField::PIPELINE` (`'pipeline'`, the `properties.pipeline` sub-analyzer list). Validation: a non-empty pipeline whose every member is an `AnalyzerOptions`, else `InvalidArgumentException` with the offending index.
  - **Tests:** unit (`PipelineAnalyzerTest` ŌĆö the exact `{ type, properties: { pipeline } }` shape; `norm` ŌåÆ `ngram` order preserved and each member serialised; the empty pipeline and a non-`AnalyzerOptions` member both rejected; `AnalyzerTypeTest` / `AnalyzerFieldTest` ŌĆö the new `pipeline` constants exposed).
  - **Docs:** FR/EN `db/analyzers.md` gains a dedicated *Pipeline analyzer* section ŌĆö the `norm` ŌåÆ `ngram` pattern for case-/accent-insensitive autocomplete, **why** a standalone `ngram` silently matches nothing on upper-cased data, a full `createAnalyzer()` example, and the **false-drift trap** (don't declare the sub-analyzer defaults ŌĆö `analyzerDiff()` compares the chain declaration-first, by position, ignoring server defaults). The "five types" heading/table becomes "six" and `pipeline` leaves the deferred list.

- **`StreamType` enum for the `ngram` analyzer's `streamType`.** A new `oihana\arango\clients\analyzer\enums\StreamType` constants class (mirroring `CaseFolding`) replaces the `'binary'` / `'utf8'` magic strings with `StreamType::BINARY` / `StreamType::UTF8`. `NgramAnalyzer`'s `streamType` parameter stays `?string`, so it is **purely additive** ŌĆö you may now pass `StreamType::UTF8` instead of `'utf8'` for autocompletion and typo-safety. The matching `locale` magic string (`'fr'`) is already covered by `org\iso\ISO639_1::FR` from `oihana/php-standards` (a dependency), reused rather than duplicated.
  - **Tests:** unit (`StreamTypeTest` ŌĆö the canonical `binary` / `utf8` values, `enums()` lists both, `includes()` accepts them and rejects anything else).
  - **Docs:** FR/EN `db/analyzers.md` ŌĆö the analyzer-options paragraph now points `streamType` at `StreamType` (and `locale` at `ISO639_1`), and the `NgramAnalyzer` example uses `StreamType::UTF8`.

- **Federated search ŌĆö per-type permission registry (foundation).** The `FederatedSearch` `requires` registry now accepts, besides the **collection-level** form (`'customers' => 'customers:list'` or an OR-list `'users' => [ 'users:list', 'users:admin' ]`, both unchanged), a **structured** form for a **polymorphic** collection ŌĆö the future per-type permission cascade. A structured value gates the collection first (level 1) then each type (level 2): `'organizations' => [ FederatedSearchParam::COLLECTION => 'org:list', FederatedSearchParam::MAP => [ '<type>' => 'cust:list', ŌĆ” ], FederatedSearchParam::FALLBACK => 'org:list' ]`. `COLLECTION` is the level-1 collection gate (a subject string or an OR-list, absent = the collection itself is public); `MAP` reuses the composite `models` vocabulary as `type => subjects` (a subject string or an OR-list, declaration order kept); `FALLBACK` governs the **unlisted** types ŌĆö absent/`null` = hidden (fail-closed default), subject(s) = required, or the literal `true` = visible (the collection gate suffices). The registry is normalised on construction (level-1 / map subjects / fallback cleaned of empty or non-string entries; a structured entry that gates nothing is dropped). The discriminator field is **reused** from the collection's composite `models` entry ŌĆö never redeclared here. New `FederatedSearchParam::COLLECTION` constant. **Strictly backward-compatible**: the string and OR-list collection-level forms parse exactly as before. The registry is **enforced** by the entry below.
  - **Tests:** unit (`FederatedSearchTest` ŌĆö a full structured cascade kept; the collection / fallback defaulting to null; the map and an OR-list collection cleaned of empty / non-string entries; `FALLBACK => true` kept with an absent map; an OR-list fallback kept and `false` cleaned to null; a structured entry that gates nothing dropped; the collection-level string / OR-list forms still kept verbatim).

- **Federated search ŌĆö the per-type permission cascade is enforced at the SEARCH.** `FederatedSearch` now applies the structured `requires` registry as a **two-level cascade**: level 1 (collection) keeps the existing `SEARCH ŌĆ” OPTIONS { collections }` gate; level 2 (per type) ANDs a discriminator predicate onto the search for every authorized polymorphic collection ŌĆö both **before the `LIMIT`**, so the page and `fullCount` stay exact. The predicate scopes itself to the polymorphic collection by **field absence** (a non-polymorphic collection does not index the discriminator), replacing `IS_SAME_COLLECTION` ŌĆö which ArangoSearch rejects inside `SEARCH` (`not implemented: ŌĆ” noncompliant expression`). Two shapes, the type value matched under the `identity` analyzer: **permissive** (unlisted types visible ŌĆö `FALLBACK => true` or a granted fallback) hides only the denied types, `!ANALYZER(doc.<disc> IN @denied,"identity")`; **strict** (unlisted types hidden ŌĆö the default) keeps only the allowed types plus any document with no discriminator, `( ANALYZER(doc.<disc> IN @allowed,"identity") || !EXISTS(doc.<disc>) )`. When a type gate is present the engine takes over the `ANALYZER` wrapping (the term keeps its own analyzer, the type predicate runs under `identity`). A **defensive** copy of the same level-2 policy runs at `rebuild()`, so a denied type never reaches a model even when `rebuild()` is called on its own. The library gate is **shape-agnostic** (`doc.<disc> IN (ŌĆ”)` / `EXISTS` are identical whether the discriminator is a string or an array). **Prerequisite**: the polymorphic collection's inverted index must index the discriminator with the `identity` analyzer ŌĆö `additionalType` when it is a **string**, `additionalType[*]` (array expansion) when it is an **array** (a field is consistently one shape per collection); no `storeValues` is needed (inverted indexes answer `EXISTS` natively). **Strict-mode typeless behaviour** follows the index shape: on an `additionalType[*]` index a document with no discriminator value is **hidden** (fail-closed); on a plain `additionalType` index it stays visible. **Strictly backward-compatible**: with no structured requirement the search is byte-for-byte unchanged.
  - **Tests:** unit (`FederatedSearchTest` ŌĆö the permissive gate binding only the denied type and emitting no `EXISTS`; the strict gate emitting the allowed-set `IN` plus the `!EXISTS` field-absence escape; the strict all-denied case collapsing to `!EXISTS`; the no-denied case adding no predicate at all; the level-1 collection gate excluding a whole collection before any type gate; a structured requirement on a non-composite model adding no gate; the defensive rebuild gate dropping a denied type and keeping an unlisted one) + **live integration** (`SearchAliasViewIntegrationTest` ŌĆö on a real `search-alias` server: permissive hides only the denied type and keeps an unlisted/typeless one; strict keeps the allowed types and a multi-typed document carrying one, and hides the denied and the typeless; a **string**-discriminator collection filtered by a plain `additionalType` index ŌĆö all with an exact `foundRows()` proving the filter ran before the `LIMIT`).

- **Conditional field projection ŌĆö `Field::WHEN` / `Field::ELSE`.** A scalar field can now guard its value behind a condition resolved **at query time** from the document's own attributes, rendered as an AQL ternary `<condition> ? <then> : <else>` (the SQL `CASE WHEN ŌĆ” THEN ŌĆ” ELSE ŌĆ”` shape). The key is always present ŌĆö only the value switches; an absent value is `null`, never a removed key (no `MERGE`, intentionally out of scope). The condition mirrors the recursive `?filter=` grammar ŌĆö a leaf (`'active'` truthiness, `[ 'visibility', 'public' ]` equality, `[ 'stock', 'gt', 0 ]` comparator, or the associative `FilterParam` `key`/`op`/`val`/`alt` form) combined with AND/OR/NOT groups and arbitrary nesting ŌĆö but compiles **inline** (no bind variables, since the projection layer carries none): values are developer-declared literals escaped by `aqlValue()` (a value that looks like `doc.<attr>` is kept raw, enabling attribute-vs-attribute comparison), and attribute names are validated by `assertAttributeName()`. - **"Understanding ArangoSearch" wiki primer.** A new `wiki/{fr,en}/getting-started/arangosearch.md` ŌĆö the pedagogical entry point of the search stack, closing the Lot S documentation: why `LIKE` hits four ceilings (relevance, tolerance, words, index), the four building blocks (Analyzer ŌåÆ View ŌåÆ `SEARCH` ŌåÆ scorers) with examples, a "what becomes possible" table, how a View lives server-side (eventual consistency, initial indexing, cost, dump/restore), the library layer map (function helpers ŌåÆ `aqlSearch()` ŌåÆ `aqlScoredSearch()` ŌåÆ the model `AQL::VIEW` block ŌåÆ the clients), good practices and pitfalls (the `==`-vs-`TOKENS` stemming trap, Analyzer features, post-`SEARCH` `FILTER`, declaration drift), and the planned evolutions. - **Facet counts now follow the View search (Lot S4b).** When a model's View search is active (`AQL::VIEW` block + `?search=` term), every `?facetCounts=` counting sub-query iterates the View with the same `SEARCH` segment as the list (and the search leaves the shared `FILTER`), closing the interim limitation of S4a: the list, `count()` and the facet buckets now always agree on the matched set ŌĆö including stemming, boosts and fuzzy matches the previous `LIKE` restriction could not see. The search expression and its binds are evaluated once and shared; relevance scoring is irrelevant for counting (`COLLECT` ignores order), so the sub-queries carry no scorer. Without an active View search the emitted facet-counts query is byte-for-byte unchanged. Covered by `ViewSearchTest` facet-counts cases + live `ViewSearchIntegrationTest::testFacetCountsFollowTheViewSearch` (also proving the View `SEARCH` is accepted inside `LET` sub-queries). - **Model-level ArangoSearch View search (Lot S4a).** A `Documents` model can now declare an ArangoSearch View through the new `AQL::VIEW` block (keys in the new `models/enums/Search` class): `Search::NAME` (required), `Search::ANALYZER` (default `identity`), `Search::FIELDS` (`field => boost` map, numeric shorthand or `[ Search::BOOST => n ]`, dotted paths, falls back to `AQL::SEARCHABLE` with boost 1), `Search::PHRASE` (exact-phrase bonus, `boost ├Ś 2`) and `Search::FUZZY` (Levenshtein distance, `0` = off).
  - **Behavior:** when the block is present and `?search=` carries a term, the list query switches automatically from the `LIKE` sweep to an index-accelerated, relevance-ranked `SEARCH` against the View ŌĆö `FOR doc IN @@view SEARCH ANALYZER(BOOST(doc.f IN TOKENS(@search_N, "az"), w) OR BOOST(PHRASE(doc.f, @search_N), w├Ś2) OR LEVENSHTEIN_MATCH(doc.f, @search_N, d) ŌĆ”, "az")` ŌĆö keeping the `?search=` contract (comma-separated terms, OR everywhere, bound values) and the whole pipeline (`?filter`/`?facets`/conditions stay as a post-`SEARCH` `FILTER`, pagination/skins/projections untouched). The base expression uses `TOKENS()` so both sides are analyzed. Without the block, the emitted AQL is byte-for-byte unchanged.
  - **Score & count:** an active search exposes the synthetic `score` sort key (`Search::SCORE`, mirroring `?near=`/`distance`): default `score DESC`, composable (`?sort=-score,name`), dropped without an active search. `count()` follows the same branch as `list()`; `?facetCounts=` keeps the `LIKE` restriction for now (lifted by S4b).
  - **Provisioning:** the View is lazily provisioned at model initialization (mirroring `initializeCollection()`/`AQL::INDEXES`): created from the declaration when missing, never altered when it exists (no automatic drift resync ŌĆö lifted by M1). New fa├¦ade trait `db/traits/ViewManagementTrait` (`viewCreate`/`viewDrop`/`viewExists`) used by `ArangoDB`; new `BindTrait::bindView()`; new `AQL::VIEW` key.

- **New `aqlScoredSearch()` operation (`db/operations/`).** Builds a complete, relevance-ranked search query over an ArangoSearch View in the canonical form `FOR ŌĆ” SEARCH ŌĆ” [OPTIONS { ŌĆ” }] LET score = BM25(ŌĆ”)|TFIDF(ŌĆ”) SORT score DESC LIMIT ŌĆ” RETURN ŌĆ”` ŌĆö the ArangoSearch counterpart of `aqlVectorSearch()`. Explicit named arguments: `view`, `search`, `limit`, plus optional `analyzer` (wraps in `ANALYZER(expr, "name")`), `options` (the `SEARCH ŌĆ” OPTIONS { ŌĆ” }` object), `scorer` (`SearchScorer::BM25` default / `TFIDF`), the tunings `k` / `b` (BM25) and `normalize` (TFIDF), `offset`, and `docRef` / `scoreRef` / `return` overrides. - **Completed the `aqlSearch()` operation.** The `SEARCH` clause now supports an Analyzer wrap and the `OPTIONS` object. - HTTP facets (`?facets=`) overhaul ŌĆö every facetable field is declared under `Arango::FACETS` and reuses the filter operator vocabulary (`FilterComparator` / `FilterArrayComparator` + `FilterParam::OP/VAL`), with new and generalised types:
  - `Facet::IN` ŌĆö array-membership facet (op-driven, default `any.in`); the former `LIST`, `LIST_FIELD`, `LIST_FIELD_SORTED` are kept as aliases.
  - `Facet::JOIN` ŌĆö simple key-join existential facet (the join counterpart of `Facet::EDGE`): keeps documents having at least one joined document whose field matches the requested value, with multi-value OR and `-` negation.
  - `Facet::JOIN_COMPLEX` ŌĆö key-join existential facet matching SEVERAL fields (AND) on the same joined document (object `{field: condition}`), sharing the new `HasFacetComplexConditions` builder with `EDGE_COMPLEX`.
  - `Facet::EDGE` generalised ŌĆö configurable operator (`Facet::OP`) and multi-field OR (`AQL::FIELDS`), multi-value CSV (OR) and `-` negation as exclusion (`LENGTH(ŌĆ”) == 0`). The multi-field `like` configuration replaces the former THESAURUS facet.
  - The URL-provided sub-field names of complex facets (interpolated into `doc.<field>`) are validated by the new `isAttributeName` / `assertAttributeName` helpers ŌĆö a dangerous name fails the facet (logged and skipped), so no untrusted fragment ever reaches the AQL.
- `alt` symmetric transformations on both `?filter=` and `?facets=` ŌĆö `alt` now wraps the comparison **value** (right side), not only the **field** (left). The object form `alt:{ key:<chain>, val:<chain> }` gives one chain per side, and `val:true` mirrors the key-side chain (e.g. `LOWER(doc.email) == LOWER(@v)`); an array value maps the chain over each element (`@v[* RETURN LOWER(CURRENT)]`). On `?filter=` it covers the flat filters (string/number/array/date) and the nested array-expansion filters (`field[*].sub` and `match` ŌĆö `LOWER(CURRENT.email) == LOWER(@v)`, with `alt` applied globally to every `match` sub-field); edge/join/document leaves inherit it through the underlying flat filter, while the structural join key stays raw. On facets the chain may come from the model definition (new `Facet::ALT`, a default for every request) or from the URL request object (`{op,val,alt}`, which takes precedence). It is wired on `FIELD`, `EDGE`, `JOIN` and `IN` (membership ŌĆö where the array field is projected element-wise, `doc.tags[* RETURN LOWER(CURRENT)]`); complex facets (`EDGE_COMPLEX` / `JOIN_COMPLEX` / `ARRAY_COMPLEX`) accept a global `Facet::ALT` applied to every sub-field. The legacy field-only string/list forms (`"lower"`, `["trim","lower"]`) are unchanged. Backed by the shared free helpers `oihana\arango\db\helpers\alterExpression()` / `resolveAltSides()`, called directly by the filter and facet builders; documented with per-case examples in `wiki/{fr,en}/db/{filter,facets}.md`.
- New `AT LEAST (n)` array quantifier for `?filter=`: declared with the array-form operator `["atLeast.<cmp>", n]` (element 0 the `atLeast.<cmp>` code, element 1 the threshold, defaulting to 1), it keeps documents where at least `n` array elements satisfy the comparison ŌĆö e.g. `{"op":["atLeast.ge",2],"val":80}` ŌåÆ `doc.scores AT LEAST (2) >= @v`. The `<cmp>` suffix reuses the filter operators (eq/ne/gt/ge/lt/le/in/nin); the threshold is cast to an int (injection-safe) and the field stays `alt`-aware. New `FilterArrayComparator::AT_LEAST*` constants.
- New `between` (range) operator for `?filter=` (number/string/date) and the `?facets=` FIELD facet: an inclusive `key >= @min && key <= @max` driven by `min`/`max` keys. Bound omission is type-driven ŌĆö number/string drop the omitted side (one-sided range), while a date resolves an omitted bound to "now" (always two-sided); a date `tz` applies to both bounds (`DATE_LOCALTOUTC(@min,@tz)`). The compared field stays `alt`-aware. Backed by the shared, unit-tested `oihana\arango\db\helpers\buildBetweenClauses()` helper.
- New `Field::ALTERS` output-side transformation on the field projection: an `alt`-style chain (same `FilterFunction` vocabulary ŌĆö `'lower'`, `['trim','lower']`, `['substring',0,3]`) wrapping the projected value at `RETURN` time, e.g. `'name' => [ Field::ALTERS => ['trim','lower'] ]` ŌåÆ `name: LOWER(TRIM(doc.name))`. The filter-side `alt` counterpart for the output, it homogenizes filters Ōåö projection. Opt-in per field, applied only to the default scalar projection (`key: doc.key`) ŌĆö ignored on typed/structural filters (BOOL, DATETIME, EDGE, JOIN, MAP, ŌĆ”); `Field::NAME` still selects the source attribute so a transformed value can be exposed under another key. Documented with worked examples in `wiki/{fr,en}/edges-joins-projection.md`.
- New aggregate facets `Facet::EDGE_AGGREGATE` and `Facet::JOIN_AGGREGATE` for `?facets=`: instead of the existential `LENGTH(ŌĆ”) > 0` of `EDGE`/`JOIN`, they aggregate a numeric field over all linked documents and compare it to a threshold ŌĆö `AGG(FOR doc_x IN <source> [FILTER <join>] RETURN doc_x.<field>) <op> @threshold`. Driven by the request object `{agg, field, op, val}` (a bare scalar = threshold), each piece overridable on the URL and falling back to the definition (`Facet::AGG`, `AQL::FIELDS`, `Facet::OP`). The aggregator (`avg`, `sum`, `min`, `max`, `count`) is whitelisted by the new `FacetAggregator` enum (unknown ŌćÆ facet skipped + logged); `count` is field-less (`RETURN 1`) and generalizes the existential. A URL-provided `field` is validated with `assertAttributeName`, the threshold is numeric-cast, and the comparison is guarded by `LENGTH(ŌĆ”) > 0` so a `lt`/`le` threshold never lets the AQL empty-set sentinel (`null` for avg/min/max, `0` for sum/count) match documents with no linked document. New keys `Facet::AGG`, `FilterParam::AGG`, `FilterParam::FIELD`; shared `HasFacetAggregateConditions` builder; documented with worked examples in `wiki/{fr,en}/db/facets.md`.
- New conditional `alt` functions `coalesce` / `notNull`: wrap the field in AQL `NOT_NULL(...)` to substitute a default when it is missing or null before comparison ŌĆö e.g. `alt:[["coalesce",0]]` ŌåÆ `NOT_NULL(doc.discount, 0) == @v`. The default value(s) come from the URL and are inlined as strict AQL literals via `json_encode` (always quoted/escaped, never the raw passthrough of `aqlValue()`), so a default is always treated as literal data ŌĆö injection-safe.
- New `pluck` `alt` function (and the `oihana\arango\db\functions\arrays\pluck()` helper): projects an array of objects onto a single sub-field via AQL's inline projection `array[* RETURN CURRENT.<field>]`. Chained with an aggregate it reduces over a property of embedded objects ŌĆö e.g. `alt:[["pluck","price"],"avg"]` ŌåÆ `AVERAGE(doc.items[* RETURN CURRENT.price])`. The sub-field name comes from the URL and is validated with `assertAttributeName` before interpolation (injection-safe; an unsafe name throws `ValidationException`).
- New AQL array-expansion helpers `oihana\arango\db\functions\arrays\arrayMap()` (`array[* RETURN expr]`) and `arrayFilter()` (`array[* FILTER cond]`), centralizing the previously hand-built `[* ŌĆ” ]` string literals. The five existing sites now build through them ŌĆö `pluck()` (a thin `arrayMap` wrapper over a validated sub-field), the array-valued `alt` projection in `FilterTrait`, the `Facet::IN` field-side projection in `HasFacetIn`, and the two array-expansion existentials in `HasFilterArray` (`LENGTH(arrayFilter(...)) > 0`). Pure refactor ŌĆö the emitted AQL is unchanged ŌĆö plus direct unit tests for both helpers.
- New AQL array "question mark" helper `oihana\arango\db\functions\arrays\arrayContains()` (`array[? <quantifier> FILTER cond]`): the boolean array-contains operator, the direct/idiomatic form of `LENGTH(array[* FILTER cond]) > 0`, with optional `ANY` / `ALL` / `NONE` / `AT LEAST (n)` quantifiers (omitted = at least one). Provided as a primitive (with direct unit tests) ŌĆö notably to express quantified conditions over arrays of objects (e.g. `doc.reviews[? AT LEAST (3) FILTER CURRENT.rating >= @v]`); the existing `LENGTH(...) > 0` existentials are left unchanged.
- Opt-in live ArangoDB integration suite (`#[Group('integration')]`, run via `vendor/bin/phpunit -c phpunit-integration.xml`): builds a disposable database from `configs/config.toml`, seeds it, executes the generated AQL for real, then drops it ŌĆö and skips cleanly (never fails) when no instance is reachable. Covers the facet builders, the `alt` transformations, and the embedded array operations (`DocumentsArrayTrait`: set dedup, sorted-set ordering, empty `[]`, move, object deep-equality, `[]` seeding, `arrayPurgeRef`) end-to-end, complementing the server-free unit suite that only freezes the AQL string.

### Changed

- `Facet::FIELD` is now operator-enabled (`FilterComparator`): it accepts an `op` (in the definition via `Facet::OP`, or per request via an `{op, val}` object), defaulting to `match` (`=~`) for backward compatibility, with generic negation. Numeric scalars keep their type, so `doc.price >= 100` never binds a string.
- The list facet family is unified under `Facet::IN` (op-driven membership); `Facet::LIST`, `LIST_FIELD` and `LIST_FIELD_SORTED` become thin aliases, and the broken `length` branch was removed. The facet dispatcher was hardened (nullable type lookup, null-safe logger).

### Removed

- `ListItemTrait` / `MultiFieldTrait` (`models/traits/`) ŌĆö orphaned, untested legacy traits composed by `Documents` but never called, and generating invalid AQL (a bind parameter used as an object attribute name; a hardcoded `doc.hasPart` filter). Replaced by `DocumentsArrayTrait` (see Added): `deleteListItem`/`insertInMultiField`/ŌĆ” map to `arrayRemove`/`arrayInsert`/ŌĆ”, `deleteReverseInMultiField` ŌåÆ `arrayPurgeRef`, and `updateDateParentMultiField` is dropped in favour of an application-level `afterUpdate` listener (documented in `wiki/{fr,en}/db/arrays.md`).
- `PositionTrait` (`models/traits/aql/`) ŌĆö dead, half-written trait: it was composed by no model, `preparePosition()` was never called, and its only AQL form had a hardcoded `@value` placeholder (no `$binds`/value parameter) plus an unfiltered `FOR doc_coll IN <collection>` scan. Removed the trait and its characterization test rather than freeze unused, broken output; a position-ranking feature, if ever needed, should be designed fresh with proper binding.
- `Facet::THESAURUS` ŌĆö it was exactly an `EDGE` facet with `AQL::FIELDS = '_key,name,alternateName'` and `op = like`, so it is now expressed as that configuration rather than a dedicated type.
- Magic `id ŌåÆ _key` rewriting in facets ŌĆö key/property aliases are now declared explicitly via `Facet::PROPERTY` everywhere, with no hidden remapping.

### Fixed

- `aqlCollect()` could emit invalid AQL combining `AGGREGATE` and `WITH COUNT INTO`, which are mutually exclusive: when both keys were supplied it appended both clauses. `AGGREGATE` now takes precedence and `WITH COUNT INTO` is dropped (to count alongside aggregates, express the count as a `LENGTH(1)` aggregate). The only caller (`EdgesCountTrait`, which passes `WITH COUNT` alone) is unaffected. The companion `aqlCollectReturn()` mirrors the same rule.
- `aqlUpsertExpression()` never rejected an `init` that defined **both** `FILTER` and `SEARCH`: the second guard repeated the first one's condition (`!isset($filter) && !isset($search)`) instead of detecting the both-defined case, so it was dead code and a conflicting init silently fell through to `$search ?? $filter` (SEARCH wins). The guard is now `isset($filter) && isset($search)` and throws `FILTER and SEARCH cannot be defined at the same time.`, matching the documented exclusive `UPSERT [ searchExpression | FILTER filterExpression ]` syntax. The two guards are split into independent `if` clauses. Calls passing exactly one of the two (every existing usage) are unchanged. Characterized in the new `AqlUpsertExpressionTest`.
- `removeValue()` array helper emitted the plural `REMOVE_VALUES` instead of `REMOVE_VALUE`, so the single-value `FilterFunction::REMOVE` (`remove`) silently no-op'd on a scalar (its second argument must be an array) and raised an AQL error when an occurrence limit was passed; it now emits `REMOVE_VALUE`, and the optional limit works as documented. The `$limit` parameter also accepts `int|string|null` and ignores non-numeric/non-positive values instead of raising a `TypeError` at call time (e.g. a `remove` filter built from a query string). `FilterFunction::REMOVES` (`removes`) keeps using `REMOVE_VALUES`.
- `ArangoCommand` was uninstantiable whenever its `init` array omitted the `directory` key: the constructor's fallback `$this->directory = $init[ŌĆ”] ?? $this->directory` read the `$directory` property *before initialization* (declared `public ?string $directory ;` with no default in `ArangoDumpAction` / `ArangoRestoreAction` / `ArangoListDumpsAction`), throwing `Error: Typed property ŌĆ” must not be accessed before initialization`. The triplicated declaration is extracted into a single `oihana\arango\commands\traits\DirectoryTrait` (property defaulted to `null` + an `initializeDirectory()` initializer matching the `initializeEncrypt()` / `initializePassphrase()` pattern), the three actions now `use DirectoryTrait`, and the constructor calls `$this->initializeDirectory( $init )`. Behavior for configs that already supplied a directory is unchanged.
- `ArangoDB` fa├¦ade hardening: `getCursor()` was typed `: Cursor` (non-nullable) but `$cursor` is null before the first `execute()` and after `streamDocuments()` resets it, so calling it in those states threw a `TypeError` ŌĆö it now returns `?Cursor`. `getExtra()` / `getFoundRows()` likewise dereferenced a possibly-null cursor and now return `[]` / `0` when no query has run. The constructor caught only `ArangoException`, but the client builder throws `InvalidArgumentException` for an invalid config (e.g. empty endpoint), so build failures were never logged; the `catch` is widened to `Throwable` (logged, then rethrown unchanged).
- `CollectionManagementTrait` (`ArangoDB` fa├¦ade) called an **undefined** `toLegacyException()` in the `catch` blocks of `collectionRename()`, `getIndex()` and `getIndexes()` ŌĆö a leftover shim from the migration off the official ArangoDB-PHP fork. On any client error those three methods crashed with `Error: Call to undefined method` instead of surfacing the failure. The dead `try/catch` wrappers are removed so the `oihana\arango\clients\exceptions\ArangoException` propagates directly, and the redundant `ArangoException as ArangoClientException` import alias (both names resolved to the same class) is dropped. The boolean / log-and-swallow methods are unchanged.
- `PropertyController` PATCH in `raw` mode (`init[Arango::RAW] = true`) returned `null` instead of the submitted property value: the handler read the prepared payload with object-property syntax (`$payload->{$this->property}`), but `PayloadsTrait::propertyPayload()` yields an **array** (`[ property => value ]`), so the access always resolved to `null`. It now reads the value with `oihana\core\accessors\getKeyValue( (array) $payload , $this->property )`, which handles arrays and objects uniformly. Non-`raw` mode (which reloads via `model->get()`) is unchanged. Characterized then fixed in `PropertyControllerTest`.
- `aqlFields` `Field::QUOTED`: it double-quoted the key used for BOTH the output label and the attribute access, emitting invalid AQL like `"my-key":doc."my-key"` (an attribute with special characters cannot be reached with double quotes ŌĆö confirmed rejected by ArangoDB with a syntax error). The attribute access now uses backticks (`"my-key":doc.`my-key``), the valid form; only the output label is double-quoted. Cases pairing `Field::QUOTED` with `Field::NAME` or `Field::ALTERS` are unchanged. Characterized then fixed in `AqlFieldsTest`.
- AQL `like()` helper: its third parameter was named `$caseSensitive` but maps to ArangoDB's `LIKE(text, search, caseInsensitive)` third argument, so it did the opposite of its name (`true` produced a case-INSENSITIVE match). Renamed to `$caseInsensitive` (default `false` = case-sensitive, matching AQL). The emitted AQL is unchanged for every call path ŌĆö `?search` stays case-insensitive and the `like` filter-function alter keeps its output ŌĆö only the (now honest) parameter name and the docs change. Added the previously missing direct unit test, and fixed the stale `SearchTraitTest` comment that described the predicates as "case-sensitive".
- `command:arangodb` document `upsert` now honors the `removeKeys` exclusion list: it was passed under the `excludes` key, which the upsert query never reads, so the exclusion silently had no effect.
- `ArangoCommand` registered the wrong signals for event dispatch: it passed `(int) [SIGINT, ŌĆ”]` (which casts to `1`) to the variadic `setSignalsToDispatchEvent()`. The intended signal list is now passed.
- Removed the unused `ignoreCollection` arangodump option (`ArangoDumpOptions::$ignoreCollection` / `ArangoDumpOption::IGNORE_COLLECTION`): no such flag exists in `arangodump`, and exclusion is resolved client-side as a `--collection` complement. The `--ignore-collection` CLI flag is unchanged.
- `ArangoConfigTrait` now validates the connection settings with exceptions instead of `assert()` (which is compiled out when `zend.assertions=-1`), so a missing database/endpoint/username never silently flows on as null. Properties are initialized; an empty password remains valid.
- Declare the previously-undeclared `ext-pcntl` requirement in `composer.json` ŌĆö `ArangoCommand`'s signal handling (`getSubscribedSignals()` / `handleSignal()` / `setSignalsToDispatchEvent()`) relies on the `SIGINT` / `SIGTERM` / `SIGUSR1` / `SIGUSR2` / `SIGALRM` constants provided by the extension.
- Security: `command:arangodb dump` / `restore` no longer run `arangodump` / `arangorestore` through a shell. The command was assembled as a single string (option values quoted only with `json_encode`) and passed to `system()`, so a value containing shell metacharacters (password, database, endpoint, directory, collection name) could break the command or inject arbitrary shell. The processes are now executed via `proc_open()` with an explicit argument vector (no shell), so values are passed verbatim. New `ArangoProcessTrait` (`optionsToArguments()` + `runProcess()`), covered by `ArangoProcessTraitTest` including a metacharacter-stays-one-argument unit test and a no-shell probe.
- `command:arangodb restore --last` could never locate the archive: it set the input file to the bare filename returned by `findFiles()` (whose `FILTER` yields `$file->getFilename()`), so `assertFile()` failed unless the current working directory happened to be the dump directory. The `--last` branch now prepends the dump directory, matching the interactive branch.

- `command:arangodb restore --date <date>` could never locate the archive. The by-date suffix was built as `Char::DASH . $database . $shouldEncrypt ? FileExtension::TAR_GZ_ENCRYPTED : FileExtension::TAR`, but `.` binds tighter than the ternary, so PHP evaluated `(Char::DASH . $database . $shouldEncrypt)` as the condition ŌĆö always a non-empty (truthy) string. The suffix therefore always collapsed to `.tar.gz.enc` and silently dropped the `-{database}` segment, so a non-encrypted restore-by-date never matched its `{date}-{database}.tar.gz` file. The non-encrypted branch also used `FileExtension::TAR` (`.tar`) instead of `FileExtension::TAR_GZ` (`.tar.gz`), which the dump action actually produces. The suffix logic is now isolated in the testable `ArangoRestoreAction::getArchiveFileSuffix()` helper and covered by `ArangoRestoreActionTest`.

- `Facet::ARRAY_COMPLEX` was broken at query time: it iterated an undefined variable (`FOR doc_workshops IN resultworkshops`, with no separator) and built its filter on the out-of-scope `result` document, so the facet never ran. It now iterates the in-scope embedded array (`doc.<key>`) and returns `1`. Confirmed against a live ArangoDB.
- `Facet::EDGE_COMPLEX` used a malformed loop variable (`AQL::DOC_REF . $key` ŌåÆ e.g. `docRefnumbers`) instead of the sibling convention `doc_<key>`; it now reads `doc_<key>` like the other builders, and gained per-field multi-value (OR) plus per-value inline negation (`!=`) on the same traversed vertex.
- Date filters with a valid `tz` always threw: the timezone-conversion path passed the bound placeholder (`@tz`) to `dateLocalToUTC()`, whose own timezone re-validation then rejected it (the literal timezone is already validated upstream by `isValidTimezone()`). The re-validation is now disabled there (`$throwable = false`), so `?filter={...,"tz":"Europe/Paris"}` emits `DATE_LOCALTOUTC(@v,@tz)` instead of crashing ŌĆö and the new date `between` honors `tz` on both bounds.

### Security

- `?sort=` in open mode (no `sortable` whitelist) interpolated the raw URL key into `doc.<key>` without validation ŌĆö an AQL injection vector (a model that exposes `?sort=` without configuring `$sortable`, the default). `SortTrait::prepareSort()` now validates each open-mode sort key with `assertAttributeName()`; the whitelisted path was already safe. Legitimate dotted paths (`address.city`) still sort; an unsafe key throws `ValidationException`.
- `aqlFields()` (the projection builder) interpolated the source attribute (`Field::NAME ?? key`) into `doc.<attr>` without validation. Although projection fields are server-config today (`?fields=` is not wired to any controller), the bare source is now guarded by `assertAttributeName()` as defense-in-depth, so a future URL-driven projection cannot inject AQL. The two legitimate non-identifier paths are preserved: the `Field::QUOTED` escape hatch (a special key reached via backticks, `doc.`my-key``) and numeric keys.
- HTTP facets enforce a strict trust contract: only bound values (`@bind`) are user-controlled. Operators are whitelisted (`getAlias`, falling back to the type default), facet keys are whitelisted against the model's `Arango::FACETS` (an unknown key is ignored), and the URL-provided sub-field names of complex facets are validated by `assertAttributeName` before being interpolated ŌĆö a dangerous name fails the facet (logged and skipped). `alt` function names are likewise whitelisted (an unknown function is a no-op), so no `?filter=` / `?facets=` input can inject AQL.
