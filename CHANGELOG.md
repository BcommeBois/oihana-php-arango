## [Unreleased]

### Added

- Structure doctor — the global declarations ↔ server health check (Lot M2), extending the Lot M1 View diff & sync to the whole declared structure. The motivating bug: the lazy provisioning only creates the `AQL::INDEXES` **when the collection itself is created**, so an index added to an existing model is silently never created on any environment (queries fall back to full scans with no error) — and the model did not even retain its index declaration after construction. Now: `ArangoTrait` keeps the declaration in the new public `$indexes` property, and three layers expose the conformity check. *Façade*: `collectionDiff( $name , $type )` on `CollectionManagementTrait` (existence + document/edge type — a drifted type is never repaired, that is a migration), and the whole index surface extracted to the new **`IndexManagementTrait`** (composing the collection trait): the legacy `createIndex` / `dropIndex` / `getIndex` / `getIndexes` — `createIndex` / `dropIndex` now typed `string|Collection` instead of `mixed`, like the shared `resolveCollectionName()` (promoted private → protected) — plus the new `indexesDiff( $collection , $indexes )` (one aggregated report: declared indexes matched by `name` or by `type`+`fields` signature, field-by-field subset-aware comparison with **order-sensitive** `fields`, the creation-only `inBackground` key excluded, the automatic `primary`/`edge` indexes out of scope, undeclared server indexes reported) and `indexesSync( $collection , $indexes , $force )` (missing indexes created; drifted ones rebuilt by drop + recreate **only when forced** — an index is immutable, the rebuild opens a window where queries lose it and a unique index may fail over duplicated data). *Model* (new `DoctorTrait`, composed by `Documents`, inherited by `Edges`): `diagnose()` returns one `DiffReport` per declared object (collection → indexes → View, the View through the Lot M1 `viewDiff()` with its coherence checks) and `repair( force )` reconciles (missing collection created with its type and indexes, missing indexes created, View synchronized). *Command*: new **`doctor` action** on `command:arangodb` — report by default with the orphan footnote (collections and Views on the server that no configured model declares — never failing, never touched), `--apply` to repair, `--force` for the index rebuilds, `--prune` for the interactive orphan removal; the exit code is a health check (fails on anything missing/drifted/invalid/unreachable in report mode, on anything unrepaired in apply mode — CI-friendly); same `ArangoCommandParam::MODELS` wiring and container lazy kill-switch as `views`. The Lot M1 report classes were generalized for the whole family: `ViewDiffReport` → **`DiffReport`** (with the new `kind` property and the `DiffKind` constants `collection`/`indexes`/`view`) and `ViewDiffStatus` → **`DiffStatus`**; the shared `models` property moved to the new `ArangoModelsTrait`, and the new `ArangoClientTrait::resolveDatabase( $input )` deduplicates the CLI-option/configuration connection resolution of the `collections` / `views` / `doctor` actions. New `arango:doctor` composer script. Covered by unit tests (`CollectionManagementTraitTest`, `DoctorTraitTest`, `ArangoDoctorActionTest`, `DiffReportTest` — 100% on the new code) and a live integration suite (`DoctorIntegrationTest`: the motivating lifecycle — baseline green, an added index provably absent, `diagnose()` MISSING with the exact line, `repair()` creates it, a flipped `unique` announced then rebuilt only with force, and the one-pass recreation of a dropped collection with its indexes). Documented in `wiki/{fr,en}/commands/arangodb.md` (new `doctor` section: examples, checks table, exit-code semantics, wiring).

- View diff & sync — closing the declaration-drift reservation of the model-level View search (Lot M1). The model provisioning is create-if-missing, so editing an `AQL::VIEW` block used to leave the server View silently out of date (an added field not indexed, a changed Analyzer matching almost nothing). The declaration ↔ server gap can now be **inspected and repaired explicitly**, at three levels. *Façade* (`ViewManagementTrait`): `viewDiff( $name , $links )` compares the desired per-collection link map with the server `properties()` — declaration-oriented and subset-aware (server defaults the declaration does not mention are ignored; `analyzers` lists compare order-insensitively; fields indexed on the server but no longer declared are reported as drift) — and `viewSync( $name , $links )` creates a missing View or repairs a drifted one through **`updateProperties()`** (verified live: PATCH replaces each declared collection link wholesale — removed fields really unindex — while the View options and the links of other collections stay untouched, and the View remains queryable during the background re-index; `replaceProperties()` would reset undeclared options). Both return the new typed **`DiffReport`** (`db/results/`: `name`, `status`, human-readable `changes` lines, `applied`) with the new **`DiffStatus`** constants (`IN_SYNC` / `MISSING` / `DRIFTED` / `INVALID` / `UNREACHABLE`). *Model* (`SearchTrait`): no-argument `viewDiff()` / `viewSync()` use the model's own declaration (new public `getViewName()` / `getViewLinks()`), and add coherence checks on top — malformed declaration (no `Search::NAME`, no searched field, no collection), Analyzer or collection unknown to the server, same-name entity of another type — all resolving to `INVALID`, never created nor synchronized. New façade trait `AnalyzerManagementTrait` (`analyzerExists()`, defensive) with the matching `ArangoTrait` delegation. *Command*: new **`views` action** on `command:arangodb` (`ArangoAction::VIEWS`) — default lists the Views (name, type, linked collections), `--diff` reports every configured model (plus the **orphan views** footnote: on the server, declared by no model — report only), `--sync[=a,b]` applies (created / resynchronized per View), `--drop[=a,b]` removes targeted Views or offers an interactive multi-select (restore-style); the models are supplied as container ids via the new `ArangoCommandParam::MODELS` init key, and the exit code fails when a model is `unreachable` (deployment-script friendly). Covered by unit tests (`ViewManagementTraitTest`, `AnalyzerManagementTraitTest`, `DiffReportTest`, `ViewSearchTest`, `ArangoViewsActionTest` — 100% on the new code) and a live integration suite (`ViewSyncIntegrationTest`: the full lifecycle — lazy provision, declaration widens, the new field is provably unsearchable, `viewDiff()` says DRIFTED with the exact change line, `viewSync()` repairs, the search finds the document, narrowing + sync really unindexes; plus the unknown-Analyzer `INVALID` path). New `arango:views` composer script alias (flags after `--`). Documented in `wiki/{fr,en}/commands/arangodb.md` (new `views` section: CLI examples, report statuses, DI wiring) and the drift notes of `db/search-views.md` / `getting-started/arangosearch.md` now point to the remedy.

- Container-aware lazy provisioning switch — the model lazy mode now resolves through the new `oihana\traits\LazyTrait` (from `oihana/php-system`): `ArangoTrait` dropped its own `$lazy` property and composes the trait, `initializeCollection()` / `initializeView()` read `$this->initializeLazy( $init )->isLazy()`, and the resolution order is **container entry `lazy` > init key `Arango::LAZY` > property default (`true`)**. The container entry is an orchestration kill-switch: tooling can `$container->set( Arango::LAZY , false )` before resolving models to inspect them without provisioning anything as a side effect — exactly what `views --diff` does (and what the future `doctor` will reuse). Host inits are unaffected (same key, same default).

- New "Understanding ArangoSearch" wiki primer (`wiki/{fr,en}/getting-started/arangosearch.md`) — the pedagogical entry point of the whole search stack, closing the Lot S documentation: why `LIKE` hits four ceilings (relevance, tolerance, words, index), the four building blocks (Analyzer → View → `SEARCH` → scorers) with concrete examples, a "what becomes possible" table mapping needs to library features, how a View lives server-side (eventual consistency, initial indexing, cost, dump/restore), the library layer-by-layer map (function helpers → `aqlSearch()` → `aqlScoredSearch()` → the model `AQL::VIEW` block → the clients), good practices and pitfalls (the `==`-vs-`TOKENS` stemming trap, Analyzer features, post-`SEARCH` `FILTER`, declaration drift), and the planned evolutions (per-field Analyzers, `?lang=` field selection, federated multi-collection search, permission-scoped search, View management command). Linked from the main READMEs (getting-started section) and referenced as the primer by `db/search-views.md` and `aql/aql-functions-search.md`.

- Facet counts now follow the View search (Lot S4b) — when a model's View search is active (`AQL::VIEW` block + `?search=` term), every `?facetCounts=` counting sub-query iterates the **View with the same `SEARCH` segment** as the list (and the search leaves the shared `FILTER`), closing the interim limitation of Lot S4a: the list, `count()` and the facet buckets now always agree on the matched set — including stemming, boosts and fuzzy matches that the previous `LIKE` restriction could not see. The search expression and its binds are evaluated once and shared by every dimension; relevance scoring is irrelevant for counting (`COLLECT` ignores order), so the sub-queries carry no scorer. Without an active View search the emitted facet-counts query is byte-for-byte unchanged. Covered by unit tests (`ViewSearchTest` facet-counts cases) and a live integration case (`ViewSearchIntegrationTest::testFacetCountsFollowTheViewSearch`, also proving the View `SEARCH` is accepted inside `LET` sub-queries on a real server). Documented in `wiki/{fr,en}/db/facets.md` and `db/search-views.md`.

- Model-level ArangoSearch View search (Lot S4a) — a `Documents` model can now declare an ArangoSearch View through the new **`AQL::VIEW`** block (keys in the new `models/enums/Search` constants class): `Search::NAME` (the View name, required), `Search::ANALYZER` (default `identity`), `Search::FIELDS` (`field => boost` map, numeric shorthand or `[ Search::BOOST => n ]`, dotted paths supported, falls back to `AQL::SEARCHABLE` with boost 1), `Search::PHRASE` (exact-phrase bonus, `boost × 2`) and `Search::FUZZY` (Levenshtein edit distance, `0` = off). When the block is present and `?search=` carries a term, the list query **switches automatically** from the `LIKE` sweep to an index-accelerated, relevance-ranked `SEARCH` against the View — `FOR doc IN @@view SEARCH ANALYZER(BOOST(doc.f IN TOKENS(@search_N, "az"), w) OR BOOST(PHRASE(doc.f, @search_N), w×2) OR LEVENSHTEIN_MATCH(doc.f, @search_N, d) …, "az")` — keeping the `?search=` contract (comma-separated terms, OR everywhere, **bound** values) and the whole pipeline (`?filter`/`?facets`/conditions stay as a post-`SEARCH` `FILTER`, pagination/skins/projections untouched). The base expression uses `TOKENS()` so both sides are analyzed (a raw `==` comparison would not match stemmed tokens). Without the block, the emitted AQL is byte-for-byte unchanged. An active search exposes the synthetic **`score` sort key** (`Search::SCORE`, mirroring the `?near=`/`distance` precedent): default `score DESC` (overrides `SORT_DEFAULT`), composable (`?sort=-score,name`), dropped without an active search. **`count()` follows the same branch** as `list()` (same View + `SEARCH`), so totals always agree; `?facetCounts=` keeps the `LIKE` restriction for now (correct, not yet View-accelerated). The View is **lazily provisioned** at model initialization (mirroring `initializeCollection()`/`AQL::INDEXES`): created from the declaration when missing — searched fields (dotted paths nested) linked with the declared Analyzer — and never altered when it exists (no automatic drift resync; update manually). New façade trait `db/traits/ViewManagementTrait` (`viewCreate`/`viewDrop`/`viewExists`, defensive like the collection management) used by `ArangoDB`, with `ArangoTrait` delegations; new `BindTrait::bindView()` (collection bind parameters are valid for View names — verified live); new `AQL::VIEW` key. Covered by unit tests (`ViewSearchTest`, `ViewManagementTraitTest`) and a live integration suite (`ViewSearchIntegrationTest`: real provisioning at construction, boosted relevance ranking, `list()`/`count()` agreement, explicit-sort override, fuzzy typo matching, and the `LIKE` fallback without the block). Documented in the new `wiki/{fr,en}/db/search-views.md` recipe page (declaration, generated AQL, URL table, JSON responses) wired into `db/README`, `db/search.md` and `db/search-and-filtering.md`.

- New `aqlScoredSearch()` operation (`db/operations/`) — builds a complete, relevance-ranked search query over an ArangoSearch View in the canonical form `FOR … SEARCH … [OPTIONS { … }] LET score = BM25(…)|TFIDF(…) SORT score DESC LIMIT … RETURN …`, the ArangoSearch counterpart of `aqlVectorSearch()`. Explicit named arguments: `view`, `search` (the raw `SEARCH` expression), `limit`, plus optional `analyzer` (wraps the expression in `ANALYZER(expr, "name")`) and `options` (the `SEARCH … OPTIONS { … }` object) behaving exactly like `AQL::ANALYZER` / `AQL::SEARCH_OPTIONS`, `scorer` (the new `SearchScorer::BM25` default / `SearchScorer::TFIDF` constants), the scorer tunings `k` / `b` (BM25) and `normalize` (TFIDF), `offset` pagination, and the `docRef` / `scoreRef` / `return` overrides. Both scorers rank better matches higher, so the sort is always `DESC` — no direction to get wrong; the score is bound to a `LET` variable so a custom `return` can expose it (e.g. `'MERGE(doc, { score: score })'`). Guards throw `InvalidArgumentException` on an unknown scorer, `k`/`b` with TFIDF, or `normalize` with BM25. Entirely composed from existing building blocks (`aqlFor`/`aqlSearch`, `bm25()`/`tfidf()`, `aqlLet`, `aqlSort`, `aqlLimit`, `aqlReturn`). Covered by unit tests (`AqlScoredSearchTest`) and a live integration case (`SearchIntegrationTest::testAqlScoredSearchBuilder`, asserting the BM25 ranking and the exposed score on a real View). Documented in `wiki/{fr,en}/aql/aql-operations.md` (new *Scored search* section).

- Completed the `aqlSearch()` operation (closing its historical TODO) — the `SEARCH` clause now supports an Analyzer wrap and the `OPTIONS` object. Two new `$init` keys next to `AQL::SEARCH`: `AQL::ANALYZER` (`'analyzer'`) wraps the search expression in `ANALYZER(expr, "name")` via the `analyzer()` helper, and `AQL::SEARCH_OPTIONS` (`'searchOptions'`) emits `SEARCH … OPTIONS { … }` — accepted as an associative array (hydrated into the new `SearchOptions` value-object: unknown keys dropped, null properties omitted), a `SearchOptions` instance, any plain/`JsonSerializable` object, or a pre-encoded JSON string, with the same tolerance as `aqlOptions()`. `SearchOptions` (new, `db/options/`, extends `QueryOptions`) carries the four documented options: `collections` (restrict to source collections), `conditionOptimization`, `countApproximate` and `parallelism`; the string values get constant classes `ConditionOptimization` (`AUTO`/`NONE`) and `CountApproximate` (`EXACT`/`COST`) instead of magic strings. `AQL::SEARCH_OPTIONS` is deliberately distinct from `AQL::OPTIONS` (the `FOR`-level options hydrated into `ForOptions` — whose hydration silently dropped the SEARCH-specific keys before this). `aqlFor()` forwards its whole `$init` to `aqlSearch()`, so all three keys work through it unchanged. Without the new keys the emitted AQL is byte-for-byte identical to before. Covered by unit tests (`AqlSearchTest`) and a live integration case (`SearchIntegrationTest::testAqlForWithSearchOptions`, running the full `FOR … SEARCH ANALYZER(…) OPTIONS {collections, conditionOptimization, countApproximate}` form against a real View). Documented in `wiki/{fr,en}/aql/aql-operations.md`.

- New `db/functions/search/` AQL function family — the native AQL *ArangoSearch functions* for querying Views, exposed through a new `SearchFunction` enum and one helper per function: context `analyzer()` / `boost()` (wrap a search expression to set its Analyzer or its scoring weight); filtering `phrase()` (string form, or array form mirroring the official AQL array syntax one-to-one — integer `skipTokens` wildcards and associative-array object tokens like `['STARTS_WITH' => ['ips']]`), `levenshteinMatch()` (typo tolerance; when a later positional option is given, earlier omitted ones are filled with the official server defaults `transpositions = true` / `maxTerms = 64`), `ngramMatch()` / `minhashMatch()` (n-gram and MinHash–Jaccard similarity; PHP signature takes the mandatory analyzer **third** and the optional threshold **last** — a required PHP parameter cannot follow an optional one — and the helper re-orders the emitted AQL), `minMatch()` (at least N sub-expressions true) and `exists()` (attribute presence/type checks; `exists($path, analyzer: 'text_en')` fills the `"analyzer"` type literal by itself), plus `inRange()` (index-accelerated range test); scoring `bm25()` (`b` given without `k` fills the official `k = 1.2` default) and `tfidf()`. `minMatch()` / `inRange()` reuse the existing `MiscFunction` constants (the functions also exist outside `SEARCH`); the new `SearchFunction` enum holds the ten ArangoSearch-only names, including `OFFSET_INFO` (constant only, helper deferred). Arguments that must be AQL string literals (Analyzer names, search targets, `exists` types, the `phrase` content) are quoted via `json_encode` — double quotes and non-ASCII characters become valid JSON-style escapes — while paths, expressions and the scorers' `doc` variable stay raw. Covered by unit tests (`SearchFunctionsTest`, 100% on the new files) and a live integration test (`SearchIntegrationTest`: helper-generated `SEARCH` expressions run against a real `arangosearch` View — phrase order/adjacency, skip-token wildcards, Damerau vs pure Levenshtein, prefix arrays with `minMatchCount`, `MIN_MATCH`, `EXISTS` + `IN_RANGE`, unicode-escaped literals, and `BM25` ranking flipped by `BOOST`). Documented on the new `wiki/{fr,en}/aql/aql-functions-search.md` page (with an ArangoSearch primer).

- `startsWith()` (string functions) now supports the ArangoSearch `STARTS_WITH(path, prefixes, minMatchCount)` form: pass an **array** of prefixes (emitted as a JSON array, so plain strings are quoted) and an optional minimum number of matching prefixes. The legacy string form is unchanged (the prefix stays a raw AQL expression that callers quote themselves), so existing call sites are unaffected.

## [1.1.0] - 2026-06-10

### Added

- Query profiling, wired through the façade and the model layer. A query can now be **run in profiled mode** and its real measurements read back as typed objects. On a model, pass `Arango::PROFILE` (`'profile'`) to `list()` / `get()` (`true` → profile level 2, or an explicit integer level) — the query executes normally and the next `getProfile()` / `getStats()` returns the measurements. On the façade, `ArangoDB::getStats(): ExecutionStats` and `ArangoDB::getProfile(): ProfileResult` read the last cursor's `extra`; the same delegations exist on every model via `ArangoTrait`. `ExecutionStats` (new `db/results/`) exposes `scannedFull()` / `scannedIndex()` / `filtered()` (the "needs an index" signal), `executionTime()`, `peakMemoryUsage()`, `fullCount()`, `writesExecuted()`, `documentLookups()`, cache counters, plus `get()` / `raw()`. `ProfileResult` adds `timings()` (per-phase seconds), `totalTime()`, `warnings()` and `plan()` on top of the `stats()`. New cursor option constants `CursorField::PROFILE` / `CursorField::OPTIMIZER`, new `Extra::PROFILE` / `Extra::PLAN` keys, and the `Arango::PROFILE` model option. Covered by unit tests (`ExecutionStatsTest`, `ProfileResultTest`, `ProfileFacadeTest`, `ProfileDelegationTest`, and profile-threading cases in `DocumentsListTraitTest` / `DocumentsGetTraitTest`) and a live integration test (`ProfileIntegrationTest`, asserting real scanned/filtered counts and per-phase timings). The `wiki/{fr,en}/db/explain-and-profiling.md` guide now documents both explain **and** profiling end to end.

- Typed query EXPLAIN, wired up to the façade and the model layer. `Database::explain()` already existed at the low-level client, but the `ArangoDB` façade did not expose it and the `Documents` model never used it — so application code had no way to check whether its queries were index-accelerated. This adds: `ArangoDB::explain( AqlQuery|string $query, array $bindVars = [], array $options = [] ): ExplainResult` on the façade (delegates to the client), `ArangoTrait::explain()` on every model, and `DocumentsListTrait::explainList( array $init = [] )` which explains **the exact query `list()` would run** for the same filter / facet / search / sort / pagination input (without executing it). The result is a typed, read-only `ExplainResult` (new `db/results/`) exposing `usesIndex()`, `indexesUsed()` (→ `IndexUse[]`: name, type, collection, fields, unique, selectivity — gathered from the plan's `IndexNode`s), `rules()` (optimizer rules applied), `collections()`, `nodeTypes()`, `estimatedCost()`, `estimatedNrItems()`, `isModificationQuery()`, `isCacheable()`, `warnings()`, plus `plan()` / `raw()` escape hatches. New `ExplainField` enum for the response keys. Covered by unit tests (`ExplainResultTest`, `ExplainFacadeTest`, `ExplainListTest`) and a live integration test (`ExplainIntegrationTest`, asserting a real persistent index is reported as used and that a non-indexed filter reports a full scan). Documented end-to-end in the new `wiki/{fr,en}/db/explain-and-profiling.md` guide.

- New `db/functions/bit/` AQL function family — the 12 native AQL *bit functions* for bitwise work on unsigned 32-bit integers, exposed through a new `BitFunction` enum and one helper per function: logic `bitAnd` / `bitOr` / `bitXor` (each accepting either a single numbers array or two operands) and `bitNegate`; inspection `bitTest` / `bitPopcount`; shifts `bitShiftLeft` / `bitShiftRight` (`value, shift, bits`); construction `bitConstruct` ⇄ `bitDeconstruct`; and bitstring conversion `bitToString` ⇄ `bitFromString`. Each returns the ready-to-drop `BIT_FUNCTION(args)` string. PHP arrays (array form of `bitAnd`/`bitOr`/`bitXor`, and `bitConstruct`) are emitted as JSON literals and the `bitFromString` bitstring as a quoted string literal; every other argument is passed through as a raw AQL expression, so columns like `doc.flags` work directly. Covered by unit tests (`BitFunctionsTest`, 100% on the new files); documented on the new `wiki/{fr,en}/aql/aql-functions-bit.md` page.

- Completed the `db/functions/documents/` AQL function family — 17 new helpers bring it to the full 21 (one per `DocumentFunction` enum constant, previously only `has` / `merge` / `translate` / `value` were implemented): key reflection `attributes` / `keys` (with `removeSystemAttrs` / `sort` flags) and `values`, `entries`, `count` / `length`; projection `keep` / `keepRecursive` and `unsetAttributes` / `unsetRecursive`; recursive merge `mergeRecursive`; dictionary building `zip`; example matching `matches`; and identifier parsing `parseCollection` / `parseIdentifier` / `parseKey` / `isSameCollection`. Each returns the ready-to-drop `AQL_FUNCTION(args)` string. Helpers whose AQL argument must be a *string literal* quote it automatically via `json_encode` (the attribute names of `keep` / `unset*`, the collection name of `isSameCollection`, and PHP arrays passed to `matches` / `zip`); the document / identifier arguments stay raw AQL expressions. `unsetAttributes()` is named so because `unset` is a reserved PHP keyword (`UNSET_RECURSIVE` → `unsetRecursive`). Covered by unit tests (`DocumentFunctionsTest`); documented in `wiki/{fr,en}/aql/aql-functions-checks.md`.

- Vector / approximate nearest-neighbour (ANN) AQL support — the building blocks for similarity search over *embeddings* (RAG, recommendations). Four new numeric function helpers under `db/functions/numerics/`, each returning the ready-to-drop `AQL_FUNCTION(args)` string: the exact, index-free distances `l1Distance()` (`L1_DISTANCE`, Manhattan) and `l2Distance()` (`L2_DISTANCE`, Euclidean), and the index-accelerated `approxNearCosine()` (`APPROX_NEAR_COSINE`) and `approxNearL2()` (`APPROX_NEAR_L2`) — the latter two taking an optional `nProbe` argument emitted as the `{"nProbe":N}` options object (more centroids probed → more accurate, slower). `APPROX_NEAR_COSINE` / `APPROX_NEAR_L2` are added to the `NumericFunction` enum (`L1_DISTANCE` / `L2_DISTANCE` were already declared). A new high-level `aqlVectorSearch()` operation (`db/operations/`) builds the complete ANN query in the canonical `FOR … SORT APPROX_NEAR_…(…) [DESC|ASC] LIMIT … RETURN …` form: the `$metric` argument (`'cosine'` default, or `'l2'`) selects **both** the function and the sort direction (cosine → `DESC`, closer to 1 is nearer; l2 → `ASC`, closer to 0 is nearer), guarding against the classic mismatch; the metric must match the `VectorIndex` metric, and an unsupported metric throws `InvalidArgumentException`. Optional `nProbe`, iteration variable (`docRef`) and `RETURN` projection are supported via named arguments. Two constant classes replace the magic strings: `VectorMetric` (`COSINE` / `L2` / `INNER_PRODUCT` — the metric values, also usable for `VectorIndex` params) and `VectorSearchOption` (`N_PROBE` — the function options key). Vector indexes are an **experimental** ArangoDB feature (server started with `--experimental-vector-index`). Covered by unit tests (`NumericFunctionsTest`, `AqlVectorSearchTest`) and a live integration test (`VectorIntegrationTest`, which seeds a cosine `VectorIndex` and asserts the nearest-neighbour ranking — validated end-to-end against ArangoDB 3.12.4 with the experimental flag; skips cleanly when the feature is off). Documented in `wiki/{fr,en}/aql/aql-functions-numerics.md` and `wiki/{fr,en}/aql/aql-operations.md`.

## [1.0.0] - 2026-06-09

First stable release. All AQL high-level operations are supported (the last one, `WINDOW`, landed in this release), alongside the full document/edge model layer, the filter / facet / group / search engine, the HTTP controllers, the ArangoDB clients (Database, Collection, Graph, View, Analyzer, Transaction, indexes including vector), and the CLI commands. Line and method test coverage is at 100%, backed by a server-free unit suite and a live integration suite.

### Added

- `WINDOW` operation builder (`aqlWindow()`) — sliding-window aggregation (running totals, rolling averages, and other statistics over neighbouring rows), completing the set of AQL high-level operations. Both forms are supported, selected by the presence of `AQL::RANGE_VALUE`: row-based (`WINDOW { preceding: N, following: M } AGGREGATE …`) and range-based (`WINDOW rangeValue WITH { preceding: …, following: … } AGGREGATE …`, where the `WITH` keyword belongs to the `WINDOW` syntax, unrelated to the collection-declaring `WITH` operation). `$init` keys: `AQL::AGGREGATE` (required; reuses the same aggregate machinery as `COLLECT` via `aqlAssignments()`), `AQL::PRECEDING`, `AQL::FOLLOWING`, `AQL::RANGE_VALUE`. Bounds are emitted as-is when numeric and single-quoted when given as strings (ISO 8601 durations like `PT1H`); a cumulative running total uses the string `'unbounded'` as `preceding` (verified live: a bareword is parsed as a collection name, the quoted string is accepted). Without an aggregate the builder returns an empty string. The bounds object is serialized by a dedicated `aqlWindowBounds()` helper (one file per helper, reusable on its own). New `AQL::PRECEDING` / `AQL::FOLLOWING` / `AQL::RANGE_VALUE` enum keys. Covered by unit tests (`AqlWindowTest`, `AqlWindowBoundsTest`) and a live integration test (`WindowIntegrationTest`). Documented in `wiki/{fr,en}/aql/aql-operations.md`.

- Automatic `WITH` declaration on anonymous graph traversals — the edge-traversal methods (`EdgesGetTrait::getVertices()` and its `getOutboundVertices` / `getInboundVertices` / `getAnyVertices` / `getFirst*` wrappers, plus `EdgesCountTrait::countVertices()` and its `countOutboundVertices` / `countInboundVertices` / `countAnyVertices` wrappers) now prefix the generated query with an AQL `WITH` clause when they traverse a raw edge collection (no named graph). In a cluster, collections reached dynamically by collection-set traversals must be locked at query start to avoid deadlocks; the reachable vertex collections are derived from the model's `_from` / `_to` vertex models by direction (`OUTBOUND` → the `_to` collection, `INBOUND` → the `_from` collection, `ANY` → both, de-duplicated). Nothing is emitted for named-graph traversals (collections already known), when the vertex models are absent, or on a single server where the clause is a harmless no-op. The declared collections can be overridden through the method's `AQL::WITH` option (a collection name or an array of names). The existing `aqlWith()` helper is now wired into the query pipeline via the new private `PrepareTraversalTrait::prepareTraversalWith()`. Results are unchanged — only the generated AQL gains the leading `WITH`. The standalone `countEdgesQuery()` (a plain `_from`/`_to` edge count, not a traversal) is untouched. Documented in `wiki/{fr,en}/aql/aql-operations.md` (`aqlWith()`) and `wiki/{fr,en}/clients/graphs.md`.

Aggregates reuse the `FacetAggregator` catalogue (`sum` / `avg` / `min` / `max`), and grouping-key transforms reuse the `alt` engine (`alterExpression`, e.g. `year = DATE_YEAR(doc.created)`); per-group counting uses `WITH COUNT INTO` alone and a `LENGTH(1)` aggregate when combined with other aggregates. The raw `Arango::COLLECT` spec stays accepted as a power-user passthrough. The dimensions are keys of the model's `Arango::FACETS` whitelist (the filterable facets become the counted facets, an unknown key is ignored); v1 supports the scalar `Facet::FIELD` and the array-membership `Facet::IN` family (`LIST` / `LIST_FIELD` / `LIST_FIELD_SORTED`). `FacetCountsQueryTrait` builds one `LET` sub-query per dimension (`COLLECT value = … WITH COUNT INTO count SORT count DESC RETURN { value, count }`) over the **same conjunctive filter** as the list (counts reflect the currently filtered set), and `DocumentsFacetCountsTrait::facetCounts()` runs them in a single round-trip; the controller (`PrepareFacetCountsTrait`) exposes the buckets under `options.facets`, next to `options.total`. - The list facet family is unified under `Facet::IN` (op-driven membership); `Facet::LIST`, `LIST_FIELD` and `LIST_FIELD_SORTED` become thin aliases, and the broken `length` branch was removed. The facet dispatcher was hardened (nullable type lookup, null-safe logger).

### Removed

- `ListItemTrait` / `MultiFieldTrait` (`models/traits/`) — orphaned, untested legacy traits composed by `Documents` but never called, and generating invalid AQL (a bind parameter used as an object attribute name; a hardcoded `doc.hasPart` filter). Replaced by `DocumentsArrayTrait` (see Added): `deleteListItem`/`insertInMultiField`/… map to `arrayRemove`/`arrayInsert`/…, `deleteReverseInMultiField` → `arrayPurgeRef`, and `updateDateParentMultiField` is dropped in favour of an application-level `afterUpdate` listener (documented in `wiki/{fr,en}/db/arrays.md`).
- `PositionTrait` (`models/traits/aql/`) — dead, half-written trait: it was composed by no model, `preparePosition()` was never called, and its only AQL form had a hardcoded `@value` placeholder (no `$binds`/value parameter) plus an unfiltered `FOR doc_coll IN <collection>` scan. Removed the trait and its characterization test rather than freeze unused, broken output; a position-ranking feature, if ever needed, should be designed fresh with proper binding.
- `Facet::THESAURUS` — it was exactly an `EDGE` facet with `AQL::FIELDS = '_key,name,alternateName'` and `op = like`, so it is now expressed as that configuration rather than a dedicated type.
- Magic `id → _key` rewriting in facets — key/property aliases are now declared explicitly via `Facet::PROPERTY` everywhere, with no hidden remapping.

### Fixed

- `aqlCollect()` could emit invalid AQL combining `AGGREGATE` and `WITH COUNT INTO`, which are mutually exclusive: when both keys were supplied it appended both clauses. `AGGREGATE` now takes precedence and `WITH COUNT INTO` is dropped (to count alongside aggregates, express the count as a `LENGTH(1)` aggregate). The only caller (`EdgesCountTrait`, which passes `WITH COUNT` alone) is unaffected. The companion `aqlCollectReturn()` mirrors the same rule.
- `aqlUpsertExpression()` never rejected an `init` that defined **both** `FILTER` and `SEARCH`: the second guard repeated the first one's condition (`!isset($filter) && !isset($search)`) instead of detecting the both-defined case, so it was dead code and a conflicting init silently fell through to `$search ?? $filter` (SEARCH wins). The guard is now `isset($filter) && isset($search)` and throws `FILTER and SEARCH cannot be defined at the same time.`, matching the documented exclusive `UPSERT [ searchExpression | FILTER filterExpression ]` syntax. The two guards are split into independent `if` clauses. Calls passing exactly one of the two (every existing usage) are unchanged. Characterized in the new `AqlUpsertExpressionTest`.
- `removeValue()` array helper emitted the plural `REMOVE_VALUES` instead of `REMOVE_VALUE`, so the single-value `FilterFunction::REMOVE` (`remove`) silently no-op'd on a scalar (its second argument must be an array) and raised an AQL error when an occurrence limit was passed; it now emits `REMOVE_VALUE`, and the optional limit works as documented. The `$limit` parameter also accepts `int|string|null` and ignores non-numeric/non-positive values instead of raising a `TypeError` at call time (e.g. a `remove` filter built from a query string). `FilterFunction::REMOVES` (`removes`) keeps using `REMOVE_VALUES`.
- `ArangoCommand` was uninstantiable whenever its `init` array omitted the `directory` key: the constructor's fallback `$this->directory = $init[…] ?? $this->directory` read the `$directory` property *before initialization* (declared `public ?string $directory ;` with no default in `ArangoDumpAction` / `ArangoRestoreAction` / `ArangoListDumpsAction`), throwing `Error: Typed property … must not be accessed before initialization`. The triplicated declaration is extracted into a single `oihana\arango\commands\traits\DirectoryTrait` (property defaulted to `null` + an `initializeDirectory()` initializer matching the `initializeEncrypt()` / `initializePassphrase()` pattern), the three actions now `use DirectoryTrait`, and the constructor calls `$this->initializeDirectory( $init )`. Behavior for configs that already supplied a directory is unchanged.
- `ArangoDB` façade hardening: `getCursor()` was typed `: Cursor` (non-nullable) but `$cursor` is null before the first `execute()` and after `streamDocuments()` resets it, so calling it in those states threw a `TypeError` — it now returns `?Cursor`. `getExtra()` / `getFoundRows()` likewise dereferenced a possibly-null cursor and now return `[]` / `0` when no query has run. The constructor caught only `ArangoException`, but the client builder throws `InvalidArgumentException` for an invalid config (e.g. empty endpoint), so build failures were never logged; the `catch` is widened to `Throwable` (logged, then rethrown unchanged).
- `CollectionManagementTrait` (`ArangoDB` façade) called an **undefined** `toLegacyException()` in the `catch` blocks of `collectionRename()`, `getIndex()` and `getIndexes()` — a leftover shim from the migration off the official ArangoDB-PHP fork. On any client error those three methods crashed with `Error: Call to undefined method` instead of surfacing the failure. The dead `try/catch` wrappers are removed so the `oihana\arango\clients\exceptions\ArangoException` propagates directly, and the redundant `ArangoException as ArangoClientException` import alias (both names resolved to the same class) is dropped. The boolean / log-and-swallow methods are unchanged.
- `PropertyController` PATCH in `raw` mode (`init[Arango::RAW] = true`) returned `null` instead of the submitted property value: the handler read the prepared payload with object-property syntax (`$payload->{$this->property}`), but `PayloadsTrait::propertyPayload()` yields an **array** (`[ property => value ]`), so the access always resolved to `null`. It now reads the value with `oihana\core\accessors\getKeyValue( (array) $payload , $this->property )`, which handles arrays and objects uniformly. Non-`raw` mode (which reloads via `model->get()`) is unchanged. Characterized then fixed in `PropertyControllerTest`.
- `aqlFields` `Field::QUOTED`: it double-quoted the key used for BOTH the output label and the attribute access, emitting invalid AQL like `"my-key":doc."my-key"` (an attribute with special characters cannot be reached with double quotes — confirmed rejected by ArangoDB with a syntax error). The attribute access now uses backticks (`"my-key":doc.`my-key``), the valid form; only the output label is double-quoted. Cases pairing `Field::QUOTED` with `Field::NAME` or `Field::ALTERS` are unchanged. Characterized then fixed in `AqlFieldsTest`.
- AQL `like()` helper: its third parameter was named `$caseSensitive` but maps to ArangoDB's `LIKE(text, search, caseInsensitive)` third argument, so it did the opposite of its name (`true` produced a case-INSENSITIVE match). Renamed to `$caseInsensitive` (default `false` = case-sensitive, matching AQL). The emitted AQL is unchanged for every call path — `?search` stays case-insensitive and the `like` filter-function alter keeps its output — only the (now honest) parameter name and the docs change. Added the previously missing direct unit test, and fixed the stale `SearchTraitTest` comment that described the predicates as "case-sensitive".
- `command:arangodb` document `upsert` now honors the `removeKeys` exclusion list: it was passed under the `excludes` key, which the upsert query never reads, so the exclusion silently had no effect.
- `ArangoCommand` registered the wrong signals for event dispatch: it passed `(int) [SIGINT, …]` (which casts to `1`) to the variadic `setSignalsToDispatchEvent()`. The intended signal list is now passed.
- Removed the unused `ignoreCollection` arangodump option (`ArangoDumpOptions::$ignoreCollection` / `ArangoDumpOption::IGNORE_COLLECTION`): no such flag exists in `arangodump`, and exclusion is resolved client-side as a `--collection` complement. The `--ignore-collection` CLI flag is unchanged.
- `ArangoConfigTrait` now validates the connection settings with exceptions instead of `assert()` (which is compiled out when `zend.assertions=-1`), so a missing database/endpoint/username never silently flows on as null. Properties are initialized; an empty password remains valid.
- Declare the previously-undeclared `ext-pcntl` requirement in `composer.json` — `ArangoCommand`'s signal handling (`getSubscribedSignals()` / `handleSignal()` / `setSignalsToDispatchEvent()`) relies on the `SIGINT` / `SIGTERM` / `SIGUSR1` / `SIGUSR2` / `SIGALRM` constants provided by the extension.
- Security: `command:arangodb dump` / `restore` no longer run `arangodump` / `arangorestore` through a shell. The command was assembled as a single string (option values quoted only with `json_encode`) and passed to `system()`, so a value containing shell metacharacters (password, database, endpoint, directory, collection name) could break the command or inject arbitrary shell. The processes are now executed via `proc_open()` with an explicit argument vector (no shell), so values are passed verbatim. New `ArangoProcessTrait` (`optionsToArguments()` + `runProcess()`), covered by `ArangoProcessTraitTest` including a metacharacter-stays-one-argument unit test and a no-shell probe.
- `command:arangodb restore --last` could never locate the archive: it set the input file to the bare filename returned by `findFiles()` (whose `FILTER` yields `$file->getFilename()`), so `assertFile()` failed unless the current working directory happened to be the dump directory. The `--last` branch now prepends the dump directory, matching the interactive branch.

- `command:arangodb restore --date <date>` could never locate the archive. The by-date suffix was built as `Char::DASH . $database . $shouldEncrypt ? FileExtension::TAR_GZ_ENCRYPTED : FileExtension::TAR`, but `.` binds tighter than the ternary, so PHP evaluated `(Char::DASH . $database . $shouldEncrypt)` as the condition — always a non-empty (truthy) string. The suffix therefore always collapsed to `.tar.gz.enc` and silently dropped the `-{database}` segment, so a non-encrypted restore-by-date never matched its `{date}-{database}.tar.gz` file. The non-encrypted branch also used `FileExtension::TAR` (`.tar`) instead of `FileExtension::TAR_GZ` (`.tar.gz`), which the dump action actually produces. The suffix logic is now isolated in the testable `ArangoRestoreAction::getArchiveFileSuffix()` helper and covered by `ArangoRestoreActionTest`.

- `Facet::ARRAY_COMPLEX` was broken at query time: it iterated an undefined variable (`FOR doc_workshops IN resultworkshops`, with no separator) and built its filter on the out-of-scope `result` document, so the facet never ran. It now iterates the in-scope embedded array (`doc.<key>`) and returns `1`. Confirmed against a live ArangoDB.
- `Facet::EDGE_COMPLEX` used a malformed loop variable (`AQL::DOC_REF . $key` → e.g. `docRefnumbers`) instead of the sibling convention `doc_<key>`; it now reads `doc_<key>` like the other builders, and gained per-field multi-value (OR) plus per-value inline negation (`!=`) on the same traversed vertex.
- Date filters with a valid `tz` always threw: the timezone-conversion path passed the bound placeholder (`@tz`) to `dateLocalToUTC()`, whose own timezone re-validation then rejected it (the literal timezone is already validated upstream by `isValidTimezone()`). The re-validation is now disabled there (`$throwable = false`), so `?filter={...,"tz":"Europe/Paris"}` emits `DATE_LOCALTOUTC(@v,@tz)` instead of crashing — and the new date `between` honors `tz` on both bounds.

### Security

- `?sort=` in open mode (no `sortable` whitelist) interpolated the raw URL key into `doc.<key>` without validation — an AQL injection vector (a model that exposes `?sort=` without configuring `$sortable`, the default). `SortTrait::prepareSort()` now validates each open-mode sort key with `assertAttributeName()`; the whitelisted path was already safe. Legitimate dotted paths (`address.city`) still sort; an unsafe key throws `ValidationException`.
- `aqlFields()` (the projection builder) interpolated the source attribute (`Field::NAME ?? key`) into `doc.<attr>` without validation. Although projection fields are server-config today (`?fields=` is not wired to any controller), the bare source is now guarded by `assertAttributeName()` as defense-in-depth, so a future URL-driven projection cannot inject AQL. The two legitimate non-identifier paths are preserved: the `Field::QUOTED` escape hatch (a special key reached via backticks, `doc.`my-key``) and numeric keys.
- HTTP facets enforce a strict trust contract: only bound values (`@bind`) are user-controlled. Operators are whitelisted (`getAlias`, falling back to the type default), facet keys are whitelisted against the model's `Arango::FACETS` (an unknown key is ignored), and the URL-provided sub-field names of complex facets are validated by `assertAttributeName` before being interpolated — a dangerous name fails the facet (logged and skipped). `alt` function names are likewise whitelisted (an unknown function is a no-op), so no `?filter=` / `?facets=` input can inject AQL.
